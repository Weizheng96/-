# Step 4 — Potentially Using Organizations

Output: `<patent_id>"潜在使用组织".md`

针对每个高优先级场景，列出**真实存在**的、有动机和能力使用该专利的组织。允许使用 `WebSearch` 进行联网验证。

## 基本要求

- 不要瞎编公司名。每个组织都要能在公开信息中找到对应业务线
- 排除专利权人自己及其明确授权的关联方
- 区分"可能性高"（有公开产品 / 论文 / 招聘 / 专利布局命中相关方向）和"可能性中"（业务相邻但无直接证据）

## 七条硬规则（按优先级顺序写入候选清单时必须全部应用）

### R-1. 按"产品族"而非"公司"展开候选

每个组织必须列**至少 2 个具体产品族**作为二级条目；Step 5 在二级条目（产品族）粒度上跑独立 verdict，不得在公司粒度合并。

原因：同一公司不同产品族在专利保护范围内的命中度可能截然相反——例如同一硬件厂的"旧代驱动 / 旧版 SDK"产品族 vs "新一代芯片 + 新 SDK"产品族；或同一软件厂的"上游开源贡献者身份" vs "商业发行版产品"——前者可能完全不构成侵权证据而后者高度可疑，必须拆分独立 verdict。

### R-2. 拓扑变体细分（条件激活——独立权字面含整数限定 / 多实例约束 / 多层结构）

当 Step 2 抽出的某条 F# 含**整数限定**（如 N ≥ 2、K ≥ 1、M ≥ 2，或"多个" / "多层" / "至少 X 项"等限定），且候选产品支持多种部署拓扑（如单实例 vs 多实例、单层结构 vs 多层结构、单端口 vs 多端口），**必须按拓扑变体拆成多个候选条目分别 verdict**——不允许把整数限定弱读、用 vendor 单实例文档去推多实例命中。

常见拓扑维度：
- **整数实例数**：N=1 单实例部署 vs N ≥ 2 多实例部署
- **每实例规模**：每实例 1 个 vs 每实例 ≥ 2 个的子结构
- **层级深度**：单层聚合 vs 两层（含）以上嵌套聚合
- **多对象数**：M=1 单对象 vs M ≥ 2 多对象

原因：同一 vendor 产品在不同拓扑下命中度可能从第 1 档跨到第 5 档。把整数限定弱读会系统性高估命中——主 agent 易把"单实例 vendor 文档"+"理论上可扩展到多实例"误判为"多实例字面命中"。

跨领域示例（拓扑变体把命中度拉开 ≥ 2 档）：
- 生物医药 / 制剂工艺：专利限定"≥ 3 种活性成分协同制剂"，仿制药 vendor 默认描述 2 成分制剂→不能直接套用；需分"2 成分单方" vs "3+ 成分协同"两个拓扑分别 verdict
- 机械 / 加工：专利限定"≥ 4 轴联动加工"，机床 vendor 默认描述 3 轴 + 旋转工作台→须分"3 轴 + 旋转工作台" vs "4 轴真正联动"两个拓扑
- 工业自动化 / 机器人：专利限定"≥ 2 台协作机器人 + 共享视觉系统"，vendor 默认描述单台机器人→须分"单台 + 独立视觉" vs "≥2 台 + 共享视觉"两个拓扑
- UI / 交互：专利限定"≥ 2 显示区域协同识别手势"，UI vendor 默认描述单显示区域→须分"单显示" vs "多显示协同"两个拓扑

拓扑拆分规则：
(a) 每个拓扑变体作为独立 `candidate_slug`（如 `<base-slug>-topology-A` / `-topology-B`）；
(b) 每个变体独立跑 Step 5 §A 19 类源穿透 + §C 子 agent 复核；
(c) 在 Step 4 候选清单末尾追加一行"该 vendor 公开文档实际描述的默认拓扑是 X；其他拓扑标公开资料不足"，避免后续 Step 5/6 误把所有拓扑写成一条 verdict。

### R-3. R-OPENSOURCE = true 时的双层激活

依赖 Step 3a R-OPENSOURCE 标志。

1. **上游开源项目本身必须作为独立候选**——含其官方组织 / Foundation / 主仓库 / 主要 maintainer 公司；不得因"上游被动 ship 不构成主观恶意"理由而合并到下游 / 弱化处置。原因：上游开源项目是**最直接的技术实施者**——是 ship 出代码本身的实体。商业发行版只是把上游代码"打包 + 推荐启用"。商业发行版的"主动推荐"加分**叠加**在上游"技术直接命中"之上，**不替代**它。技术上判定时，上游与最高置信商业发行版并列独立成行、各自独立跑 §A 协议。

2. **商业发行版必须独立于上游开源项目作为单独候选**——商业发行版的"主动推荐配置"是比上游"被动 ship 代码"更强一档的**法律 aggressiveness** 信号（recommending config = 主观恶意 +1）。如商业发行版在官方 procedure 中把某 config 参数（如 `feature_x: true`）作为推荐配置明确写出，就构成"主动推荐启用"档证据；而上游仅 ship 该 feature 但默认关闭、不推荐启用，证据强度低一档。

跨领域示例（与项目下任何评测专利无显然语义关联——示例仅展示双层激活机制）：
- 生物医药 / 仿制药领域：原研药企工艺 + 公益专利池授权方 / 各国仿制版本均独立成行 / 各自独立跑 §A 协议
- 农化 / 育种领域：核心品种持有方 + 开放许可作物联盟 / 商业仿制方均独立成行
- 制造 / 加工工艺领域：工艺原创工厂 + 行业联盟实施工艺 / 商业代工方均独立成行
- UI / 交互领域：开源参考实现 + 浏览器引擎社区 + 商业产品集成方均独立成行

当 R-OPENSOURCE = false（专利完全是 vendor 自研闭源），跳过此规则。

### R-4. 主体覆盖检查（强制收尾自检）

写完候选清单后，主 agent 必须显式输出一段"主体覆盖检查"——**按 Step 3 给出的本领域 5 大主体类型清单**逐条 check，对每条说明"已纳入 N 家"或"经评估不适用，理由 X"，并展示 Step 3 必跑参考清单中的具体厂商名是否都被点名评估（即便最终判"不适用"）。

这是防漏检的最后防线；**不允许任何主体类型 / 任何代表性厂商被默默跳过**。

### R-5. 长尾覆盖检查（强制 — 防欠覆盖召回不全）

完成主体覆盖检查后，主 agent 必须额外执行下述兜底 query 至少 3 条，并把命中的非头部 / 海外上市 / 细分赛道玩家**主动补回 Step 4 候选清单**（不是仅在 caveat 里提）。

**query 设计硬约束 — 技术词驱动 而非 业务定位驱动**：

兜底 query 必须**用 ≥ 2 个独立的本专利核心技术词组合**作为定位关键字，而非仅用本领域的"业务定位词"。原因：
- 业务定位 query（"<本领域业务定位词> 上市公司"）天然偏向"主业是该领域"的玩家，会召回行业头部 + 主流
- **主业不在该领域、却有该领域细分技术线**的玩家（往往是 niche 候选最高浓度区）只在使用**核心技术词**作为关键字时才能被召回——这些玩家的 SEO 标签是其他赛道，业务定位 query 把它们漏掉

**core 技术词 = F1-F# 中可识别的具体技术名词**（来自 Step 2 的"必要技术特征清单"以及"等同特征讨论"），跨领域示例：
- 半导体 / EDA / 芯片设计：`时序收敛 STA`、`光刻 EUV 多曝光`、`Chiplet 互连 UCIe`
- 生物制药 / 基因测序：`mRNA LNP 递送`、`CRISPR 基因编辑 脱靶`、`长读长测序 ONT`
- 工业自动化 / 机器人：`PLC 实时调度 EtherCAT`、`机械臂 力位混合控制`、`SCADA 报警分级`
- 音视频 / 编解码：`AV1 帧间预测`、`HDR 动态元数据`、`空间音频 头追踪`
- 消费电子 / UI 交互：`折叠屏 铰链 应力疲劳`、`视网膜投影 Micro-OLED`、`触觉反馈 力学建模`

**query 模板**（按金融市场覆盖角度展开；每条都用核心技术词组合定位）：
- 模板 1：`<核心技术词 1> <核心技术词 2> 上市公司`
- 模板 2：`<核心技术词 1> <核心技术词 2> NASDAQ OR NYSE 中概股`（专门召回 NASDAQ/NYSE 上市的中国概念股小盘股——漏检风险最高一档）
- 模板 3：`<核心技术词 1> <核心技术词 2> 独角兽 OR "新三板" OR 港股`
- 模板 4（可选）：`<核心技术词 1> <核心技术词 2> 创业公司`

如果本专利核心技术词只有 1 个明确的（如纯算法 / 纯材料 / 纯协议），用该单一技术词 + 1 个邻近场景词组合（如 `<核心技术词> 应用` / `<核心技术词> 解决方案`），仍要避免单纯的"业务定位词"。

兜底 query 命中的厂商如**完全不在原 Step 4 候选清单**，且**业务（哪怕是非主营业务线）与本专利领域核心特征对应**，必须**新增 candidate 条目**——不能仅在 verdict 末尾顺带提一句。判定"是否对应"时不要按主营业务卡得太严——本规则就是为了召回主营业务在其他领域的细分技术玩家。

#### 用户驱动反向兜底（强制）

当用户在 Step 5/6 阶段主动提出某个具体目标做侵权评估时（如"分析 XXX 公司是否侵权"）：
- 主 agent 首先检查该目标是否在原 Step 4 候选清单内
- 如**不在**：立即反思"Step 3 的 5 大主体分类 + 必跑清单是否有同类厂商被漏掉？"——用上述长尾兜底 query 补检
- 补检后**先更新 Step 4 候选清单**（含同类的其他被漏厂商，不只是用户提的那一家），再做该目标的 verdict

原因：用户提名一个原始候选清单里没有的目标，意味着 Step 4 候选枚举漏了某条筛选路径——往往还有同类的其他细分玩家也被遗漏。

#### 领域过拟合自检（强制）

自检段落末尾必须显式声明"本 Step 4 候选清单**没有**包含任何与本专利领域无关的厂商示例"。如果发现误套，必须立刻回退 Step 3 重做领域分类。

### R-6. 核心技术候选归并（强制 — 防"同一核心技术被拆成数十个产品候选"过散）

当 Step 4 候选清单里**多个产品候选共享同一可识别的核心技术**，且**该核心技术正是 F1-F# 的实质载体**（即专利保护范围本体）时，必须把"核心技术 + 该技术的原创 / 权属组织"作为**单独的候选条目**列入清单，并把所有使用该技术的下游产品**作为该候选的应用案例 / 证据材料**而非各自独立成行。

判定"核心技术候选"成立的三个准则（**全部满足**才触发归并）：

1. **技术存在独立的可识别身份**——它有正式名称、存在于代码库 / 标准文本 / 论文 / 数据集 / 工艺规范 / 公开 SDK / 协议规范中，可被任何第三方独立引用
2. **F1-F# 的全部命中链路在该技术中已完整存在**——并不是"集成它就侵权"，而是该技术内部已经实现了 F1-F#，下游产品仅做"集成 + 配置 + 推荐启用 + 渲染"，没有引入独立的二次创新
3. **存在可归因的原创 / 权属组织**——开源项目宿主 / 主 maintainer 公司、商业 vendor 总部、标准化机构、研究机构、个人发明人——能在法律层面定位到具体责任主体

跨领域示例：
- **生物医药 / 工艺**：某药物缓释微球的核心制剂工艺由 Y 制药公司发明并申请专利，10+ 家药企在自家成药中沿用该工艺。归并后候选清单是"该缓释微球工艺 + Y 公司"一条
- **UI / 交互**：某手势识别算法由 X 大学研究组提出 + 配套开源参考实现，N 家手机 / 平板 OEM 在系统 UI 中调用该算法。归并后候选清单是"该手势识别算法 + X 研究组"一条
- **工业自动化 / 机器人**：某六轴机械臂运动规划库由 K 机器人公司开源并主导维护，多家工业机器人厂在产品控制器中集成。归并后候选清单是"该 motion-planning library + K 公司"一条
- **金融科技 / 风控**：某贷款风险评分模型由 M 公司提出 + 论文公开 + SaaS 售卖，多家银行在内部决策引擎中接入。归并后候选清单是"该风险评分模型 + M 公司"一条

**归并后的执行约束**：
- 核心技术候选**作为单独一行**进入 Step 4 清单及 Step 6 状态总表；"违约产品"列填写"<技术名> + <原创 / 权属组织>"
- 该核心技术下游使用方**不再单独成行**——而是作为该核心技术候选 verdict 文件的"应用案例"段落（含使用方名称 / URL / 部署证据 / 推荐配置 / 公开市占率）出现
- 例外（保留独立成行的情形）：
  - 某下游使用方**对该核心技术做了独立的二次创新**（如自家 fork 增加了独立的 F# 命中分支），则下游 fork 仍单独成行，并在 verdict 中标注"基于 <核心技术> 的衍生改造"
  - 某下游使用方在**完全不同的市场 / 法域 / 客户群**中以**独立责任主体**形态部署，独立成行可便于法律可分主张
- 不归并的情形：
  - 多个产品**只是同领域 / 同概念**，但**底层技术实现互不相同**（各自从头实现）——必须保留独立候选
  - 同一公司的不同产品族**确实使用了不同的核心技术**——按各自的核心技术分别归并

**为什么这条机制层规则重要**：当核心技术与具体产品强解耦时（开源框架 / 算法库 / 标准规范 / 工艺等普遍如此），"按产品族展开"会把同一核心技术拆成 N 个独立候选，导致状态总表前列被"使用者"填满、技术原创者被淹没。归并到核心技术候选后，状态总表呈现"原创者主导 + 使用者作为应用面证据"的二级结构，更准确反映"谁最实质地实施了该专利"。

### R-7. 架构层级不符的预排除（强制 — 防 Step 5 浪费检索预算）

某些候选名义上属于本领域大类，但其具体技术架构层级与本专利所述抽象层不在同一层（例如本专利定义在某抽象层 X，候选实现却在抽象层 Y——同属一个产业大类但完全不同的执行模型）。这类候选应在 Step 4 末尾**显式标记为"已排除（架构层级不符）"**，不在 Step 5 浪费完整 §A 19 类源检索。

判定准则：候选是否能命中 F1（专利核心架构存在性）？如果连 F1 概念上都不构成（既不是字面命中也不是合理等同），则该候选既不可能在 §A 后 10 类非主流来源里出现强证据，也不可能通过反向工程升档——直接预排除。

跨领域示例：
- 数据库专利定义在"分布式事务协调器"层级，候选实现却是"单机存储引擎"层级——架构层级差距明显
- 自动驾驶专利定义在"传感器融合算法"层级，候选实现却是"高精地图制图工艺"层级——同属自动驾驶产业但执行模型完全不同
- UI 交互专利定义在"手势识别算法"层级，候选实现却是"显示驱动硬件"层级——同属消费电子但抽象层不在同一层

预排除清单格式：

```markdown
## 架构层级不符预排除清单

| 候选名 | 名义所属领域 | 实际架构层级 | 与本专利 F1 的概念差距 | 排除依据 |
| --- | --- | --- | --- | --- |
```

预排除的候选**不进入** Step 5 / Step 6 的完整 verdict 流程，但**必须保留在 Step 6 状态总表**作为"第 5 档：已排除（架构层级不符）"档以保证审计可追溯（避免后续维护者误以为"漏检"）。
