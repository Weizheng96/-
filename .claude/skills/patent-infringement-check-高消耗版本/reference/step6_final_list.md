# Step 6 — Final Infringement List

Output: `<patent_id>"违约列表".md` + 每个候选独立的 `候选/<slug>/关键证据.md`

仔细复核 Step 5 中所有 verdict。对每一项做以下三项核查后再决定保留或剔除：

1. **来源真实性**：链接是否可访问？是否官方一手？必要时再次 `WebFetch` 确认原文。
2. **特征完整性**：是否真的命中**所有**必要技术特征？只要缺一个就降级为"疑似"或剔除。
3. **时间合规性**：发布日期确实晚于授权日？

## 专利法律状态对状态机的影响（强制后置调整）

Step 1 抓到的法律状态必须在此处显式处理：

| 法律状态 | 处置 |
|---|---|
| **已授权（Granted / Active）** | 标准流程，状态机直出三档 |
| **未授权（Pending / 公开未授权）** | 所有"确认侵权"候选**降级标注**为"**等待授权 — 临时保护期**"——技术上 F1-Fk 可能全部命中，但权利尚未生效，**不能立即提起侵权诉讼** |
| **失效（Expired / Lapsed）** | 所有候选标 `已排除（专利已失效）` |
| **无效宣告进行中** | 标注"宣告无效程序待决，授权范围可能变化" |

### Pending 状态紧凑模板（4 项必填，原样套用即可）

当 Step 1 法律状态 = Pending 时，Step 6 必须在状态总表下方加专属段落：

```markdown
## ⚠️ 专利未授权处置（仅当 Step 1 法律状态 = Pending 时输出）

(a) **临时保护起算日**：<专利公开日 YYYY-MM-DD>
(b) **可执行性**：本专利尚未授权 → **不能直接提起侵权诉讼**。所有"确认侵权"档候选均标"等待授权 — 临时保护期"
(c) **追溯许可费基础**：临时保护期内（公开日 → 授权日）第三方使用记录构成授权后追溯主张许可费的证据基础
(d) **审查风险点**：审查中专利可能被驳回 / 缩限 → 当前"确认侵权"档候选清单需在专利授权后重新审视；如缩限发生，状态机需按缩限后的权利要求重跑 Step 5
```

**关键约束**：不要因 Pending 状态把候选移出"确认侵权"档——技术判定（5 档排序键）只看 F# 命中度，与法律状态无关；Pending 影响的是"维权可行性"（写入"后置调整及原因"列），不影响排序。这与 §D 的"等同回避偏倚"+"法律不确定性回流"两条状态机硬约束保持一致。

```markdown
## 状态机
每个候选的每条适用独立权都有一个状态：
- `已排除`：在该独立权下，至少一项必要技术特征字面 + 等同均不命中（或时间不合规、产品形态明显偏离专利权人指出的相邻方案）

- `确认侵权`：在该独立权下，所有要件都拿到字面或等同的直接公开证据，且时间合规
- `标准合规候选`（**仅当 Step 3a R-STANDARD=true 且本专利已声明 SEP / 已纳入强制国标必要专利清单**）：技术上必然命中 F1-Fk，但维权应通过 FRAND / 标准强制许可框架；不进入"高置信侵权候选"节
- `待标准合规状态确认`（仅当 R-STANDARD=true 且工具能力下无法确认标准声明状态）：标准实施类候选默认归此状态

```

### 状态总表排序规则

排序键 = Step 5 §D 给出的 5 档细化排名。**关键：排序只看"技术判定"列**——技术上能不能命中——不看后置法律调整层（Pending 降级 / 现有技术 caveat / R-STANDARD 转移 / 合同豁免）。后置调整在"后置调整及原因"列体现，但**不影响排序位置**。

理由：(a) 用户最关心"技术上谁最像"——这是定位侵权候选的核心信号；(b) 法律可执行性是律师 / 法院的工作，会随时间变化；(c) 把法律因素回流降低排序会重蹈"等同回避偏倚"覆辙。

### "一行一产品 OR 核心技术"模型（必须遵守）

状态总表中**每一行对应一个产品 OR 一个核心技术**——产品 = `candidate_slug` 粒度；核心技术 = Step 4 归并规则识别出的"核心技术候选"。两类行都不对应"组织"。

1. **同一产品由多组织共建时，"组织"列允许列多个**——典型情形：
   - 开源共建项目（项目宿主 + 若干主要 maintainer 公司）
   - 合资联合产品（多家公司联合 ship 同一品牌产品族）
   - 含上下游联合 vendor 的 reference architecture（系统集成方 + 关键组件提供方）

2. **同一产品命中多条独立权时，"适用独立权"列允许列多条**——例如某产品同时落入方法权 + 装置权 + 含特定硬件约束的整机权。

3. **同一组织如有多条独立产品族，按产品分别成行**——与 Step 4 "按产品族而非公司展开候选"一致。

4. **不允许把"上游开源项目本身"写成单一抽象主体**——如果一个开源项目作为产品入选，"组织"列必须显式列出项目宿主（基金会 / 非营利）+ 主要 maintainer 公司清单；后者才是法律可诉的主体。

5. **核心技术候选行优先于其下游使用者行**——按 Step 4 归并规则，核心技术候选独立成行；其下游使用方**融入该候选 verdict 文件的"应用案例"段落**，**不在状态总表单独成行**（除非下游做了独立 F# 命中的二次创新或属"独立责任主体"例外）。

6. **同一技术档位内的子排序键（强制顺序）**：
   - **6.1 核心技术候选** > **6.2 该核心技术的原创 / 权属组织自营产品** > **6.3 该核心技术的第三方集成方 / 商业发行版** > **6.4 仅引用 / 推荐启用该技术的部署方**
   - 原因：维权时核心技术权属人是首要责任主体；原创组织自营产品是"主动推荐启用"档证据；第三方集成方依次降档。

### 列定义

| 列名 | 内容 |
| --- | --- |
| 序号 | 整数序号，按技术判定 5 档排序 + 同档内核心技术优先子排序键 |
| **类型** | `核心技术` / `产品族`——核心技术行依据 Step 4 归并规则成立；产品族行对应 `candidate_slug` 粒度 |
| **违约产品** | **只写一个干净的产品名或技术名**——不允许用加号 / 括号 / 斜杠等任何分隔符堆叠多个名字或附加限定语。规则：(a) 单一具体产品的侵权 → 写**产品名**；(b) 多个产品共用同一项底层技术导致侵权 → 写该**技术名**；(c) 拓扑变体行（同一产品在不同部署形态下命中度不同）→ 重复使用同一产品名，靠"适用独立权" / "技术判定" / "关键证据"三列的差异区分，**不要**在产品名里加"（T-A）"、"（多卡 HA）"等注解。**禁止**形式：`X + Y`、`X / Y`、`X（注解）`、`X with Y`。产品名应是 vendor 公开使用的产品名或行业通用技术名，干净到能作为 search 关键词独立返回结果。组件 / 部署形态 / 拓扑细节放到 verdict / 关键证据 文件展开 |
| 组织 | 与该产品 / 核心技术相关的法律责任主体（一个或多个）；标注上市代码 / 法律性质 |
| 适用独立权 | 引用 Step 2 的独立权编号；可多条 |
| **技术判定**（排序键）| 5 档之一：第 1 档"确认侵权（高）" / 第 2 档"确认侵权（中）" / 第 3 档"公开资料不足（强候选）" / 第 4 档"公开资料不足（弱候选）" / 第 5 档"已排除" |
| 后置调整及原因 | 列出 Step 5 §D 的 7 条后置调整逐条核查结果（等同三步法 / 反向证据 vs 限定语 / 法律状态降级 / 现有技术 caveat / R-STANDARD 转移 / §5.0 豁免 / patent license 承诺）；没触发的也明示"未触发" |
| 关键证据 | **指向当前候选独立的 `关键证据.md` 文件**：`[关键证据](候选/<slug>/关键证据.md)` 形式的可点击链接 |

### per-candidate `关键证据.md` 通用强制规则

- **每条证据原文优先使用 verbatim 引文**——如无字面引文，写"未找到字面引文"或"公开资料不足"，不要伪造
- **每条都给出可点击 URL**——付费墙 / 登录墙 / 链接失效的，写"没有相关链接"或注明原因
