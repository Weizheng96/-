# Step 5 — Evidence Collection（全候选证据协议）

Output:
- `<patent_id>"潜在违约".md` = 候选总索引 + 检索时间窗 + 按主体类型分组 + 每候选指向其 `_verdict.md` 的指针 + 一句话状态摘要
- `候选/<slug>/_sources.md` + `候选/<slug>/_verdict.md` = 每候选独立的取证 / 合议产物

Step 5 对 **Step 4 列出的所有候选 × 它们各自适用的所有独立权**强制执行。**找到一个高置信侵权候选后不得提前终止**；必须把所有候选跑完一轮。

**关键时间约束**：只看 patent **授权日 / 公开生效日之后**的发布内容。授权日之前的工作不构成侵权（但作为现有技术抗辩依据另算 — 见 §A.9）。

## 5.0 执行顺序与并行化

### 分级执行（不是"跳过整类源"，只是 query 深度缩放）

所有候选都跑完整前 19 类源（§A 第 1-19 类）+ 状态机判定（§C-D）+ 三栏 verdict。第 20 类反向工程视成本启用。

- **P0**（高可能性 / 行业头部 / 长尾兜底命中 / 用户驱动 / 上游开源）：每类源深查 + 多语言 cross + 多轮反向专利墙；**必 spawn 子 agent 独立审**；§D 三栏完整
- **P1**（中-高）：每类源 2-3 条 query；主 agent 直审（可不 spawn 子 agent）；§D 三栏完整
- **P2**（中 / 低）：每类源 1-2 条 query 验证空命中；主 agent 直审；§D 三栏完整

**关键**：分级缩放的是"每类源 query 深度"，**不是**"跳过整类源"——P2 候选不得跳过 §A 任何一类。

### 不得"找到一个就停"

候选只要在清单中就必须跑完。原因：(a) 维权通常需多被告策略；(b) 不同候选落在不同独立权下；(c) 单点评估会让其他候选停留在含混"公开资料不足"。

### 候选间并行执行

§A（取证）+ §B（归档）+ §C（子 agent 审阅）应**按候选分批并行**——一次处理 3-5 个候选（受工具调用并发与 WebFetch / WebSearch 速率限制）。每批内部，§C 的多文档子 agent 也并行 spawn（一条消息内多个 `Agent` 调用）。

### 用户驱动反向触发

当用户主动提某具体候选 → 该候选立即升级到 P0，跑全协议。同时按 Step 4 用户驱动反向兜底机制：先反思该候选的同类是否在原 Step 4 漏掉，更新清单后再做 verdict。

### P0 候选必须实际跑完协议（纯机制层规则）

任何被标为 P0 的候选——无论是因为可能性等级"高"自然进入、还是因长尾兜底 / 用户驱动 / 上游开源等机制带入——主 agent 都必须：
1. 跑完整 §A 19 类源 + §B 本地存档 + §C 子 agent 复核——禁止以"工具能力受限"为由跳过任何步骤
2. 在 Step 6 状态总表中**按真实命中度排序到前列**——禁止把已升级到 P0 的候选写到总表中段或末尾
3. 如最终 verdict 为"公开资料不足"或"已排除"，必须在 verdict 中明示升级路径建议（反向工程 / 法务取证 / NDA 渠道 / 招股书披露 / 客户合作披露 / 标准化会议查询等具体动作）

### "待证"字眼自动触发 query（强制）

候选条目的任何字段——产品族 / `candidate_slug` / 适用独立权 / 主体类型 / 命中场景 / 公开资料链接——若出现 `（待证）` / `（TBD）` / `（pending）` / `（待核实）` / `<slug 待证>` 等任何 placeholder 字眼，必须自动 trigger 至少 2 条该候选的独立 WebSearch query 后再写定该字段；不得让任何"待证" placeholder 在 Step 4 / Step 5 / Step 6 终稿中残留。

### 豁免（必须在 `_verdict.md` 中明示豁免理由）

- 时间不合规（候选公开材料全部在授权日之前）→ 直接 `已排除（时间不合规）`
- 候选明显不在专利领域 → 直接 `已排除（领域无关）`
- 候选已经在另一个候选条目下被同等评估（重复条目）→ 引用主条目即可

## §A. 信息源穿透扫描（19 类必跑 + 第 20 类高成本兜底）

对每个候选 × 每条适用独立权，跑下述源。每类至少跑 1 条 query；**P0 候选每类源至少 2-3 条 query**；某类完全未命中时必须明示"<类> 在 <来源 X / Y> 检索 0 命中"，不能默默跳过。

**类目分层**：
- 第 1-9 类 = 主流来源（所有候选必跑）
- 第 10-19 类 = 非主流来源（前 9 类无法把候选定位为"确认侵权" / "已排除"时必跑——即任何 verdict 落在"公开资料不足"档的候选都必须续跑这 10 类）
- 第 20 类 = 高成本兜底（反向工程；仅在候选商业价值高 / 法务确认要追诉时启用，其他情况下在穷尽性证明表中标"成本超出本轮，建议法务决定"）

### 检索单元构造硬规则（机制层 — 适用所有 19 类源，跨领域适用）

#### 3×2 矩阵：identifier 维度 × 语言维度

对每个候选 × 每类检索性源（§A.1 / §A.2 / §A.7 / §A.10 / §A.11 / §A.12 / §A.13 / §A.14 / §A.15 / §A.16 / §A.17 / §A.18 / §A.19），query 构造**至少跨 2 个 identifier 维度 × 2 种语言**：

**identifier 维度（至少跨 2 个）**：
- (1) **vendor 本地法定名 / 中文 / 母语名**（如适用本地市场）
- (2) **vendor 国际通用名 / 英文名**（如适用国际市场或顶级国际阵地）
- (3) **vendor 邮箱域 / 主域名**（`@<domain>` / `site:<domain>`）—— 通常最稳定，**强烈推荐必含**

**语言维度（至少跨 2 种）**：
- (A) **vendor 总部 / 主营市场的语言**
- (B) **本领域研究阵地主导语言**（参见 Step 3a "本领域研究阵地清单"——主导语言由主 agent 根据该阵地现场判定）

每个候选 × 每类源至少跑 **6 条 query 单元中的 ≥ 50%（即 ≥ 3 条）**，覆盖至少 2 个 identifier 维度 + 2 种语言。

#### 为什么这一步关键（机制层动机）

- vendor 本地名 + 本地语 query 召回 vendor 在本地市场的产品页 / 招聘 / 财报
- vendor 国际名 / 邮箱域 + 国际语 query 召回 vendor 在国际研究阵地 / 标准化组织 / 跨境客户案例的产出
- **关键漏检模式**：很多 vendor 在本地市场用本地名 + 本地语，但**研究产出 / 国际标准 contribution / 跨境合作白皮书**几乎一定用国际语 + 国际名 + `@vendor-domain` 邮箱域。只查本地语会**系统性漏掉**这一档高产证据；反之亦然
- **邮箱域 / 主域名作为 identifier 比 vendor 名鲁棒**——vendor 名有缩写 / 翻译 / 子品牌歧义，邮箱域 / 主域名是固定的，跨语言不变

#### query 单元留痕（强制 — 配合"执行纪律 verifiable 证据"规则）

每条 query 单元执行后必须在候选的 `_sources.md` 或 verdict 文档中留下证据：
- **命中**：query 原文 + 命中 URL + 1-2 句话命中要点 + §A.9a 时间档分类
- **0 命中**：query 原文 + 阵地 / 数据库名 + "0 命中" 显式声明
- **工具能力受限**：明示"<工具> 不支持 <检索方式>，建议法务通过 <账号 / 工具> 补查"

**不允许的措辞**：
- "建议法务深读" / "建议补查" — 必须**先**跑 query，跑完后才能转 "建议升级"
- "vendor 工程师在 <顶会> 发表论文" 等**断言**而非**检索**——必须有 specific URL 或 0-hit 声明
- "公开度有限" / "公开材料丰富" 等模糊评价——必须替换为具体 query 命中数

主 agent 在每个候选 §A 收尾时必须 grep 自检——任何 §A.1-19 行只要含 "建议" / "据悉" / "通常" / "据说" / "推测" 等推测性词汇而无 verifiable URL，必须**返回去补跑 query** 或显式标 0 命中后才能进入 Step 6。

### 主流来源（前 9 类，所有候选必跑）

#### 1. 专利墙（★ 最优先 + 强制两轮检索）

候选组织在授权日后的同主分类专利。对实现细节型专利这是命中率最高的一类源——厂商在产品文档中通常不披露技术细节，但会在专利申请中详细描述同一发明点。

- **第一轮（粗扫）**：用 `assignee:<候选组织名> + cpc:<本专利主分类号> + after:<授权日>`，渠道 IncoPat / 智慧芽 / Patentscope / [Google Patents Advanced](https://patents.google.com/advanced)
- **第二轮（深读）**：对第一轮命中的代表性专利**逐条比对独立权 1-3 与目标专利的特征重合度**——按目标专利的 F1-Fk 必要技术特征逐项打勾；如有 ≥ 60% 字面重合，作为"专利墙佐证"加入证据链

**工具能力诚实降级**：仅有 WebSearch / WebFetch / curl 工具的环境下，**无法可靠执行 Google Patents Advanced 的复合查询**——Google Patents 是 JS 渲染、IncoPat / 智慧芽是登录墙。空命中时必须在 verdict 中明确写"反向专利墙因工具能力限制未充分检索，建议人工通过 IncoPat / 智慧芽账号补查"。

#### 2. 学术 / 研究阵地深度检索

按 Step 3a "本领域研究阵地清单"中的每个阵地，针对**每个候选**至少跑 1 条 query（每条 query 按 3×2 矩阵构造，至少跨 2 个 identifier 维度 + 2 种语言）。SKILL 不点名任何具体阵地——清单由 Step 3a 现场推演。

**强制执行纪律**：每条 query 必须留 verifiable 证据（命中 URL 或 0-hit 显式声明），不允许写"建议法务深读"代替实际 query 执行（见上文"检索单元构造硬规则 — query 单元留痕"）。

**单一阵地 0 命中的不构成豁免**：任何阵地 0 命中必须由**至少 2 条不同 identifier 的 query**验证（如 vendor 名 query + vendor 邮箱域 query 各 0 命中），才能确认该阵地无相关公开。

**为什么这一类源最高产**：vendor 在产品文档 / 公关博客中通常只写抽象功能描述，但在研究阵地（顶会 / 顶刊 / 标准化 contribution / 行业白皮书）发表时**会披露具体算法 / 触发条件 / 整数限定 / 配置参数**——因为发表的目的就是说服同行评审 / 客户 / 合作伙伴这套方法成立。这是 vendor 闭源产品下 F# 字面证据**最高产出**的一档源。

#### 3. 宣传材料

官网产品页、技术博客、媒体报道、官方公众号、招股书 / 财报技术披露、投资者关系页。

#### 4. 使用手册 / 技术文档

产品 datasheet、用户手册、SDK / API 文档、开发者门户、参考架构（reference architecture）、配置示例。

#### 5. 行业标准 / 测试规范 / 合规公告

按 Step 3a "本领域研究阵地清单"中的 **标准化组织 / 行业联盟 / 监管部门**子集逐家跑 query。每条 query 按 3×2 矩阵构造（identifier × 语言）。SKILL 不点名任何具体标准化组织——清单由 Step 3a 现场推演。

强制留 verifiable 证据（命中 URL 或 0-hit 显式声明）。

#### 6. 第三方联合案例研究（条件激活，依赖 R-PARTNER）

当 Step 3a 判定本领域有联合案例文化（R-PARTNER = true）时，重点检索 vendor × 上游平台厂 / 客户 / 系统集成方的联合案例 / reference architecture / 联合白皮书。

**领域无关的机制规律**：联合案例研究**比 vendor 自营页粒度深得多**——为了向最终客户证明集成可行性，联合案例往往写出 vendor 自营页不愿披露的具体配置 / 参数 / 规格 / 算法细节 / 性能指标。

**主 agent 现场推演本专利领域典型的联合案例阵地**——可能是：
- 上游平台厂的 reference design / certified partner program
- 行业头部客户的公开 case study / customer success story
- 系统集成方的 deployment guide / 实施指南
- 政府试点示范单位的公开案例库

**SKILL 不点名任何具体厂商或具体平台**——由主 agent 根据 Step 3a "本领域研究阵地清单"中的"行业联盟白皮书 / 头部客户公开案例"子集推演。

#### 7. 上游开源贡献者归因检索（条件激活，依赖 R-OPENSOURCE）

当 R-OPENSOURCE = true 且 vendor 产品基于活跃开源项目 / 公开维护的算法库 / 标准参考实现时，从项目的 `AUTHORS` / `MAINTAINERS` / `CONTRIBUTORS` 文件 + git log / 提交历史检索带 vendor 邮箱域的贡献者，再搜该工程师在该项目的邮件列表 / 年度 conference / 工作组 meeting 的演讲与 contribution。

主 agent 根据 Step 3a 给出的领域分类 + R-OPENSOURCE 标志，现场推演**本领域代表性上游项目**（≥ 3 个，含主仓库 URL 与项目宿主组织）。**SKILL 不点名任何具体上游项目**——清单由主 agent 现场推演（不同领域典型上游差异极大：软件类 vs 硬件 IP 类 vs 工艺类 vs 生物医药类的"上游"性质完全不同）。

归因价值：当某 vendor 工程师邮箱出现在某上游项目的核心 maintainer 列表 + 顶会演讲时，把该 vendor 从"产品页字面引用某上游项目"提升到**"公司层面的设计参与"**——这是远比"引用 / 集成"强的间接证据档。

**具体可执行 query 模板**（按工具能力降级排列；适用任何代码 / 文档托管平台）：
- 代码托管全文搜索：`site:<host-domain> <upstream-project> "Author:" "@<vendor-domain>"`（命中 commit 提交者邮箱域）
- 代码托管高级搜索：`<upstream-project> in:file path:*MAINTAINERS* "@<vendor-domain>"`
- 代码 API（需 token）：跨 vendor 的所有相关项目检索
- 本地 git log（已 clone 仓库时）：`git log --author="@<vendor-domain>" --since="<patent-grant-date>" -- <upstream-repo-path>`
- 项目邮件列表 / patchwork：`site:<project-list-domain> "<vendor-domain>"`
- 文件级 grep：upstream 仓库的 `MAINTAINERS` / `AUTHORS` / `CODEOWNERS` / `CONTRIBUTORS` 文件中搜 `@<vendor-domain>`

**工具能力诚实降级**：本工具能力下代码托管的全文 / Code 搜索部分查询受限（需登录 / API rate limit / JS 渲染）；如此类查询返回 0 命中，**不等于"该 vendor 没有上游贡献"**——必须诚实降级标注"工具能力受限，未深度跑过"。

当 R-OPENSOURCE = false（专利涉及全闭源产品 / 算法 / 硬件 IP），跳过此规则；改用 §A.2 学术 / 研究阵地 author 归因（论文署名厂商）。

#### 8. 开源 fork 仓库独立评估（条件激活，依赖 R-OPENSOURCE）

当 vendor 维护开源 fork（`<vendor>/<upstream-project>` 形态）时，fork 是关键侵权载体，**必须作为独立 candidate**。fork 的 NEWS / 关键源文件需直接抓取并独立子 agent 审。vendor 的常规商用驱动包可能不含某 feature，但 vendor 维护的 fork 分支可能为某客户启用该 feature。

#### 9. ★ 现有技术 / 在先公开（强烈推荐 — 通用必跑）

检索专利**申请日之前 ≤ 24 个月**内同主分类的公开资料。如发现现有技术覆盖目标专利核心要件 ≥ 60% → 影响专利**新颖性 / 创造性**——这是决定专利能否授权 / 已授权能否被无效宣告的关键。

- query 模板（按 3×2 矩阵构造）：`<本专利标题核心技术词> before:<专利申请日>` / `<本专利主分类号> before:<专利申请日>` —— 必须用本领域研究阵地的主导语言至少跑一次
- **渠道**：按 Step 3a "本领域研究阵地清单" + 本领域专利数据库（国家级 / 国际 / 商业专利搜索引擎）覆盖。**SKILL 不点名任何具体数据库**——主 agent 现场推演本领域适用的专利 / 研究公开数据库（CS 类、生物医药类、机械类、农化类的可用数据库差异极大）
- 命中时在 verdict 中标"**现有技术抗辩潜在依据**：[在先公开物] 公开于 [日期]，早于专利申请日 [日期] X 个月，与权 N 的 F1-Fk 重合度 m/k"

#### 9a. ★ 时间线交叉验证（强制 — 每条候选证据必跑）

每条候选证据被纳入 verdict 前，必须按以下三个时间点交叉判定：
- **专利申请日**（filing date / priority date）
- **专利授权日**（grant date / publication date）
- **候选证据发布日**（vendor doc / commit / paper / 产品 release）

| 时间关系 | 法律意义 | 处置 |
| --- | --- | --- |
| 证据发布日 < 专利申请日 | **现有技术抗辩潜在依据** | 不计入侵权证据；按 §A.9 路径处理 |
| 专利申请日 ≤ 证据发布日 < 专利授权日 | **临时保护期证据** | 可追溯主张许可费但不能直接起诉 |
| 证据发布日 ≥ 专利授权日 | **正常侵权证据** | 标准处置 |

每条 verdict 文档的 _sources.md 中**每条证据必须显式标注三档之一**。

**特殊情况**：上游开源项目持续发布——同一上游 feature 可能横跨三档。处理规则：以**评估的目标候选实际部署 / 引用的版本发布日**为准，不以"feature 首次 commit 日期"为准。

### 非主流来源（10-19 类，"确认侵权" / "已排除"档可豁免，其余必跑）

10. **招聘 JD / Offer / LinkedIn 简介**：`<vendor> 招聘 高级工程师 <技术词>` / `LinkedIn engineer profile <vendor> <技术词>` / `BOSS 直聘 <vendor> <技术词>`
11. **财报 / 招股说明书 / 年报技术披露**：SEC EDGAR / HKEXNews / 巨潮资讯网 / 上交所信息披露——"研发投入 / 核心技术 / 竞争优势"章节
12. **招标书 / 中标公告 / 技术应答书**（R-PROCURE = true 时强烈推荐）：按本专利领域 vendor 实际服务的政企客户类型逐家跑——主 agent 现场推演本领域典型公开招标平台（如中央 / 省级 / 市级政府采购、行业准入采购、医院 / 学校 / 国企 / 公用事业采购等），**SKILL 不点名任何具体平台**
13. **会议演讲 / 技术分享视频**：按本专利领域典型的视频 / 演讲分享平台逐家跑——可能包括行业大会录像、公司技术博客、社交媒体技术内容平台、第三方知识分享平台等；主 agent 现场推演本领域的典型平台
14. **Bug tracker / Issue tracker / 邮件列表 archive**：开源项目 issue / mailing list 按 vendor email 域 grep
15. **标准化组织会议纪要 / contribution 文件**（适用于 R-STANDARD = true）：按 Step 3a "本领域研究阵地清单"中的标准化组织子集逐家跑——主 agent 现场推演本领域适用的标准化组织 meeting archive / contribution database，**SKILL 不点名任何具体组织**
16. **多语言源 cross-verify**：中文未公开的可能在英文 / 日文 / 韩文 / 德文公开
17. **客户案例 / Case Study / Reference Architecture deep dive**：`<vendor> case study <client name> <技术词>`
18. **国际同族专利**：PCT / US / EU / JP / KR 同族；Espacenet / WIPO PATENTSCOPE 跨库检索
19. **关联企业 / 子公司 / 控股投资公司专利**：先查工商注册关联关系（天眼查 / 启信宝 / OpenCorporates）→ 扩展专利墙到关联实体名称

### 第 20 类高成本兜底

20. **反向工程 / 产品取证测试 / 客户合作披露**：购买产品做实验室测试 + 解包 / 反编译 SDK / 抓 API 调用——法务取证级路径，仅在候选商业价值高时启用；其他情况下在穷尽性证明表中标"成本超出本轮，建议法务决定"

### 为什么不存在"明显不值得多查的弱候选"

候选既然进了 Step 4 清单，就经过领域适配 + 主体覆盖 + 长尾召回多重筛选。把它打到"弱候选"是 §A 前 9 类未发现实证后的结果，但这不意味着"该候选不可能在非主流源中有强证据"——niche 候选的证据往往就藏在 §A 后 10 类非主流来源里。

## §B. 本地化证据归档

为每个候选在 `专利集/<patent_id>/候选/<candidate_slug>/` 创建子文件夹（`candidate_slug` 用 ASCII 友好的连字符形式）。把每条命中的资料**下载到该子文件夹**：
- **PDF**（论文 / 白皮书 / datasheet / 标准）：`curl -ksSL -o <name>.pdf <url>`
- **网页**：用 `WebFetch` 抓 markdown 后保存为 `<name>.md`，文件头注明 `> 来源 URL: <url>`、`> 抓取时间: <YYYY-MM-DD HH:mm>`；或 `curl -ksSL -o <name>.html <url>` 抓 HTML 直接落盘
- **GitHub 仓库**：单文件用 raw URL 抓取；整仓需要时 `git clone --depth=1`
- **拿不到的资料**（付费墙 / 登录要求 / 链接失效）：在子文件夹内的 `_inaccessible.md` 中明示 URL + 原因 + 替代证据线索

子文件夹必备索引文件：
- `_sources.md` — 列每条材料的：原 URL / 抓取时间 / 类型 / 适用独立权 / 本地文件名
- `_verdict.md` — 见 §C-D

## §C. 子 agent 逐文档审阅（P0 必跑；P1/P2 可选）

对子文件夹内**每一份**本地化证据，**单独 spawn 一个 `general-purpose` 子 agent**（用 `Agent` 工具），传入：
- 该文档的本地绝对路径
- 本专利的 F1-F# 必要技术特征（从 `<patent_id>"专利介绍".md` 抽出）
- 本候选适用的独立权 + 主体类型
- 抽证任务："**只看这一份文档**，按 F1-F# 逐项给出 (字面命中 / 等同命中 / 否) 三档判定 + 原文证据片段；**不要参考任何其他文档**；**不要做综合判定**，只对这份文档负责"

子 agent 必须独立执行——**禁止把多份文档合并到一个 agent 的上下文**。多份文档应**并行 spawn**（一条消息内多个 `Agent` 调用）。

每个子 agent 返回一张该文档的 F1-F# 命中表 + 引用片段；主 agent 汇总到 `_verdict.md`。

## §D. 状态机判定与三栏 verdict（合议 → 输出）

**总流程**：投票汇总 F# → 三档状态判定（细化为 5 档排名）→ 字面命中边界与硬约束核查 → 三栏 verdict 输出 → 后置调整逐条核查。

### 投票汇总规则

主 agent **不得"鸽王式 adopt"sub-agent 的 verdict**——必须按下面强制步骤做独立复核：

1. **重读 claim 1 全文 + 所有适用从属权全文**（不是只看 Step 2 抽取的简化版 F# 清单）
2. **对每条 F#，列出所有可能的 satisfaction paths**（独立权字面 / 从属权 alt / 等同）
3. **对 sub-agent 每条"否（反向证据）"判定，反向证据必须按下表分类**：

   | 引文模式 | 是否构成反向证据 |
   | --- | --- |
   | `Y does NOT do X` / `Y 不支持 X` / `Y 不实现 X` | ✅ **真反向证据** |
   | `Y supports only Z (not X)` / `Y 仅支持 Z` | ✅ **真反向证据** |
   | `Y explicitly excludes X` / `Y 明确排除 X` | ✅ **真反向证据** |
   | `we don't implement X` / `we leave X to future work` | ⚠️ **作用域限定**（不是反向证据）|
   | `X can work together with Y` / `X is orthogonal to Y` | ⚠️ **作用域限定 / 互补语** |
   | `X is out of scope` / `X is beyond this paper` | ⚠️ **作用域限定** |
   | `Y achieves X via different mechanism` | ❌ **不是反向证据**——这反而是等同命中信号 |

   如 sub-agent 引用的"反向证据"落在⚠️或❌行，主 agent 必须降级该 F# 为"公开资料不足"或"等同命中候选"——不得直接判"否"。

4. **对每个未达"字面命中"的 F#，强制跑等同三步法**（doctrine of equivalents），在 verdict 中显式输出 4 行：
   - 同样手段：候选实现 vs 专利描述
   - 同样功能：是否解决相同技术问题
   - 同样效果：是否达到同一性能 / 可靠性指标
   - 显而易见性：本领域技术人员是否能直接想到这种替换

   只有这 4 行**全部不成立**时，F# 才能判"否"；否则判"等同命中"。

5. **F# 投票汇总规则**：
   - **字面命中**：任一子 agent 报"字面命中"，且主 agent 复核 claim text 后认可 → 字面命中（取并集）
   - **等同命中**：未达字面命中，但等同三步法支持 → 等同命中
   - **真反向证据**：上表第 1-3 行模式 → 反向命中
   - **公开资料不足**：所有子 agent 都报"否（无相关描述）"，且无真反向证据，等同三步法也不支持 → 公开资料不足

### F# 字面命中的判定边界

"字面命中"≠"候选输出恰好与权利要求字面一字不差"。判定标准是：
- 候选实现使用了权利要求列出的**具体输入 / 步骤 / 部件**（如 claim 1 enumerated 的输入列表，候选使用其中至少一项 → 输入端字面命中）
- 候选实现产生了权利要求要求的**功能效果**

**不要给权利要求加额外限定**。读得太严会把字面命中误降为等同命中、再误降为否。

### 禁止"上游继承"推断

候选只承认使用了某个**上游开源项目名称**（如"基于 <某上游项目>"、"集成 <某上游库>"等表述）**不构成** F# 的字面或等同命中。F# 必须有**该候选自己文档里的具体描述**或**该候选自己代码 / 配置 / datasheet 中的字面证据**。

子 agent 报告里出现"字面命中（继承）"、"字面命中（基于 <某上游项目>）"等表述时，主 agent 必须将该 F# 降级为"否（仅有上游引用，无候选自身字面证据）"。

### 三档状态 + 5 档细化排名

每条独立权下，按 F# 投票汇总输出**三档状态**：

| 状态档位 | 进入条件 |
| --- | --- |
| `确认侵权` | F1-Fk 全部"字面或等同命中"，且时间合规 |
| `已排除` | 任一 F# 在所有文档里有**真反向证据**（上表第 1-3 行模式）|
| `公开资料不足` | 介于两者之间——部分要件命中、部分公开资料不足、无反向证据 |

**5 档细化排名**（用于 Step 6 状态总表的排序键）：

| 排名档位 | 进入条件 |
| --- | --- |
| 第 1 档：**确认侵权（高）** | F1-Fk 全部**字面**命中 |
| 第 2 档：**确认侵权（中）** | F1-Fk 全部命中，含 ≥1 个**等同**命中 |
| 第 3 档：**公开资料不足（强候选）** | ≥ 60% F# 命中，剩余 F# 公开资料不足无反向证据 |
| 第 4 档：**公开资料不足（弱候选）** | < 60% F# 命中，剩余 F# 公开资料不足 |
| 第 5 档：**已排除** | 任一 F# 有真反向证据 / 时间不合规 / 领域无关 |

### 状态机硬约束（4 条 — 禁止"等同回避"、"法律不确定性回流"、"反向脑补过宽读"、"已排除门槛跨越"）

状态机问的是**技术特征是否落入保护范围**，不是问"法律解读是否会买账"或"法院最终判决会怎么走"。

#### 硬约束 1：等同回避偏倚（禁止）

把"等同命中"当二等公民，看到含"等同"的 verdict 就潜意识降级到"公开资料不足"——错。专利法里**等同侵权与字面侵权同档**，状态机也明确说"字面**或**等同"都进 `确认侵权`。如 F1-F# 全部为字面或等同命中（即使全是等同），状态就是 `确认侵权`，不得降级。

#### 硬约束 2：法律不确定性回流（禁止）

F# 全部命中后又因"法律解读可能不认可"等担忧把状态降为"公开资料不足 ⇄ 风险显著"——错。法律解读由律师 / 法院做，写入 verdict 的"置信度 / 建议下一步"字段；**不应回流降低状态机判定**。状态机表达的是"本 skill 基于公开资料所能给出的最强技术性结论"，而非法律最终结论。

#### 硬约束 3：反向脑补过宽读（与硬约束 1 对偶 — 同样禁止）

当 vendor 自身公开文档只覆盖 F1-Fk 的**子集**时，主 agent 不得对**未覆盖的 F#** 直接标"字面命中"，即便该实现"看起来明显应该做 X" / "理论上 vendor 应该也支持 Y"。

判定"vendor 文档明示某 F#"的硬条件：
- (i) F# 的**核心动作 / 触发条件 / 整数限定 / 多层结构 / 状态属性等关键限定词**在 vendor 文档原文中可定位到 verbatim 段落
- (ii) 该段落清晰对应该 F# 而非邻近特征
- (iii) 段落覆盖的部署形态与专利字面要求的拓扑（按 Step 4 R-2 拓扑变体细分规则识别）**一致**

任一条不满足 → 该 F# 标"公开资料不足"或"等同命中候选 — 待等同三步法验证"，**不**写"字面命中"。

**特别警示：整数限定的拓扑外推禁令** — 当独立权字面含整数限定（如 N ≥ 2、K ≥ 1、M ≥ 2）而 vendor 文档默认描述的是**该整数下界以下的拓扑**（如 vendor 默认描述 N=1 单实例部署），主 agent **不得**写"vendor 理论上可扩展到 N ≥ 2 → 字面命中"——除非 vendor 文档自己显式描述了 N ≥ 2 的多实例拓扑。此情形下应回到 Step 4 R-2，把候选拆为拓扑变体后分别 verdict。

跨领域示例（拓扑外推禁令）：
- 生物医药：专利 N ≥ 3 成分协同制剂，vendor 文档只描述 2 成分制剂 → F# 不命中
- 工业自动化：专利"≥ 2 台协作机器人 + 共享视觉" — vendor 文档只描述单台机器人 → F# 不命中
- 金融科技：专利 K ≥ 3 风控模型集成投票，vendor 文档只描述单模型决策 → F# 不命中

反向脑补的高发场景：(a) vendor 公开度低、文档稀薄，主 agent "替 vendor 想象"实现细节；(b) F# 涉及整数限定 / 多层结构，vendor 文档默认描述更简单形态；(c) F# 与某个常识性等价方案"看起来明显"。所有这些情形都必须按本规则降档为公开资料不足。

#### 硬约束 4：已排除门槛硬约束

判 `已排除` 必须满足下面任一硬条件，**否则一律归 `公开资料不足`**：

| 硬条件 | 适用情形 |
| --- | --- |
| (a) 至少一条 F# 在所有归档证据中命中真反向引文模式（"Y does NOT do X" / "Y supports only Z (not X)" / "Y explicitly excludes X"）| F# 字面级反向证据 |
| (b) 时间不合规（候选公开材料全部在专利公开日之前 = prior art only） | §5.0 豁免之一 |
| (c) 候选明显不在专利领域（领域无关） | §5.0 豁免之一 |
| (d) 架构层级不符（候选实现层级与专利所述抽象层不在同一层；详见 Step 4 "架构层级不符的预排除"段）| 架构层级 |
| (e) 候选已经在另一个候选条目下被同等评估（重复条目）| §5.0 豁免之一 |

**0 命中 ≠ 已排除** — 若候选在 §A 全部 19 类源都未拿到字面 / 等同证据，但**没有**真反向证据 / 时间不合规 / 领域无关 / 架构错位 / 重复，则该候选状态为 `公开资料不足`（第 4 档弱候选），**不是** `已排除`。

`已排除` 表示"我们有理由认为本候选不可能落入保护范围"；`公开资料不足` 表示"公开材料范围内未拿到证据，但不排除取证升级后可能命中"。把"未拿到证据"误标为"已排除"会错误终结后续取证升级路径。

sub-agent 报告里如出现"0 命中 → 已排除"或"未发现公开 → 已排除"等表述，主 agent 必须**强制改判为 `公开资料不足`**（除非主 agent 能在该候选的归档证据中独立指出真反向引文 / 时间 / 领域 / 架构 / 重复硬条件之一）。

### 三栏 verdict 输出格式

verdict 文档**必须**显式分三栏输出，让"状态机原始判定 ↔ 各层后置调整 ↔ 最终判定"全流程可审计——**不允许**只写最终判定。模板：

```markdown
## 状态机三栏判定

| 独立权 | 状态机原始判定 | 后置调整记录 | 最终 verdict |
| --- | --- | --- | --- |
| 权 N | 三档之一（确认侵权 / 已排除 / 公开资料不足）+ 5 档排名 | 列出哪些后置规则被应用 | 经调整后给出的最终状态 |

### 状态机原始判定的来源
- F# 投票汇总 → 三档状态 + 5 档排名
- 这一栏**纯技术判定**，不含法律 / 时间 / 标准 / 豁免任何调整

### 后置调整记录（按下列顺序逐条核查，全部应用且记录，没触发的也要明示"未触发"）
1. **等同三步法复核**：等同命中是否经过同手段/同功能/同效果/显而易见 4 行论证？
2. **反向证据 vs 限定语区分**：sub-agent 的"否"判定是否被误归为反向证据？
3. **专利法律状态降级**：Granted / Pending / Expired / 无效宣告中——参见 Step 6 法律状态处理
4. **现有技术 caveat**：是否存在专利申请日前的现有技术覆盖度 ≥ 60%？
5. **R-STANDARD 转移**：候选是否合规实施标准且应转入"标准合规候选"节？
6. **§5.0 豁免**：候选是否符合时间不合规 / 领域无关 / 重复条目豁免？
7. **Patent license / pledge / 专利池承诺核查**：专利权人是否对候选所在的开源项目 / 标准组织 / 行业专利池做过 patent grant / pledge / non-assert / FRAND 承诺？常见来源类型——(i) 防御性专利联盟成员承诺、(ii) 上游开源项目宿主基金会的 patent commitment、(iii) 标准化组织成员发布的 IPR statement、(iv) 行业 / 国家级专利池声明、(v) 专利权人公司单方公开的 patent pledge。**主 agent 应根据 Step 3a 给出的领域分类，现场列出该领域具体的 (i)-(v) 实例**而非套用固定清单。命中即在 verdict 标注"许可可能覆盖此实施"+"维权前需先解除许可承诺"，**不直接降低技术判定排序**，但作为最终维权可行性的关键约束写入"后置调整及原因"列。工具能力下无法核查时诚实降级标"建议法务通过 IPR 律师补查"。

   跨领域示例：(a) 生物医药 / 仿制药 → 查 Medicines Patent Pool / 公益专利池；(b) 农化 / 种子 / 育种 → 查公益作物专利池（如 IRRI 国际水稻所开放许可）；(c) 机械工艺 / 制造 → 查行业协会专利池声明；(d) UI / 交互 → 查 W3C / 浏览器引擎社区的 RF（royalty-free）IPR 声明；(e) 任何参与开源代码库 maintainer / 标准化组织成员身份的专利权人 → 查该组织的 IPR 政策。本项与 R-STANDARD 不重叠——R-STANDARD 处置"标准必要专利转移"，本项处置"专利权人单方做出的让渡承诺"。

### 最终 verdict
经上述调整后给出的可执行状态。**不得回流降低状态机原始判定**——法律不确定性写在调整层，不回流降低技术判定。

## 总结一句话
<≤ 120 字的一行总结，必须显式写出最终落档（如"落第 3 档（公开资料不足强候选）+ Pending 降级"），便于 Step 6 索引脚本与人工复审快速抓取该候选的核心结论。
```

**为什么三栏而非单列**：单列 verdict 容易把"状态机原始 = 确认侵权"在写作中被法律不确定性回流偷偷改写为"公开资料不足"——一旦写定，复审者无法看到原始技术判定与各层后置调整的差异。三栏强制把"状态机原始判定"和"后置调整"拆开。

**"总结一句话"位置硬约束**：该段落必须放在 verdict 文档的**最后一段**，标题严格用 `## 总结一句话`（二级标题）；不要写到中段或文档头部。位置漂移会导致 Step 6 索引脚本的正则锚点失效。

### config-driven 类特征的双引证要求（条件激活，依赖 R-CONFIG）

当 R-CONFIG = true 时，verdict 中该 F# 必须**同时引用**：
1. 散文描述（prose quote）——例："automatic queue rebalancing... triggered when load threshold exceeds 95% for one minute"
2. 参数名 + 默认 / 推荐值（key:value）——例：`feature_x: true`, `threshold_y: "70"`, `improvement_z: "25"`

只有 prose quote 而无具体参数值 → 仅证明"产品支持该特性"；同时有参数值 → 证明"产品**主动推荐启用**该特性"。

**对其他证据类型（R-CONFIG = false 的领域）的等价规则**：
- **算法步骤型专利**（AI / 推荐 / 编解码 / 加密）：双引证 = 散文描述 + 伪代码 / 公式 / 论文段落引用
- **UI / 交互型专利**：双引证 = 文字描述 + 截图 / 录屏 / 操作步骤逐帧引用
- **硬件 / 芯片接口型专利**：双引证 = datasheet 文字描述 + 引脚图 / 时序波形 / 寄存器表引用
- **流程 / 工艺型专利**（生物制药 / 制造）：双引证 = 工艺描述 + 流程图 / 操作 SOP 引用

主 agent 必须根据 Step 3 给出的领域分类**主动选择**适配的双引证形式，不能机械套用 config-driven 模板。

## §E. 穷尽性证明（"公开资料不足"档候选必填）

如果 §A 全 19 类全部跑过仍维持"公开资料不足"——必须在 verdict 末尾输出**"穷尽性证明"**段落：

```markdown
## 证据穷尽性证明（"公开资料不足"档候选必填，无强弱区分）

已尝试取证手段（19 类源 + 第 20 类反向工程）：
| # | 源类别 | query / URL | 命中要点 / 0 命中 |
| --- | --- | --- | --- |
| 1 | §A.1 专利墙 | `<vendor> assignee + cpc + after:` 共 N 轮 | 0 命中 / 命中 X 不相关 |
| ... | ... | ... | ... |
| 20 | §A.20 反向工程 | （成本超出本轮） | 待法务决定 |

最终缺失证据：F# = X 仍 unprovable
最强证据缺口：vendor 不公开 [具体技术细节]
建议升级路径：[反向工程 / 法务取证 / NDA 渠道 / 律师询证函 / 企业内部线人 / 客户合作披露]
```

只有完成"穷尽性证明"才能维持"公开资料不足"判定。
