# Step 3 — Domain Adaptation + Application Scenarios + R-STANDARD Subprocess

Step 3 拆为 3a / 3b / 3c：
- **3a** 是 SKILL 后续步骤的"中央枢纽"——领域分类 + R-* 标志 + 必跑清单
- **3b** 是基于 3a 的轻量场景推演
- **3c** 仅当 R-STANDARD = true 时执行（标准合规判定）

按 3a → 3b 顺序执行；3c 仅在激活时执行。

---

## Step 3a — Domain adaptation → `<patent_id>"领域适配".md`

主 agent 在推演场景之前，必须先把本专利分类到一个具体领域，并据此**临时构造**后续 Step 4 / Step 5 所需的"主体类型清单 + 必跑参考清单 + 适用规则集"。这一步是 SKILL **去过拟合**的关键——SKILL 不再硬编码任何领域的厂商清单，每次跑 skill 时由主 agent 现场推演。

### 领域定锚强制规则（防过度外扩）

**领域分类必须基于 Step 1 输出（`<patent_id>.md`）中的"背景技术"章节内容**，而不是从权利要求字面措辞或主分类号自由发散。

- **必须直接引用**专利"背景技术"章节中**至少 1 段** verbatim 文本作为主领域判定依据，写入 Step 3a 输出的"领域定锚证据"小节
- **禁止**把背景技术**未提及**的相邻领域纳入主 / 次领域——即便 Step 4 / Step 5 中某个候选可能跨入相邻领域，也只能在该候选的 verdict 中作为"跨领域延伸"标注，不得反过来扩大 Step 3a 主领域
- 如背景技术只描述场景 A（如"数据中心宿主机虚拟交换"），主 agent **不得**把领域扩展到场景 B（如"边缘计算 / 嵌入式 / 工业控制"）——除非背景技术明示这些扩展场景

**自检**：Step 3a 写完后，重读"背景技术"verbatim，对每个写入的主 / 次领域 + 5 大主体类型 + 研究阵地清单**反向核查**——背景技术是否能**直接支撑**该分类？如果某项需要靠主 agent"延伸联想"才能成立，必须从 Step 3a 删除，留到 Step 4 候选 verdict 中按需提及。

**为什么这条最先满足**：过度外扩领域会让 Step 4 候选清单系统性扩大（拉入背景技术未涉及的厂商），Step 5 §A 检索也跟着外扩，最终输出大量"领域外滥伸"候选稀释了核心候选的判定准确度。背景技术是发明人**自己对发明所属领域的定锚**——主 agent 不能比发明人更"懂"该专利的领域。

### 研究阵地清单的推演方法（机制层 — 跨领域适用）

主 agent 推演研究阵地时遵循下述思路（不预设领域）：

- **学术 / 顶会顶刊**：本领域研究成果通常发表在哪个学科或专业方向的顶级期刊 / 会议？产业界发表偏好的会议 vs 纯学术界发表偏好的会议有何差异？
- **标准化阵地**：本领域有无国际 / 国家 / 行业级标准化组织？这些组织的 IPR 数据库 + meeting contribution archives 是否公开可检索？
- **监管 / 政府公开**：本领域有无强制公示制度（如药品批准、专利信息披露、合规审查公告等）？相关数据库是否公开？
- **行业联盟 / 公益专利池**：本领域有无行业自律组织 / 联盟 / 公益专利池？其成员承诺 / 标准声明是否公开？
- **预印本 / 行业白皮书**：本领域研究公开常用哪些 preprint 仓库或厂商白皮书发布平台？

跨领域示例（与项目下任何评测专利无显然语义关联——示例仅展示推演方法）：

- 制药 / 制剂工艺类专利 → 临床医学顶级期刊 / 制药化学顶级期刊 / 国家药品审评中心公开通报 / WHO 公益专利池数据库；预演 3-5 个阵地各给主域名
- 农化 / 作物育种类专利 → 作物学顶级期刊 / 国际农业研究机构 preprint / 国家品种审定公告平台 / FAO 数据库
- UI / HCI 类专利 → 人机交互顶级会议 / 浏览器引擎社区 IPR 声明 / 设计系统社区文档
- 工业自动化 / 机器人类专利 → 机器人 / 控制类顶会 / 工业标准化组织数据库 / 行业联盟参考方案
- 金融科技 / 风控类专利 → 应用计算机科学顶会 + 金融数据科学期刊 + 监管科技公开沙盒披露

主 agent 必须现场为本专利领域推演 ≥ 3 个具体阵地（带主域名），**不能套用上述跨领域示例本身**。

### 多视角强制要求（防长尾遗漏）

每个主体类型必须**从下述至少 3 个视角分别列举**，每个视角至少 1-2 家——避免单一视角的认知盲区漏掉细分玩家：

- **(a) 产业全景视角**：按业务赛道分（行业头部 + 主流玩家）—— 默认视角，但单独使用时容易偏向头部 + 国内 A 股
- **(b) 金融市场视角**：按上市板块分——A 股 / 港股 / NASDAQ 中概股 / 新三板 / 一级市场独角兽未上市；主动补充 NASDAQ / NYSE 上市的中国概念股小盘股（业内宣传少但可能是关键侵权候选）
- **(c) 政府名录 / 行业协会成员视角**：本领域如有政府或行业协会发布的"重点企业名单"、"试点示范单位"、"标准制定参与方"，必须把这些名单中的非头部成员也纳入候选清单

**为什么不能只跑视角 (a)**：单一产业全景视角天然偏向行业头部 + 国内主流 / A 股；NASDAQ / HKEX 中概股、政府试点名录中的非头部成员、产业协会专业组成员等"主流之外的细分玩家"恰恰是产业全景视角最容易漏掉的一档。任何领域都有这条长尾。

### R-* 标志判定标准

| 标志 | 触发条件 | 激活效应 |
|---|---|---|
| **R-OPENSOURCE** | 本专利涉及**开源依赖** —— 不限领域，可以是开源代码库 / 开源硬件设计 / 开放算法 / 开放数据集 / 开放标准参考实现 / 公益专利池授权的工艺等。主 agent 根据本专利领域现场判定 | Step 4 "商业发行版独立评估"+"上游贡献者归因检索"+"上游开源项目本身列入 P0" |
| **R-CONFIG** | 侵权证据可能体现为**配置参数 / 启动开关 / YAML / Helm chart / API 调用参数** | Step 5 "config-driven 双引证" |
| **R-PARTNER** | 本领域是否存在 vendor × 上游平台厂 / vendor × 头部客户 / vendor × 系统集成方 的**联合案例研究文化**——联合白皮书 / reference architecture / customer success story 是否常见？主 agent 现场判定本领域的联合案例阵地 | Step 5 "第三方联合案例研究" |
| **R-PATENTWALL** | 本领域 IPC / CPC 主分类下的**反向专利墙**是否容易被检索？影响因素：(i) 该主分类的总申请量；(ii) 主要 vendor 是否在该主分类下持有大量专利且公开度高；(iii) 该领域专利申请是否倾向于详细披露技术细节。主 agent 根据本专利 IPC / CPC 主分类现场判定 | Step 5 是否花精力跑反向专利墙 |
| **R-PROCURE** | 本领域客户是否通过公开招标采购（运营商 / 政企 / 医院招标书可能暴露技术规格——医疗 / 通信 / 工业自动化高；2C 互联网产品低）| Step 5 是否值得检索招标网 |
| **R-STANDARD** | 本专利涉及的技术是否被任何强制 / 准强制标准覆盖。三类候选：(i) **国际 SEP（标准必要专利）**——由国际标准组织声明的必要专利；(ii) **国家强制标准**——本国法规要求合规实施的国家级标准；(iii) **行业 / 政府准入标准**——细分行业准入要求。主 agent 根据本专利领域**现场推演**适用的标准库 / IPR 数据库（参见 Step 3a "本领域研究阵地清单"）| 若 R-STANDARD = true 必须在 Step 4 之前完成 Step 3c |

### Step 3b → Step 4 衔接强制

每个场景下"具名产品列表"列出的所有 named product **必须在 Step 4 候选清单中以独立产品族条目形式出现**。

原因：大厂在同一场景下往往有数十个 niche / 子品牌产品（多代号、多区域、多 SKU、跨业务线延伸），真正命中专利保护范围的常常**不是主流认知前列那 2-3 个**——而是某个不在排行榜上的子品牌或跨业务线 SKU。仅在 Step 3b 用产品类别作为场景描述、Step 4 只展开"该公司最广为人知的 2-3 个产品族"，niche product 会被默默跳过。

## Step 3c — R-STANDARD 子流程（仅当 R-STANDARD = true 时执行）

输出可放在 `<patent_id>"领域适配".md` 末尾。

涉及标准的专利的维权策略与普通专利**根本不同**：

- **国际 SEP**：权利人必须按 FRAND（公平、合理、无歧视）条款许可，被告也可援引 FRAND 抗辩；不能简单起诉合规实施标准的厂商
- **国家 / 区域强制标准**：标准合规是产品销售的法定要求；如本专利特征与强制标准要求重合，所有合规厂商**自动落入**专利保护范围——但维权可能受标准强制实施约束（"必要专利"性质），通常需通过标准化组织指定的许可程序，而非常规侵权诉讼
- **行业准入标准**：类似强制国标但范围限定到行业；落入合规范围的厂商有"合规即命中"风险

### 主 agent 必须做的

1. **判定标准类别 + 查相关数据库**（best-effort，工具能力下可能受限）：
   - 国际 SEP：按本领域适用的国际标准组织 IPR 数据库（由 Step 3a "本领域研究阵地清单"现场推演）
   - 国家强制标准：按本国 / 主要市场国家的强制标准信息公共服务平台（现场推演）—— 查标准号 + 标准全文 + 实施日期
   - 行业准入：按本领域适用的主管部委 / 行业协会公开文件库（现场推演）
   - 简单 query：`<专利权人> <专利号> SEP declaration` / `<本专利标题关键词> 强制标准 实施` / `<本专利分类号> 行业准入`

2. **判定本专利与标准的关系**——三种结果决定 Step 4 候选清单的处置：

   | 判定结果 | 对 Step 4 候选清单的影响 | Step 6 verdict 处理 |
   | --- | --- | --- |
   | 已声明 SEP / 已纳入强制国标必要专利清单 | 把"标准合规实施类候选"整体移到"标准合规候选清单"；只保留"标准外的具体优化型候选"作为常规侵权候选 | 单列"标准合规候选"节，**不计入常规侵权候选**；维权走 FRAND / 标准强制许可框架 |
   | 未声明 / 未纳入 | 候选清单不变 | 按常规侵权流程；标准实施类厂商仍可能侵权（标准外的具体优化部分） |
   | 不明 / 工具能力不足 | 候选清单不变，每个标准实施类候选标"标准合规争议" | verdict 中明示"标准合规状态需法务确认"，**所有标准实施类候选标'待标准合规状态确认'**而非"确认侵权" |
