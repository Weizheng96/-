## 领域分类

1. **主领域**：通信 / 云 / 网络
2. **次领域 / 子方向**：
   - **数据中心网络虚拟化**（vSwitch / virtio / SR-IOV / VF passthrough）
   - **智能网卡（SmartNIC）/ 数据处理单元（DPU）硬件流表卸载**（fast-path offload）
   - **链路聚合 / 多 NIC 高可用（LAG / bond / team / multi-NIC reliability）**
   - **公有云 / 私有云数据面**（IaaS overlay 网络转发面）

## R-* 标志判定（决定 Step 5 激活规则）

| 标志 | 触发判定 | 理由 | 激活效应 |
| --- | --- | --- | --- |
| **R-OPENSOURCE** | **TRUE** | vSwitch 领域几乎所有商业产品都基于上游开源（OVS / Linux bonding / DPDK / FRR / OVN）二次开发；多 NIC LAG 是 Linux bonding/team/OVS bond 默认能力 | 上游开源项目本身（OVS、Linux kernel、DPDK、OVN）必须列为 P0 候选；商业发行版（Red Hat / SUSE / Canonical / 国产麒麟 / 统信 + 各 DPU 厂私有 vSwitch）独立评估；按上游 contributor 邮箱域归因（华为、思科、英特尔、英伟达 contributors） |
| **R-CONFIG** | **TRUE** | F4（同步卸载）的开关在 OVS 中是 `hw-offload=true` + `bond_mode=balance-tcp` + `lacp=active` 等；F2 LACP 在 Linux bonding 中是 `mode=4 (802.3ad)`；F3 跨 NIC 第二层聚合常以"bond-of-bonds"或"VF-LAG"等配置形式出现 | Step 5 必须双引证（vendor 文档承认能力 + 配置样例 / Helm chart / startup 参数 / Ansible playbook） |

## Step 3c — R-STANDARD 子流程

**触发判定**：R-STANDARD = PARTIAL。F2（LACP）= IEEE 802.1AX 标准，但属于"标准合规免责"范围，**与本专利新颖点（F3-F4）正交**——LACP 谁都能用，本专利不主张 LACP 本身。

**核查项**：
- 检索"华为 + CN114531405B + SEP declaration / FRAND" → 未检索到 IEEE / ITU / 3GPP SEP 声明
- 检索"H04L45/76 强制国标"、"虚拟交换机流表卸载 国家强制标准 GB" → 未检索到强制国标覆盖本专利新颖点
- 检索"行业准入 多 NIC 硬件流表卸载" → 未检索到行业准入要求

**判定结果**：未声明 / 未纳入。候选清单不变，按常规侵权流程处理。**Step 6 verdict 不需走 FRAND 框架**。
