# CN114531405B 领域适配

## 领域分类

### 1. 主领域

- **主领域**：通信 / 云 / 网络（具体子方向：**虚拟交换机 + LACP 多网卡 bond + 硬件 offload 多卡同步卸载**）
- **次领域**：半导体 / 芯片设计（落入 DPU / SmartNIC fastpath 卸载场景时）

判定依据：权 1 / 11 / 23 / 33 字面限定"虚拟交换机 vSwitch + N ≥ 2 网络接口卡 + LACP 卡内聚合 + 跨卡映射到第一端口 + 精确流表多卡同时卸载"——核心是 NFV / 云数据中心宿主机内的 vSwitch HW offload 高可用方案；权 35 芯片系统权延伸到 DPU / SmartNIC 芯片侧。

### 2. 5 大主体类型清单（针对本领域的产业切片）

按"上游 → 中游 → 下游 → 用户 → 工具方"5 段切片：

| 编号 | 主体类型 | 角色 | 命中独立权 |
|---|---|---|---|
| **S1** | **vSwitch / 数据面软件提供方** | OVS / VPP / 商业 SDN 控制器；实现 bond port + tc-flower / OVS-DOCA offload 的软件栈 | 权 23（装置）+ 权 36（介质）|
| **S2** | **NFV / 云 / 运营商 / 数据中心部署方** | 实际启用多 NIC LACP bond + 全卡 HW offload 的部署运行方 | 权 1 / 11（方法）|

### 4. 适用规则集

| 标志 | 是否激活 | 判定依据 |
|---|---|---|
| **R-OPENSOURCE** | ✅ **TRUE** | OVS 主线 / OVN / Linux kernel bonding driver / mlx5 driver / DPDK rte_eth_bond 全栈开源 → 激活上游开源项目作为独立 P0 候选 + 商业发行版独立评估 + 上游贡献者归因检索 |
| **R-CONFIG** | ✅ **TRUE** | F2 LACP bond 配置 + F3 bond port 抽象 + F5 多卡同时 offload 都体现为具体配置参数（`bond mode=802.3ad`、`ovs-vsctl add-bond`、SR-IOV switchdev mode、bond up-link representors）→ 激活双引证（prose + key:value）|
| **R-PARTNER** | ✅ **TRUE** | 本领域有典型的 vendor × 上游平台厂联合参考方案文化（NVIDIA × Red Hat / VMware vSphere 8 DSE × DPU 厂 等）→ 激活联合案例研究检索 |
| **R-PATENTWALL** | ✅ **TRUE** | H04L45 / H04L49 主分类公开度高；NVIDIA / Mellanox / Intel / Red Hat / Cisco 等在此主分类专利墙活跃 → 激活反向专利墙两轮深读 |
| **R-PROCURE** | ✅ **TRUE** | 运营商 / 政企 / 金融 NFV 集采的招标书会公开技术规格（含 bond / HW offload 要求）→ 激活招标书检索 |
| **R-STANDARD** | ⚠️ **部分 TRUE** | LACP（IEEE 802.3ad / 802.1AX）是国际开放标准——但本专利涉及"跨卡精确流表同步 offload"具体优化，不直接构成 SEP；R-STANDARD 不激活 Step 3c 完整子流程；维持常规侵权流程，§A.5 / §A.15 仍可作为辅助来源 |

### 5. 领域分类对后续步骤的影响

本专利领域分类为"通信 / 云 / 网络（NFV / 虚拟交换 / 多 NIC HW offload）"。因此：

- **Step 3b 场景推演**：围绕 (a) 公有云宿主机多 NIC HA + OVS bond offload；(b) DPU + VF-LAG 跨卡卸载；(c) 运营商 NFV / vEPC LAG；(d) 商业 NSX-T / vSphere DSE LAG；(e) 开源 OVS 主线本身展开 ≥ 5 个场景
- **Step 4 候选清单**：按 S1-S5 5 大主体类型展开（每主体至少 3 家）；R-OPENSOURCE 双层激活——OVS 上游 / DPDK 上游 / Linux kernel mlx5 driver / 商业发行版分别独立 P0 候选
- **Step 5 激活规则**：双引证（R-CONFIG 配置参数 + prose）+ 联合案例研究（R-PARTNER NVIDIA × Red Hat × VMware）+ 反向专利墙两轮（R-PATENTWALL — 重点核查 NVIDIA / Mellanox / Intel 2017-2020 mlx5 commit history 影响新颖性）+ 招标书源（R-PROCURE）+ 上游贡献者归因（R-OPENSOURCE — 本地 git clone + `git log --author="@<vendor>.com"` 跑归因）
- **Step 5 跳过规则**：Step 3c 完整标准合规判定子流程（R-STANDARD 不激活完整子流程，但 LACP 本身是标准这一点会在每候选 verdict 中作为常识背景）
