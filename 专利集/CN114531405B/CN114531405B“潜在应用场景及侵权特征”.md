## 一、潜在应用场景（严格来自背景技术 — 不外扩）

### 场景 1：数据中心计算虚拟化下的虚拟交换机数据转发
- **背景技术 verbatim 依据**：> "在计算虚拟化后，一台主机被虚拟化成多台虚拟机（Virtual Machine，VM），同一主机内的虚拟机通过虚拟交换机（Virtual Switch，VSw），例如通过虚拟交换机实现数据转发"
- **场景说明**：云/数据中心服务器把一台物理机虚拟化成多台 VM，VM 之间以及 VM 与外部网络的流量由宿主机上的虚拟交换机（vSwitch）转发。
- **使用本专利的典型形式**：宿主机虚拟交换机软件（vSwitch / 转发面）+ 流表（flow table）转发逻辑。

### 场景 2：流表硬件卸载交换（SmartNIC / 硬件卸载网卡）
- **背景技术 verbatim 依据**：> "硬件卸载交换技术由于软硬协同机制复杂存在跨网络接口卡可靠性问题，只能实现单一网络接口卡内不同链路的可靠性"
- **场景说明**：为给主机 CPU 减负、提升转发性能，把虚拟交换机的流表转发动作从软件下沉（卸载）到网络接口卡（网卡）硬件中执行，即"硬件卸载交换技术"。
- **使用本专利的典型形式**：可卸载流表的智能网卡 / 硬件加速网卡 + vSwitch 卸载控制面（把"精确流表"下发到网卡硬件）。

### 场景 3：跨网卡高可靠（多网卡冗余 / 消除单网卡单点故障）
- **背景技术 verbatim 依据**：> "只能实现单一网络接口卡内不同链路的可靠性，如果该网络接口卡故障，那么整个系统就故障了，存在单点故障的风险"
- **场景说明**：单网卡卸载存在单点故障——网卡一旦故障整机转发中断。本专利通过把 N（≥2）个网卡聚合为一个逻辑"第一端口"、并把同一精确流表同步卸载到全部 N 个网卡，实现跨网卡冗余可靠性。
- **使用本专利的典型形式**：多网卡 bond / LACP 链路聚合 + 把精确流表同步卸载到聚合内全部网卡的卸载控制逻辑。

## 二、独立权利要求 + 侵权特征 F1-F5

### 权 1 verbatim（独立权 — 方法 / 应用于虚拟交换机）
> 一种流表处理方法，应用于虚拟交换机，所述虚拟交换机连接 M 个虚拟机和 N 个网络接口卡，M、N 为大于或者等于 2 的整数，其特征在于，所述方法包括：建立所述 N 个网络接口卡对应的 N 个逻辑端口的 N 个端口标识与一目标端口标识之间的映射关系，以将所述 N 个逻辑端口聚合为第一端口；其中，每个网络接口卡对应的逻辑端口是每个网络接口卡的物理端口基于链路汇聚控制协议 LACP 聚合形成的逻辑端口；在所述 N 个网络接口卡中的目标网络接口卡查询到所述目标网络接口卡中不存在与所述目标网络接口卡接收到的报文对应的卸载流表时，通过所述第一端口将精确流表卸载至所述 N 个网络接口卡。

### F1-F5 清单
- **F1（多虚机 + 多网卡架构，N≥2）**：> "应用于虚拟交换机，所述虚拟交换机连接 M 个虚拟机和 N 个网络接口卡，M、N 为大于或者等于 2 的整数" —— **关键整数限定：N 个网卡（N≥2），即必须是跨多块网卡的场景，单网卡不命中**。
- **F2（N 个逻辑端口聚合为"第一端口"）**：> "建立所述 N 个网络接口卡对应的 N 个逻辑端口的 N 个端口标识与一目标端口标识之间的映射关系，以将所述 N 个逻辑端口聚合为第一端口"
- **F3（每个网卡逻辑端口由其物理端口基于 LACP 聚合形成）**：> "每个网络接口卡对应的逻辑端口是每个网络接口卡的物理端口基于链路汇聚控制协议 LACP 聚合形成的逻辑端口"
- **F4（卸载流表 miss 作为触发条件）**：> "在所述 N 个网络接口卡中的目标网络接口卡查询到所述目标网络接口卡中不存在与所述目标网络接口卡接收到的报文对应的卸载流表时"
- **F5（精确流表跨全部 N 个网卡卸载）**：> "通过所述第一端口将精确流表卸载至所述 N 个网络接口卡" —— **关键限定：卸载至"N 个"网卡（全部），即一次 miss 触发把同一精确流表同步下发到聚合内所有网卡，而非只下发命中的那一块**。

**拆分说明**：权 1 字面"其特征在于，所述方法包括"后含 3 个并列动作（映射聚合 / LACP 来源限定 / miss 触发下卸载至全部网卡）；F1 为前序架构限定，F3 为对 F2 逻辑端口来源的进一步限定，F4 为 F5 的触发条件。判定时 F2+F5 是核心创新点（多网卡聚合 + 流表向全部网卡同步卸载），F3（LACP）与 F1（N≥2）是硬限定。

### 其他独立权（存在多条独立权）
- **权 11（方法 / 应用于物理机）**：与权 1 同一机制，但视角放到"物理机（主机+N 个网卡）"，并显式增加"目标网卡接收报文（来自物理交换机或目标虚拟机）"步骤。差异在主体（物理机整机）与报文来源描述，核心 F2/F3/F5 一致。
- **权 23（装置 / 应用于虚拟交换机）**：权 1 的装置化（建立单元 + 卸载单元），特征与权 1 一一对应。
- **权 33（物理机整机）**：包含权 23 装置的物理机，机制同权 11。
- **权 34（物理机硬件型）**：处理器+存储器+通信接口执行权 1-10 或 11-22 方法。
- **权 35（芯片系统）**：芯片系统执行权 1-10 或 11-22 方法。
- **权 36（计算机存储介质）**：存储介质承载执行权 1-10 或 11-22 方法的程序。

> 说明：权 34/35/36 为标准"装置/芯片/介质"载体型独立权，落入与否完全取决于其承载的方法是否命中权 1/权 11 的 F1-F5；判定时以权 1 的 F1-F5 为准绳即可。

### 关键限定词与隐含约束
- **N ≥ 2（多网卡）**：N 个网络接口卡，N 为大于等于 2 的整数——单网卡卸载方案不落入本专利（这是与现有单网卡硬件卸载方案的区分点）。
- **M ≥ 2（多虚机）**：M 个虚拟机，M 为大于等于 2 的整数。
- **LACP 限定（F3）**：聚合必须"基于链路汇聚控制协议 LACP"形成逻辑端口；若仅用静态 bond / 非 LACP 聚合需个案判断是否等同。
- **跨网卡同步卸载（F5）**：精确流表要卸载至"N 个"（全部）网卡，而非仅故障切换时被动复制——同步到全部网卡是消除单点故障的关键限定。
- **触发时机（F4）**：以"目标网卡 miss（查不到卸载流表）"作为卸载触发，属 cache-miss-driven offload 语义。
