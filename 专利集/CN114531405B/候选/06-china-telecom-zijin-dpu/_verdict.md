## 关键反向证据（黄金证据）

**子 agent A 从《2025 AI 网络白皮书》p.60 抽出原文**：
> "**样本面：服务器使用 2×100G 双口接入两台 Leaf 交换机**，对于 GPU 服务器的接入，**bond 采用 mode 1 主备方式**，对于 HPFS 存储服务器，由于可以运行自研操作系统，**实现 arp 双发，接入设计上采用去堆叠方案，服务器 bond 模式使用 mode4**"

> "**业务面：……服务器采用 2×25G 紫金 DPU**，并上联接入天翼云自研交换机"

逐项对比专利 F1-F6：
- **GPU 服务器**：`bond mode 1 主备` —— 直接命中专利明确排除的相邻方案"多网卡 active/standby 主备方案"。F5 反向。
- **HPFS 服务器**：`bond mode 4 + ARP 双发 + 去堆叠`：
  - mode 4 = Linux bonding driver 802.3ad LACP，是**主机 OS 层 LACP**，不是"NIC 内部物理端口"基于 LACP 聚合 → F3 字面不命中
  - "去堆叠"= 双卡上联**不同 Leaf 交换机**，不是聚合到同一目标端口标识 → F4 反向
  - "ARP 双发"是协议层方案，靠对端交换机 LACP 哈希哈希分流，**不是把同一精确流表复制到两张 NIC 同时分发** → F5 字面 + 等同均不命中

## 落入专利明确排除的相邻方案
GPU 服务器场景命中专利"明显不在保护范围内的相邻方案"清单：
- **第 2 项 多网卡 active/standby 主备方案**（mode 1 = active-backup）—— **直接字面命中相邻方案**

HPFS 场景命中：
- **第 5 项 仅在 NIC 内部做 LACP，但 vSwitch 没有第二级"目标端口标识"映射**（甚至更弱：连 NIC 内部都不 LACP，是 OS 层 LACP；vSwitch 也无第二级映射；甚至双卡上联**不同 Leaf**，不是同一上联端口）

## 最终结论
**中国电信紫金 DPU 在权 1 / 11 部署侧已排除**。强反向证据如下：
- GPU 接入面字面 = active/standby 主备（mode 1），直接命中专利明确排除的相邻方案
- HPFS 接入面 = OS 层 LACP + 去堆叠（双卡跨 Leaf），同样不属于"NIC 内部 LACP + vSwitch 同一目标端口聚合"
- 业务面（2×25G 紫金 DPU）的具体 bond 模式白皮书未明，**但前述两个面已足以确认天翼云在该方向的整体路线选择是传统 Linux bonding，不是专利保护的"NIC 内部 LACP + 多卡同表分发"**

> 注意：本结论仅针对**部署侧**。**紫金 DPU 芯片侧（权 35）**仍维持"公开资料没有足够证据"——紫金 DPU 固件 / SDK 是否含 F1-F6 实现代码仍是黑盒（参考违约列表 §1）。客户用 mode 1 部署不代表芯片不能 mode 4 active/active；但目前公开资料无法穿透到芯片层。

## 升级前提
若中国电信将 GPU/HPFS 部署改为"vSwitch + NIC 内部 LACP + 同表多卡分发"模式，需重新评估。截至 2026-04-27，2025 AI 网络白皮书反映的部署架构稳定在 mode 1 / mode 4 + 去堆叠路径上。
