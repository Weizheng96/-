## 摘要
本发明实施例公开了一种应用于虚拟交换机的流表处理方法。虚拟交换机连接 M 个虚拟机和 N 个网络接口卡（NIC），方法包括：建立 N 个网络接口卡对应的 N 个逻辑端口的 N 个端口标识与一目标端口标识之间的映射关系，将 N 个逻辑端口聚合为第一端口；每个网络接口卡对应的逻辑端口通过链路汇聚控制协议（LACP）聚合形成；通过该第一端口将精确流表卸载至 N 个网络接口卡。该方案实现多网络接口卡同时转发与卸载，提高系统可靠性。

## 技术领域
通信技术领域，尤其涉及虚拟交换机的流表处理。

## 背景技术与要解决的问题
现有硬件卸载交换技术（如 OVS Offload、SR-IOV Bond、弹性虚拟交换机 eVS 等）将虚拟交换机连接到多个网络接口卡时，由于不同网络接口卡对应不同的逻辑端口标识，**会基于多个网络接口卡的多个不同逻辑链路形成数据回环**，导致物理交换机与虚拟交换机间出现环路；此外，单网络接口卡硬件直通时该网卡故障会直接影响虚拟机连通性。

## 核心技术方案（两级绑定）
1. **第一级绑定**：每个网络接口卡内部多个物理端口基于 LACP 聚合形成一个"逻辑端口"。
2. **第二级绑定**：虚拟交换机建立 N 个网络接口卡对应的 N 个逻辑端口标识与**同一目标端口标识**之间的映射关系，将 N 个逻辑端口聚合为"第一端口"。
3. **多卡同时卸载**：当某个网络接口卡查询到不存在与该报文对应的卸载流表时，虚拟交换机通过第一端口**把同一精确流表同时卸载到全部 N 个网络接口卡**，使任一网卡都能就地处理后续同流报文。
4. **故障切换**：网卡故障时从 Bond 中摘除并将其卸载流表置为无效；恢复时重新入 Bond 并从其他网卡同步卸载流表；硬直通故障时支持自动切换为软直通。
5. **零拷贝**：在主机与网卡之间支持零拷贝数据流传输。

## 有益效果
- 避免多网卡多逻辑端口标识造成的数据回环。
- 多网卡同时具备同一精确流表，单网卡故障可平滑切换，提升报文处理可靠性。
- 支持硬直通 / 软直通的运行时切换。
- 通过零拷贝降低主机侧 CPU 与内存开销。

## 系统架构 / 实施例
物理机由主机和 N 个网络接口卡组成：
- 主机上运行虚拟交换机（含 **核心转发模块、vSW-Bond 模块、Vhostdp 软直通模块**）以及 M 个虚拟机。
- 每个网络接口卡含硬件 Datapath、Bond、虚拟功能 VF。
- 三种 IO 绑定模块：
  - **VM Bond**（虚拟机内）：硬直通时聚合多个 NIC 的 VF。
  - **VF Bond**（虚拟交换机内）：软直通时聚合多个 NIC 的 VF。
  - **vSW-Bond**（虚拟交换机）：将多个网卡的 Bond 进一步聚合为虚拟交换机的"上联端口"。

### 流表处理主要流程（图 7~图 11）
- S701 建立 N 逻辑端口与目标端口标识的映射，聚合为第一端口。
- S702 虚拟机经 VM Bond / VF Bond 基于哈希选择 VF 发送报文。
- S703 网卡查询本地卸载流表是否存在。
- S704 若不存在，将报文/上送消息通过第一端口送至虚拟交换机。
- S705~S706 虚拟交换机执行慢路径学习，生成精确流表。
- S707 判定精确流表是否满足卸载条件（连接稳定、流速率 ≥ 阈值等）。
- S708 通过第一端口将精确流表**同时**卸载至全部 N 个网卡。

### 多网卡流表一致性
- 虚拟交换机的精确流表构成"精确流表集合"；N 个网卡的卸载流表构成"卸载流表集合"，**N 个卸载流表集合内容互相一致**。
- 删除时虚拟交换机向 N 个网卡同步发删除指令。
- 检测到预设条件时（流过期 / 拓扑变化）下发更新指令。
- 检测到某网卡异常 → 标记其卸载流表无效并从 Bond 摘除；恢复时重新加入并补齐流表。

## 权利要求

### 独立权利要求

**权利要求 1（虚拟交换机侧方法）**
一种流表处理方法，应用于虚拟交换机。虚拟交换机连接 M 个虚拟机和 N 个网络接口卡，M、N ≥ 2。该方法包括：
- 建立 N 个网络接口卡对应的 N 个逻辑端口的 N 个端口标识与一目标端口标识之间的映射关系，将 N 个逻辑端口聚合为第一端口；其中每个网络接口卡的逻辑端口由该网卡的物理端口基于 LACP 聚合形成；
- 当 N 个网卡中的目标网卡查询不到与其接收报文对应的卸载流表时，**通过该第一端口将精确流表卸载至全部 N 个网络接口卡**。

**权利要求 11（物理机侧方法）**
一种流表处理方法，应用于物理机。物理机含主机和 N 个网卡，主机上运行虚拟交换机和 M 个虚拟机，N 个网卡通过主机接口连主机、通过网络接口连外网，M、N ≥ 2。方法包括：
- 虚拟交换机建立同上的两级聚合映射；
- 目标网卡接收来自外部物理交换机或目标虚拟机的报文；
- 在目标网卡查询不到对应卸载流表时，虚拟交换机通过第一端口将该报文对应的精确流表卸载至全部 N 个网卡。

**权利要求 23（流表处理装置）**：实现权 1 方法的"建立单元 + 卸载单元"装置。

**权利要求 33**：包含权 23-32 装置的物理机。

**权利要求 34**：处理器+存储器+通信接口物理机，调用代码执行权 1-10 或 11-22 方法。

**权利要求 35**：芯片系统，含至少一个处理器、存储器、接口电路，执行权 1-10 或 11-22 方法。

**权利要求 36**：计算机存储介质，存储被处理器执行后实现权 1-10 或 11-22 方法的程序。

## 必要技术特征清单（用于侵权对比的"尺子"）
落入权利要求 1 / 11 保护范围必须**全部满足**：

| 编号 | 必要技术特征 |
| --- | --- |
| F1 | 主体是**虚拟交换机**（运行在物理机/主机上） |
| F2 | 一台物理机连接 **M ≥ 2 台虚拟机** 和 **N ≥ 2 张网络接口卡（NIC）** |
| F3 | **每张 NIC 的多个物理端口基于 LACP** 在 NIC 内部聚合为该 NIC 的"逻辑端口" |
| F4 | 虚拟交换机将 **N 张 NIC 的 N 个逻辑端口标识 → 同一目标端口标识** 建立映射，**对外聚合为同一个"第一端口"** |
| F5 | 当目标 NIC 查询不到对应卸载流表（即慢路径触发）时，虚拟交换机**通过该第一端口** 把精确流表 **同时下发到 N 张 NIC** |
| F6 | 应用场景是**硬件流表卸载（offload）**：网卡侧持有"卸载流表"用于快路径处理 |

进一步增强（与从属权 4-10 / 14-22 对应，作为加重命中证据）：
- 卸载条件：连接稳定、流速率门限。
- N 张网卡的卸载流表集合互相一致。
- 网卡故障 / 恢复时的同步管理（摘除、置无效、重新加入复制）。
- 软直通 / 硬直通模式与切换（VM Bond / VF Bond / vSW-Bond）。
- 主机—网卡间零拷贝。

## 等同特征讨论
可能落入等同保护范围的替换方案：
- 第一级聚合协议不是 LACP 而是其他链路聚合协议（PAGP、静态 trunk）→ 由于权 1 明确写"基于 LACP"，**严格直接侵权可能不成立**，但等同侵权仍可能讨论。
- 把"第一端口"换称为"上联 bond"、"uplink port-channel" 或在 OvS 中表现为 **bond port** 同名实现 → 仍属同一手段。
- 卸载主体不是 OvS 而是其他用户态/内核态 vSwitch（如 VPP、华为 eVS、阿里 AVS、腾讯 vSwitch）→ 名称不同但只要满足 F1-F6 仍命中。
- 卸载目标不是 ASIC NIC 而是 SmartNIC / DPU / IPU（Mellanox / NVIDIA BlueField、Intel IPU、AWS Nitro、阿里 CIPU、华为擎天），属于权 1 的"网络接口卡"。
- "目标端口标识"在不同实现里可能叫 ofport / dp_port / virtual port id，名义不同实质相同。

## 明显不在保护范围内的相邻方案
- 单网卡（N=1）方案，无论是否做 LACP，都不命中 F2/F4。
- 多网卡 active/standby 主备方案：只有一张主 NIC 在转发卸载，未做"N 张同时卸载同一精确流表"，不命中 F5。
- 纯软件 OvS（无任何卸载到 NIC），不命中 F6。
- 物理交换机层面的 MLAG / Stack：与"虚拟交换机—网卡"卸载场景无关。
- 仅在 NIC 内部做 LACP（无第二级 vSwitch 端口聚合到同一目标端口标识），不命中 F4。
