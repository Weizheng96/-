## 排除清单

- **专利权人**：华为技术有限公司（含全资 / 控股子公司，例如海思半导体、华为云、华为数字能源等业务集团）—— 自营无侵权风险
- **明确被授权方**：截至 2026-05-11 未公开任何 CN114531405B 的对外许可方；保守假设无授权
- **F4 标准合规免责的部分**：F2 LACP（IEEE 802.1AX 实施）部分不视为侵权——但本专利新颖点在 F3-F4 跨 NIC 二层映射 + 同步卸载，**与 F2 标准合规正交**

## 全局 R-2 拓扑变体说明（F1 整数限定 N ≥ 2）

本专利 F1 字面要求 **N ≥ 2 张 NIC**。绝大多数 vendor 默认产品形态以"单 NIC 多端口"作为常见示例，"≥ 2 NIC 跨卡聚合"是一个独立的部署变体——按 R-2 强制拆分两个候选（topology-A "单 NIC 多端口"通常预排除或低档；topology-B "多 NIC 跨卡"才是命中候选）。

## 组织清单（按 5 大主体类型分组）

### A. vSwitch 软件方 — 上游开源 + 商业发行版（R-3 双层激活）

#### A.1 Open vSwitch 上游项目（openvswitch.org）— **R-OPENSOURCE 上游 P0**

- 类型：开源项目（Linux Foundation 项目；主 maintainer 含 Red Hat / NVIDIA / VMware / Intel / 华为等 contributor）
- 主体类型：A. vSwitch 软件方（上游）
- 关联独立权：权 1, 11, 23, 36
- 产品族二级条目：
  - **OVS-vswitchd + OVS-DPDK + bonding 模块（含 hw-offload 接口）**：`01-ovs-upstream-bond` — 上游内置 LACP bond + bond_mode=balance-tcp + ovs-tc hw-offload；为下游所有发行版/DPU 厂的二次开发提供基础设施
  - **OVN 上游（含 OVN-IC、OVN multi-NIC 资源调度）**：`02-ovn-upstream-multinode` — OVN 在多 NIC 编排维度的上游能力
- 命中场景：场景 1 / 2 / 3 / 4 / 5
- 合理性：OVS 是数据中心 vSwitch 事实上的上游标准；其 bond + hw-offload + tc-flower 接口是所有商业 vSwitch 二次开发的基础；F2-F4 各 satisfaction paths 的 path 4.2 / 4.3 都依赖 OVS bond + hw-offload 机制
- 可能性等级：**高**（但 R-2 拓扑细分后：单 NIC 多端口 LACP 不命中 F1；bond-of-bonds 跨 NIC 才命中）
- 检索关键词：`ovs bond hw-offload`、`ovs-tc bond_mode balance-tcp lacp active`、`ovs bonding multi-nic`、`open vswitch bond-of-bonds`、`ovs-vsctl bond_mode hw_offload`

#### A.2 Linux Kernel — bonding / team / tc-flower offload（kernel.org）— **R-OPENSOURCE 上游 P0**

- 类型：开源项目（Linux Foundation；netdev maintainer 含 Red Hat / Intel / NVIDIA / 华为 / 阿里等）
- 主体类型：A. vSwitch 软件方（上游） + B. 硬件适配层（NIC driver）
- 关联独立权：权 1, 11, 23, 36
- 产品族二级条目：
  - **Linux bonding driver (mode=4 / 802.3ad LACP) + tc-flower offload + mlx5 / ice / nfp / efct 等驱动支持的 hw-offload**：`02-linux-kernel-bonding-mlx5-lag` — 含 mlx5 VF-LAG（NVIDIA ConnectX/BlueField）、ice E810 / E830（Intel）、nfp（Netronome / 已被华芯收购的 SmartNIC）、efct（Marvell）等驱动
  - **team driver（teamd） + tc-flower offload**：`02b-linux-kernel-team-driver` — `teamd` 用户态控制器 + 内核 team driver 提供与 bonding 类似但更灵活的 LAG，tc-flower 同样适用
- 命中场景：场景 1 / 2 / 3 / 4 / 5
- 合理性：内核 mlx5 驱动从 4.18 起原生支持 VF-LAG（实质是 F2-F3 的 satisfaction path 3.4）；tc-flower offload 是 F4 的 satisfaction path 4.2 的内核侧载体；任何使用 mainline Linux + 上游 mlx5 / ice 驱动的发行版都被动包含此能力
- 可能性等级：**高**（同样 R-2 区分：mlx5 VF-LAG 默认是单卡多端口 vs 多卡 multi-host LAG —— 拆分判定）
- 检索关键词：`mlx5 VF-LAG bond_mode 802.3ad`、`tc flower offload bonding`、`linux kernel bonding hw-offload`、`team driver tc-flower offload`、`netdev bonding multi-nic`

#### A.3 DPDK 上游（dpdk.org）

- 类型：开源项目（Linux Foundation；maintainer 含 Intel / NVIDIA / Marvell / 中兴 / 华为）
- 主体类型：A. vSwitch 软件方（用户态 DPlane 上游）
- 关联独立权：权 1, 23, 36
- 产品族二级条目：
  - **DPDK link-bonding PMD (rte_eth_bond) + rte_flow hw-offload**：`03-dpdk-bonding-rte-flow` — 用户态 LACP + LAG 实现，支持 mode=4 + tx_load_balance=balance-l34
  - **OVS-DPDK 上游集成（OVS user-space datapath with DPDK）**：`03b-ovs-dpdk-integration` — 与 A.1 的内核数据面互补，覆盖电信 NFV 场景
- 命中场景：场景 1 / 2 / 5
- 合理性：DPDK link-bonding PMD 实现 LACP 多 NIC 聚合；rte_flow hw-offload 是 F4 在用户态 DPlane 的 satisfaction path；任何高性能 vSwitch（OVS-DPDK / VPP / 自研 DPlane）都依赖之
- 可能性等级：**中**（与 A.1 部分重叠；独立性体现在 user-space vs kernel 数据面）
- 检索关键词：`dpdk rte_eth_bond LACP`、`rte_flow hw-offload`、`dpdk bonding multi-nic`、`OVS-DPDK link aggregation`
