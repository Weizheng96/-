# CN114531405B 专利介绍

- 标题：一种流表处理方法及相关设备
- 申请人 / 专利权人：**华为技术有限公司**
- 发明人：李力军、周敬滨、李晓
- 申请日 / 公开日 / 授权日：**2020-10-31 / 2022-05-24 / 2023-06-06**（预期失效日 2040-10-31）
- 法律状态（截至检索日 2026-04-27）：**有效（Active）**
- 主分类号（IPC）：H04L45/76、H04L49/70、H04L45/745、G06F9/45558、H04L45/245

## 一句话概括
该专利通过"NIC 内部 LACP 聚合 + 虚拟交换机把多张网卡的逻辑端口再聚合到同一目标端口标识（第一端口）"的**两级绑定**架构，把同一份精确流表**同时卸载到全部 N 张网卡**，从而消除多网卡逻辑端口标识不一致带来的数据回环、并在任一网卡故障时保持业务可靠转发。

## 解决的技术问题
- 现有硬件流表卸载（如 OVS Offload、SR-IOV Bond、弹性虚拟交换机 eVS）一旦把虚拟交换机连接到多张网卡，因为不同网卡对外是不同的逻辑端口标识，**容易在物理交换机和虚拟交换机之间形成数据回环**。
- 单网卡硬直通时，该网卡故障 = 虚拟机断网，**多网卡冗余 + 流表同步**机制缺失。
- 卸载流表跨网卡保持一致性、网卡上下线时的同步管理、以及软/硬直通运行时切换，缺乏统一处理。

## 核心技术方案（要点）
1. **第一级聚合（NIC 内）**：每张网卡的多个物理端口基于 **LACP** 聚合为该网卡的一个逻辑端口。
2. **第二级聚合（vSwitch）**：虚拟交换机将 N 张网卡的 N 个逻辑端口标识 → **同一目标端口标识**，对外表现为单一的"第一端口"。
3. **慢路径学习 + 多网卡同时卸载**：当目标网卡查不到匹配的卸载流表时，把报文/上送消息送到虚拟交换机；虚拟交换机做慢路径学习，生成精确流表，**通过第一端口将该精确流表同时下发到全部 N 张网卡**，使任一网卡都拥有相同的卸载流表。
4. **流表一致性管理**：删除 / 更新 / 异常 / 恢复 时，虚拟交换机统一向 N 张网卡下发删除、更新、置无效、重新同步指令，保证 N 个卸载流表集合内容一致。
5. **三种 IO 绑定模块**：
   - VM Bond（虚拟机内、硬直通）：聚合多 NIC 的 VF；
   - VF Bond（虚拟交换机内、软直通）：聚合多 NIC 的 VF；
   - vSW-Bond（虚拟交换机内）：把多 NIC 的 Bond 再聚合为上联第一端口；
   - 支持硬直通 ↔ 软直通的运行时切换。
6. **卸载条件**：精确流表满足"连接稳定 + 流速率达到阈值"等条件后才下发，避免低价值流表挤占网卡硬件表项。
7. **零拷贝**：在主机与网卡之间使用零拷贝传输数据流。

## 关键权利要求解读

### 权利要求 1（核心独立权 — 虚拟交换机方法）
独立权 1 包含两段限定特征，可以拆成 6 个**必要技术特征**（全部满足才落入保护范围）：

| 编号 | 特征 | 解读 |
| --- | --- | --- |
| F1 | 应用于**虚拟交换机** | 主体是 vSwitch；纯硬件交换机或纯应用层网关不算 |
| F2 | 虚拟交换机连接 **M ≥ 2 个虚拟机** 和 **N ≥ 2 张网络接口卡** | 必须是多 VM、多 NIC，N=1 直接出局 |
| F3 | **每张 NIC 内部多个物理端口基于 LACP 聚合**，形成该 NIC 的逻辑端口 | "基于 LACP" 是字面限定，非 LACP 的链路聚合需走等同侵权 |
| F4 | 虚拟交换机把 **N 个逻辑端口的 N 个端口标识** 与**同一目标端口标识**建立映射 → **聚合为第一端口** | 第二级绑定，是该专利与单纯 LACP 方案的关键区别 |
| F5 | 在某 NIC 查询不到对应卸载流表时，**通过该第一端口将精确流表卸载至全部 N 张 NIC** | "同时多卡卸载"是核心新颖性 |
| F6 | 隐含场景：**网卡侧具有卸载流表，用于硬件快路径处理** | 即必须是 NIC 硬件 offload 而不是纯软件转发 |

### 权利要求 11（独立权 — 物理机方法）
把权 1 的视角搬到物理机层面，并增加了"目标网卡接收报文（来自外部物理交换机或来自虚拟机）"的约束。要点同 F1-F6。

### 权利要求 23 / 33-36
- 权 23 是装置版本（建立单元 + 卸载单元）；
- 权 33 是包含该装置的物理机；
- 权 34/35/36 分别覆盖**处理器+存储器组合**、**芯片系统**、**计算机存储介质** —— 这意味着 SmartNIC / DPU / IPU 厂商的芯片产品本身、以及含上述方法软件的产品形态都在保护范围内。

### 关键从属权要点
- 权 4 / 16：流表"卸载条件"（稳定 + 速率阈值）；
- 权 5 / 17：精确流表集合 ↔ 卸载流表集合的对应关系；
- 权 6-9 / 18-21：流表删除、更新、异常、恢复的多卡同步；
- 权 10 / 22：主机—网卡零拷贝；
- 权 15：明确"软直通 / 硬直通"两种工作模式与切换。

## 保护范围边界（重要）

### 必要技术特征清单（缺一不可）
1. **虚拟交换机**为主体（F1）。
2. **多 VM 多 NIC**（M ≥ 2、N ≥ 2，F2）。
3. **NIC 内部物理端口的 LACP 聚合**形成逻辑端口（F3）。
4. **虚拟交换机第二级聚合**：N 个逻辑端口标识映射至同一目标端口标识 / 第一端口（F4）。
5. **慢路径学习触发后，把同一精确流表同时下发给全部 N 张 NIC**（F5）。
6. NIC 侧具有**硬件 offload 流表（卸载流表 / 快路径表）**（F6）。

### 等同特征讨论（可能仍落入保护范围）
- LACP 替换为其他链路聚合（静态 trunk、PAGP、Hash-LB 等）—— 字面不命中 F3，但等同侵权可讨论。
- "第一端口"在实现里被叫作 uplink bond / port-channel / ofbond，仅命名差别。
- 卸载目标从普通 SmartNIC 替换为 DPU / IPU（NVIDIA BlueField、Intel IPU/Mount Evans、AWS Nitro、阿里 CIPU、华为擎天）—— 仍属"网络接口卡"。
- 把 OvS 换成 VPP、自研 vSwitch（华为 eVS、阿里 AVS、腾讯 vSwitch、字节 vSwitch、Cilium-bpf vSwitch）—— 仍属虚拟交换机。
- "精确流表"称呼为 megaflow、exact-match flow、cache-line flow、SmartNIC fast-path entry，仅命名差别。
- 卸载条件以"连接数 / 报文数 / TCP 连接稳定性 / 流寿命"等表达 —— 仍属 F4 等同。

### 明显不在保护范围内的相邻方案
- 单网卡（N=1）+ LACP：不满足 F2/F4。
- 多网卡 **active/standby**（主备）：只有主网卡持有流表，不满足 F5 的"同时下发到 N 张"。
- 纯软件 vSwitch（无任何 NIC 卸载）：不满足 F6。
- 物理交换机侧的 MLAG / VPC / Stacking：不在虚拟交换机—网卡卸载场景。
- 仅在 NIC 内部做 LACP，但虚拟交换机没有第二级"目标端口标识"映射：不满足 F4。
- 多 NIC 各自独立维护流表（不要求集合一致）：F5 / 权 5 不满足。
