## 一句话概括

虚拟交换机连接 M ≥ 2 VM + N ≥ 2 网络接口卡时，把**每张网卡内部物理端口先用 LACP 聚合为该网卡的逻辑端口**，再把 N 个逻辑端口的端口标识通过映射**聚合为统一的"第一端口"**；当任一网卡接收报文但本地无对应卸载流表时，vSwitch 通过第一端口**将精确流表同时下发到所有 N 张网卡**——实现多网卡的转发与卸载，避免单网卡硬件 offload 单点故障。

## 解决的技术问题

现有硬件卸载交换技术（HW offload）多基于单网卡：流表只卸载到目标网卡（接收对应报文那一张），当目标网卡故障 / 链路断时，已卸载的流表丢失，HW offload 路径中断。本专利通过**两层端口聚合 + 多网卡同时卸载机制**规避单点故障。

## 核心技术方案

1. **两层端口聚合架构**：
   - 第一层（卡内）：每张网卡的多个物理端口基于 **LACP**（802.3ad）聚合为一个**逻辑端口**
   - 第二层（卡间）：vSwitch 把 N 张网卡的 N 个逻辑端口通过端口标识映射**聚合为第一端口**
2. **目标网卡 miss 触发**：当某张目标网卡接收报文但查无对应卸载流表 → 触发 vSwitch 控制面下发动作
3. **多网卡同时卸载精确流表**：vSwitch 通过第一端口把**同一精确流表同时下发到所有 N 张网卡**——之后任何网卡接收同流报文都能命中
4. **配套机制**（从属权扩展）：精确流表满足卸载条件（VM 连接稳定 + 流速率 ≥ 阈值）；卸载流表与精确流表一一对应；任一卡上目标卸载流表被删除时向其他卡同步删除；预设条件（慢路径变更 / 连接老化 / 端口删除 / TCP 状态变化）触发更新；网卡状态异常时解除聚合 + 标记无效，状态恢复时重新聚合
5. **第二端口（从属权 3）**：N 个网卡对应的 N 个虚拟功能 (VF) 在 vSwitch 内聚合成的逻辑端口——用于接收 VM 发送的报文并分发到 N 个 VF
6. **零拷贝（从属权 10）**：VM 虚拟网卡 K 队列对应 N 个 VF（K 为 N 整数倍，每 VF 负责 K/N 个队列），从 VM 内存直接拷贝到网卡内存

## 关键权利要求解读

### 权 1（方法权·vSwitch 视角）字面五段流程

1. 前置硬件配置：vSwitch 连接 **M ≥ 2 个 VM** 和 **N ≥ 2 个网络接口卡**
2. 建立映射：N 个网卡对应的 **N 个逻辑端口的 N 个端口标识 ↔ 一目标端口标识** 的映射 → 把 N 个逻辑端口聚合为**第一端口**
3. 卡内 LACP：每个网卡的逻辑端口 = 该网卡的物理端口基于 **LACP** 聚合形成
4. 目标网卡 miss：N 个网卡中目标网卡接收报文 + 查询本地不存在对应卸载流表
5. 同时卸载：vSwitch 通过第一端口将**精确流表卸载至所有 N 个网卡**

### 各独立权特有限定

- **权 11**（物理机视角）：额外限定"主机接口 / 网络接口"——网卡通过主机接口与主机相连、通过网络接口与外部网络相连；目标网卡接收的报文来源是物理交换机或目标 VM
- **权 23**（装置）：额外限定"建立单元 + 卸载单元"——必须有可识别的两个功能模块
- **权 33**（物理机·含装置）：额外限定整机三方联动——vSwitch + 目标网卡 + 目标 VM 在整机中共存
- **权 34**（处理器+存储器机器）：限定 **处理器 + 存储器 + 通信接口** + 程序代码
- **权 35**（芯片系统）：限定 **处理器 + 存储器 + 接口电路** + 指令；处理器与接口电路通过线路互联
- **权 36**（存储介质）：限定计算机程序载体

### 从属权扩展（仅列与 F# satisfaction paths 相关的关键 alt）

- **权 2 alt**：目标网卡 miss 时向 vSwitch 发送消息（含报文 + VF 标识），vSwitch 基于报文 + VF 标识确定精确流表
- **权 3 alt**：vSwitch 通过**第二端口**接收 VM 报文；第二端口 = N 个 VF 在 vSwitch 内聚合的逻辑端口；vSwitch 将报文发送至目标 VF
- **权 4 alt**：精确流表满足**卸载条件**——VM 连接稳定 / 流速率 ≥ 预设阈值（任一）
- **权 5 alt**：精确流表集合 ↔ N 个卸载流表集合 一一对应；N 个卸载流表（同一精确流表对应的）相同
- **权 6 alt**：第一网卡删除目标卸载流表 → 向其他 N-1 张网卡发删除指令
- **权 7 alt**：预设条件触发更新（慢路径变更 / 连接老化 / VM 端口删除 / TCP 状态变化）
- **权 8 alt**：网卡状态异常 → 解除聚合 + 标记卸载流表无效；权 9 状态恢复 → 重新聚合
- **权 10 alt**：K 队列 + N VF 对应（K 为 N 整数倍，每 VF 负责 K/N 队列）+ 零拷贝

### 每条 F# 的 satisfaction paths

```
F1（M ≥ 2 VM + N ≥ 2 网络接口卡 + vSwitch）：
  Path 1.1（独立权字面）：宿主机有 vSwitch + ≥ 2 张独立物理 NIC 卡 + ≥ 2 个 VM
  Path 1.2（等同）：DPU/SmartNIC 部署中 ≥ 2 张 DPU 卡 + 每卡承担独立 vSwitch fastpath + ≥ 2 VM
  Path 1.3（拓扑变体边界 — 不命中）：N=1 单 NIC 部署（即使该 NIC 有 dual-PF）不命中——专利"避免单卡故障"动机隐含 N ≥ 2 必须是独立物理卡

F2（卡内物理端口 LACP 聚合）：
  Path 2.1（独立权字面）：每张 NIC 卡内部有 ≥ 2 个物理端口，通过 802.3ad LACP 协议绑定为一个逻辑端口
  Path 2.2（从属权 5 alt）：该逻辑端口同时维护"卸载流表集合"与 vSwitch 精确流表集合一一对应
  Path 2.3（等同）：静态 802.3ad LAG（无 LACP 协议交互的静态绑定）—— 等同度需个案分析

F3（端口标识映射 → 第一端口）：
  Path 3.1（独立权字面）：vSwitch 内部端口表把 N 个逻辑端口的端口 ID → 1 个 target port ID
  Path 3.2（从属权 3 alt — 注意"第二端口"是不同概念）：vSwitch 内还有"第二端口" = N 个 VF 在 vSwitch 内聚合的逻辑端口（接收 VM 报文用）；第一端口 vs 第二端口 是两个独立抽象
  Path 3.3（等同）：Linux team driver master + bond / OVS bond port + 控制面端口表

F4（目标网卡 miss 卸载流表触发）：
  Path 4.1（独立权字面）：HW offload 第一包查询失败 → upcall 到控制面
  Path 4.2（从属权 2 alt）：目标网卡发送消息携带"报文 + VF 标识"给 vSwitch，vSwitch 基于二者确定精确流表
  Path 4.3（等同）：tc-flower miss handler、OVS dpif-netdev miss → upcall、DPU eSwitch miss-to-host

F5（同时下发到所有 N 张网卡）：
  Path 5.1（独立权字面 + 从属权 5）：同一精确流表生成 N 份相同的卸载流表，分别下发到 N 张卡
  Path 5.2（从属权 6/7/9 alt）：与同步删除指令 / 更新指令 / 状态恢复重新卸载 配套形成完整生命周期
  Path 5.3（等同）：mirror/replicate hardware flow table to all bond slaves；VF-LAG offload to multiple PFs (但需 PFs 是 ≥ 2 张独立卡)
```

### 等同特征讨论

- "网络接口卡" vs "DPU/SmartNIC 整卡" —— 等同（DPU 在主机视角下表现为 NIC）
- "LACP 聚合" vs "static 802.3ad LAG" —— 部分等同（无 LACPDU 协议交互但功能等价）
- "第一端口" vs "OVS bond port / Linux teaming master interface / DPDK rte_eth_bond 抽象" —— 字面等同
- "精确流表" vs "OVS megaflow exact match / tc-flower with fixed prefix / DPU eSwitch 5-tuple offload rule" —— 字面等同
- "同时下发到所有 N 网卡" vs "broadcast install / NIC bond mirror offload" —— 字面等同

### 明显不在保护范围内的相邻方案（反向证据形态）

1. **单网卡 HW offload**（N=1）—— 不命中 F1（专利限定 N ≥ 2 独立网卡）
2. **Active/standby 主备 bond**（仅一张卡 active）—— 不命中 F2（LACP 是 active-active）+ 不命中 F5（只卸载到 active 卡）
3. **物理交换机 LAG + 单台服务器 1 NIC** —— 不命中 F1
4. **Linux kernel bonding 纯软件 bond，无 HW offload** —— 不命中 F5（精确流表未卸载到硬件）
5. **VM/容器应用层多路径**（非 vSwitch 转发层）—— 不命中 F1
6. **单 NIC 多 PF VF-LAG**（N=1 卡 + 卡内 PF bond）—— 与 F1 拓扑外推禁令直接冲突；该 vendor 文档实际描述的拓扑 ≠ 专利 N ≥ 2 拓扑（按 Step 4 R-2 拓扑变体细分规则在 Step 5 拆开处置）
