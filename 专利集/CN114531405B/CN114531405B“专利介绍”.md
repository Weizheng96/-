## 一句话概括

为解决"虚拟交换机做硬件流表卸载时，单网卡（NIC）故障即整机不可用"的单点故障问题，本专利在一台物理机内将 **N（N ≥ 2）张 NIC 的 N 个 LACP 逻辑端口再做一次聚合**，对外形成单一"第一端口"，并在生成精确流表（exact-match flow）时**同时下发同一份流表到所有 N 张 NIC**——任意 NIC 故障可即时切到其它 NIC，且数据流不走环路；配套提供 K 队列零拷贝、流表删除/更新一致性同步、NIC 异常感知与恢复重聚合等机制。

## 解决的技术问题

- 现有"硬件卸载交换"技术（vSwitch 把 fast-path 流表卸载到智能网卡 / DPU）只能在**单张 NIC 内**做端口冗余——一旦该 NIC 整卡故障，整台物理机的 fast-path 转发即不可用，构成单点故障（SPoF）。
- 在多 NIC 拓扑下，简单地把多张 NIC 都接入虚拟交换机会产生 **VM ↔ vSwitch ↔ pSwitch 之间的数据环路与 MAC 学习抖动**，因为 vSwitch 看不到对端聚合关系。
- 需要一种让 N 张 NIC 对内对外**同时**对外形成单一逻辑端口、对内**同时**承载相同卸载流表的机制，以同时实现：(a) 跨 NIC 的故障切换；(b) 卸载流表跨 NIC 一致性；(c) 数据流零拷贝路径不被改变。

## 核心技术方案

1. **两层端口聚合**：第一层用标准 LACP（IEEE 802.3ad / 802.1AX）把每张 NIC 的多个物理口聚合为该 NIC 的"逻辑端口"；第二层在 vSwitch 里把 N 张 NIC 的 N 个逻辑端口再做一次"端口标识 → 目标端口标识"的**映射聚合**为对外的"第一端口"。
2. **首包未命中触发跨 NIC 同步卸载**：任一目标 NIC 收到报文且本地无对应卸载流表时，向 vSwitch 上报；vSwitch 根据报文 + VF 标识生成精确流表，并通过"第一端口"将该精确流表**同时下发到全部 N 张 NIC**。
3. **N 个卸载流表集合一致性**：vSwitch 维护一个精确流表集合；N 张 NIC 各自的卸载流表集合中"满足卸载条件的精确流表"与之一一对应，**N 份卸载流表内容完全相同**（多 NIC 等价镜像）。
4. **第二端口（VF 聚合端口）+ 零拷贝**：vSwitch 内将 N 张 NIC 对应的 N 个 VF（虚拟功能）聚合为"第二端口"用于接收 VM 报文；VM 虚拟网卡配置为 K 个队列（K 为 N 的整数倍），每个 VF 负责 K/N 个队列；数据流通过 VM 内存与 NIC 内存之间的 K-VF 映射做**零拷贝**传输。
5. **流表生命周期同步**：第一 NIC 删除某条卸载流表 → 向其余 NIC 广播删除指令；触发 TCP 状态变化、慢路径规则变更、连接老化、VM 端口删除等"预设条件"时下发更新指令保持一致性。
6. **NIC 故障感知与恢复**：检测到第二 NIC 状态异常 → 解除其逻辑端口与其他逻辑端口的聚合 + 向其下发"标记指令"将本地卸载流表标记为无效；状态恢复正常 → 重新聚合 + 下发精确流表 + 重建一一对应关系。

## 关键权利要求解读

### 权 1（核心独立权 — 方法 / vSwitch 视角）

逐字段拆解：
- **应用场景限定**：vSwitch 连接 **M ≥ 2 个 VM** 与 **N ≥ 2 张 NIC**——单 NIC 拓扑明确不在保护范围。
- **第一层聚合（前提）**："每个 NIC 对应的逻辑端口是该 NIC 的物理端口基于 **LACP** 聚合形成的逻辑端口"——明确点出 LACP（IEEE 802.3ad / 802.1AX），而非 active-backup / balance-rr / balance-xor 等 non-LACP 模式。
- **第二层聚合（核心新点）**："建立 N 个 NIC 逻辑端口的 N 个端口标识 与 **一**目标端口标识之间的**映射关系**，以将 N 个逻辑端口聚合为**第一端口**"——对外表现为单端口。
- **触发条件 + 同步动作**：目标 NIC "查询到不存在与所接收报文对应的卸载流表"时，"通过第一端口将精确流表卸载至 N 个 NIC"——同步下发到全部 N 张。

### 权 11（核心独立权 — 方法 / 物理机视角）

与权 1 同样的核心机制（F1-F4），但视角在物理机层：明确报文可以是"物理交换机发送的"或"目标虚拟机发送的"，覆盖**入向（北→南）+ 出向（南→北）**两个方向。

### 权 23-36 简述

权 23 = 权 1 装置形式；权 33 = 整机集成；权 34/35/36 分别覆盖通用机器、芯片、存储介质——形式权，本质都是 F1-F4 在不同载体上的体现。

### 从属权扩展（权 2-10、12-22、24-32 中的关键 alt 路径）

- 权 4 / 16 / 26：**卸载条件**——连接稳定 OR 流速率阈值（择一即可）
- 权 5 / 17 / 27：**N 份卸载流表内容相同**（一致性硬约束）
- 权 6 / 18 / 28：删除指令同步（"删除一处 → 通知其余 N-1 张"）
- 权 7 / 19 / 29：更新指令同步（4 类预设条件中**任一种**触发）
- 权 8 / 20 / 30：故障感知（解聚合 + 标记无效）
- 权 9 / 21 / 31：故障恢复（重聚合 + 重新卸载 + 重建对应关系）
- 权 10 / 22 / 32：**K 队列零拷贝**（K 为 N 的整数倍；VM 内存 ↔ NIC 内存直接 DMA）
- 权 15：VM 虚拟网卡两种工作模式——**软直通**（报文经过 vSwitch）与**硬直通**（VM 直接通过"第三端口"写入 VF）

### 每条 F# 的 satisfaction paths（强制 — 防漏读从属权 alt）

**F1（多 VM × 多 NIC 拓扑）**
- Path 1.1（独立权字面）：M ≥ 2 个 VM + N ≥ 2 张 NIC，全部接入同一 vSwitch / 同一物理机
- Path 1.2（等同 — 容器替代）：把 VM 解读为隔离的工作负载实例（VM、Pod、轻量容器、Firecracker microVM、Kata 安全容器），M ≥ 2 即可——主张容器形态在功能上等价于 VM
- Path 1.3（等同 — Bare-metal 多租户）：虽无 VM，但有"裸金属多租户"且每个租户共享同一物理机的 N 张 NIC，租户实例对 vSwitch 透明——以"租户实例"作为 VM 等价物

**F2（每张 NIC 内 LACP 聚合）**
- Path 2.1（独立权字面）：每张 NIC 的多端口基于 LACP（mode=4 / 802.3ad）聚合，形成单一逻辑接口
- Path 2.2（从属权 alt — 单口 NIC）：当每张 NIC 仅有 1 个物理端口时，"逻辑端口 = 物理端口"——无需聚合也满足"逻辑端口"概念（"基于 LACP 聚合形成"被弱化为退化情况；本路径在产品中常见）
- Path 2.3（等同）：等价的厂商专有聚合协议（思科 PAgP、华为 LAG / Eth-Trunk 静态模式 + LACP 模式）；只要协议达成"多物理端口对外一逻辑端口 + 链路状态协商"即可——但要注意，**纯 active-backup / balance-rr 等 non-LACP 模式不属于等同**（LACP 有协商帧，static round-robin 没有）

**F3（跨 NIC 第二层映射聚合为"第一端口"）**
- Path 3.1（独立权字面）：vSwitch 内显式建立"N 个端口标识 → 1 个目标端口标识"的映射表
- Path 3.2（等同 — bond-of-bonds）：在 Linux 内核 bonding / OVS bond / FRR / DPDK link-bonding 之上**再做一次 bonding**（"嵌套 bond"）；只要外层把 N 个 LACP bond 视作单一上行口
- Path 3.3（等同 — multi-uplink team）：teamd / NetworkManager team 在多 NIC 之上提供"team-of-teams"模式
- Path 3.4（等同 — VF-LAG / multi-host LAG）：在 SmartNIC / DPU 上由硬件团队驱动把多张 NIC 的 VF/PF 合并为单一上行（NVIDIA VF-LAG、Intel LAG、AMD/Pensando 多 PF 聚合等通用术语）；若硬件实现等价于"映射 N → 1"亦构成等同

**F4（跨 NIC 同步流表卸载）**
- Path 4.1（独立权字面）：vSwitch 在生成精确流表时，通过"第一端口"将该流表**同时**下发到 N 张 NIC 全部（N 份完全相同）
- Path 4.2（等同 — 多次单播）：vSwitch 串行/并行向 N 张 NIC 各下发一次该流表（无显式"第一端口"统一接口，但下发结果等价：N 份相同流表存在于 N 张 NIC）
- Path 4.3（从属权 alt — 一致性是 first-class）：从属权 5/17/27 明确"N 个卸载流表集合中与同一精确流表对应的 N 个卸载流表相同"——即只要事实上达到"N 份相同"即满足，无论使用什么内部 channel
- Path 4.4（等同 — 控制面集中下发）：通过集中式控制器（如 SDN 控制器、OVN 北向数据库、自研 control-plane agent）一致地把流表 push 到所有 NIC fast-path——只要"同一精确流表 ↔ N 份相同卸载流表"的对应关系存在

### 等同特征讨论

- **LACP 与厂商专有 LAG**：本质都是 IEEE 802.1AX 链路聚合的实现，应可主张等同；但 active-backup / round-robin 等非协商型 bond 不构成等同。
- **bond-of-bonds vs 显式映射表**：只要外层把 N 个内层 bond 当作单一可路由 uplink 使用，应构成等同（Path 3.2/3.3）。
- **K 队列零拷贝**（F5）：可被实现为 vDPA、virtio-pci passthrough + multi-queue、SR-IOV multi-VF queue split——只要"VM K 队列 → N 个 VF 各负责 K/N 队列 + 内存直接拷贝"成立即等同。

### 明显不在保护范围内的相邻方案

- **单网卡多端口 LACP**（N = 1，多物理端口）：明确不满足 F1（N ≥ 2）。
- **active-backup 跨 NIC**（多张 NIC，但同时只 1 张 active）：不满足 F4（不存在"同时下发到 N 张"的同步卸载，且 backup NIC 的卸载流表也不在线）。
- **纯软件 vSwitch 多 NIC bonding 但不做硬件流表卸载**（无 fast-path offload 到 NIC）：不满足 F4（不存在"卸载流表"概念）；如 OVS-DPDK 纯用户态 fast-path 但未下沉到 NIC TC/eBPF/HW offload 即不命中。
- **每张 NIC 独立维护各自的卸载流表（无跨 NIC 一致性约束）**：不满足 F4（"N 份相同"是字面要求）。
- **传统 NPB / TAP 网络分流多 NIC 模式**（无 vSwitch、无 VM、无流表卸载）：不满足 F1 与 F4。
