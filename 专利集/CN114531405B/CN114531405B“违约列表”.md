## 0. 必要技术特征 + 主体映射速查

**F1-F6（共享于全部 7 条独立权）**：
| 编号 | 特征 |
| --- | --- |
| F1 | 主体是**虚拟交换机**（vSwitch） |
| F2 | M ≥ 2 VM + **N ≥ 2 张物理 NIC** |
| F3 | **每张 NIC 内部物理端口基于 LACP 聚合**为该 NIC 的逻辑端口 |
| F4 | vSwitch 把 N 张 NIC 的 N 个逻辑端口标识 → **同一目标端口标识**（第一端口） |
| F5 | 慢路径触发后，**同一精确流表同时下发到 N 张 NIC**（非主备、非分流） |
| F6 | NIC 侧具有硬件 offload 流表 |

## 3. 方法学小结：为什么 0 个"确认侵权"

### 关键事实
1. **本专利保护的是实现细节型权利要求**。F3-F5 三项核心要件（NIC 内部 LACP + vSwitch 二级映射到同一目标端口 + 同一精确流表同时下发到全部 N 张 NIC）属于**白盒实现细节**，几乎不会出现在产品宣传 / 官方文档 / 媒体报道中。
2. **公开渠道天花板**：本 skill 当前工具集（WebSearch / WebFetch / pdf 抽取）已穷尽：
   - SIGCOMM 2024 一手论文（Triton 14 页 pdfplumber 全文）
   - 122 页《2025 AI 网络白皮书》、65 页《中国移动云智算白皮书》全文扫描
   - 阿里云 / 百度云 / 火山引擎 / 腾讯云 / 中国移动 / 中国电信 / 联通官方技术文章
   - SNAI 2024 案例报道
   - GitHub `asterfusion/Helium_DPU` 开源仓库
   - 即便如此，**没有任何一份公开材料同时披露 F1-F6 全要件**。
3. **直接排除的反向证据明确**：
   - VMware：dual-DPU 配置文档明文 active/standby
   - 中国电信：白皮书明文 bond mode 1
   - NVIDIA：VF-LAG 单 NIC 限定文档明文
   - Helium：GitHub 全仓库可审，无相关代码
   - AWS：Nitro 不走 OvS / 不走 LACP

### 升级到"确认侵权"的真实路径
- **专利墙反向比对**（IncoPat / 智慧芽 / Patentscope MCP，需接入专业库——Google Patents WebSearch 索引深度有限）
- **产品取证**：购买含目标 DPU 的服务器实例，模拟单卡故障抓取流表行为
- **法律文件**：招股书、年报、监管披露中的技术披露章节
- **学术追踪**：SIGCOMM 2025-2026、APNet、NSDI 中"多 NIC 流表同步"主题
- **客户招标技术规格书**：电信 / 金融 / 政务 NFV 招标书中的 bond mode 与多卡冗余具体要求

- 本表所有判定基于公开 WebSearch / WebFetch 在 2026-04-27 拿到的内容，**不构成对任何组织的侵权指控**。
- "已排除"仅指公开资料能拿到反向证据；不代表该组织在所有产品形态、所有客户部署中都不可能落入保护范围。
