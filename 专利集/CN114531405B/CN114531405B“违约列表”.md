## 0. 必要技术特征清单（用于全局对比）

| 编号 | 特征 |
| --- | --- |
| F1 | 主体是**虚拟交换机**（vSwitch） |
| F2 | M ≥ 2 VM + **N ≥ 2 张物理 NIC** |
| F3 | **每张 NIC 内部物理端口基于 LACP 聚合**为该 NIC 的逻辑端口 |
| F4 | vSwitch 把 N 张 NIC 的 N 个逻辑端口标识 → **同一目标端口标识**（第一端口） |
| F5 | 慢路径触发后，**同一精确流表同时下发到 N 张 NIC**（非主备、非分流，是同表多卡） |
| F6 | NIC 侧具有硬件 offload 流表（卸载流表 / 快路径表） |

**专利明确排除的相邻方案**（来自专利介绍"保护范围边界"小节）：
- 单网卡（N=1）—— 不命中 F2
- **多网卡 active/standby 主备**（只主卡持流表）—— 不命中 F5
- 纯软件 vSwitch（无 NIC offload）—— 不命中 F6
- 物理交换机层 MLAG / VPC / Stack —— 不在 vSwitch-NIC 卸载场景
- 多 NIC 各自独立流表（无集合一致性）—— 不命中 F5 / 权 5

| 特征 | 字面命中 | 等同命中 | 证据 |

## 3. 方法学小结：为什么 0 个"确认侵权"

### 关键事实
1. **本专利保护的是实现细节型权利要求**。F3（NIC 内部 LACP 聚合）+ F4（vSwitch 二级映射到同一目标端口）+ F5（同一精确流表同时下发到全部 N 张 NIC）这三项核心要件，几乎不会出现在产品宣传 / 官方文档 / 媒体报道中——它们属于**白盒实现细节**。
2. **公开渠道天花板**：本 skill 当前工具集（WebSearch / WebFetch / pdf 抽取）能拿到的最深证据是：
   - SIGCOMM 2024 / NSDI / OSDI 一手论文（已用 pdf skill 完整抽取 Triton 论文 14 页全文）
   - 厂商技术博客（百度 / 火山引擎 / 阿里云开发者社区）
   - 行业白皮书（2025 AI 网络白皮书 122 页 / 中国移动云智算白皮书 65 页）
   - 行业奖项报道（SNAI 2024 灵耀获奖）
   - 即便如此，**没有任何一份公开材料同时披露 F1-F6 全要件**。
3. **直接排除的 9 家**已用反向证据明确（NVIDIA VF-LAG 单 NIC 限制 / VMware vSphere 8 U3 active/standby 配置 / 中国电信 bond mode 1 / Helium 开源代码无相关描述 / AWS 不走 LACP / Cilium 不走 LACP / Broadcom-百度时间不合规 / 国产 DPU 创业公司公开资料停留在概念）。

### 升级到"确认侵权"的真实路径（公开 WebSearch 之外）
- **专利墙反向比对**：在 IncoPat / 智慧芽 / Patentscope 上以 F1-F6 关键词组合检索阿里 / 腾讯 / 字节 / 百度 / 中国移动等申请人名下的同向专利。Google Patents 是 SPA，常规 WebSearch 索引深度不够。
- **产品取证**：购买含目标 DPU 的服务器实例（移动云 / 阿里神龙 / 百度太行 / 火山引擎），实验室复现"双卡 + 模拟单卡故障 + 抓取流表行为"。
- **法律文件**：招股书（字节 IPO）、年报、监管披露中的技术披露章节。
- **学术追踪**：跟踪 SIGCOMM 2025 / 2026、NSDI、APNet 中是否出现"多 SmartNIC 流表同步"主题论文。
- **客户案例**：电信运营商、金融、政务的招标技术规格书往往会涉及 bond mode 与多卡冗余的具体要求。

### 给本 skill 的方法学反馈
- 这一类**实现细节型专利**在当前工具集下，**结构上无法仅通过公开 WebSearch 锁定"确认侵权"**。
- 建议在 [SKILL.md](.claude/skills/patent-infringement-check/SKILL.md) 中加一条说明："对于 F3-F5 是黑盒实现细节的专利（H04L / 芯片 / 算法类），公开来源往往无法升级到'确认侵权'，需要专利墙反向比对 + 实验室取证才能升级；初筛 + 终筛只是把候选锁定到可投入取证资源的 1-3 家"。
- 已建议接入专业专利数据库 MCP（IncoPat / 智慧芽）以打通"专利墙反向比对"环节。
