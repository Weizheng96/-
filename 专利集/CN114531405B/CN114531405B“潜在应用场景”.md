# CN114531405B 潜在应用场景

> 本文件是 Step 3 的第二部分；前置 Step 3a 输出在 [`CN114531405B"领域适配".md`]。
> 本文件不重复领域分类 / R-* 标志 / 必跑清单——参见 Step 3a。

## 推演方法

从 F1-F5 出发——专利核心：vSwitch 内多 NIC LACP 两层聚合 + 精确流表多卡同步硬件 offload，主要为防单卡 HW offload 单点故障。今天 2026-05-11，结合 (a) 公有云宿主机 HA 双 NIC 部署已成熟；(b) DPU/SmartNIC 商业化进入 BlueField-3 / Pensando 第二代；(c) 电信运营商 5G UPF 集采明示 NFV 数据面 HA 要求；(d) VMware vSphere 8 DSE / Red Hat OSP × DPU 集成参考架构铺开；(e) 容器网络（Antrea / Kube-OVN）开始引入 DPU 卸载 — 等五条主轴推演场景。

## 场景列表

### 场景 1：DPU / SmartNIC 多卡 HA + VF-LAG 全卡 HW offload

- 场景描述：宿主机配 ≥ 2 张 DPU/SmartNIC 卡（每卡 dual-port）；卡内 LACP（VF-LAG）+ 跨卡 OVS bond port 聚合；精确流表通过 vSwitch 控制面同时下发到所有 N 张卡的 e-switch
- 使用形式：DPU 整机 + vendor SDK（DOCA / DSC SDK / IPDK / OCTEON SDK）+ OVS-DOCA 控制面
- **具名产品列表**：
  - **NVIDIA**：BlueField-2 / BlueField-3 + DOCA SDK + OVS-DOCA + ASAP² Direct + VF-LAG（核心候选 — 卡内 VF-LAG 是 NVIDIA 公开文档明示能力）
  - **AMD Pensando**：DSC-25 / DSC-200 + Penguin OS + Pensando SDK
  - **Intel**：IPU E2000 (Mount Evans) + IPDK + OPI Project
  - **Marvell**：OCTEON 10 DPU + Smart Networking Stack
  - **中科驭数**：K2 KPU + HADOS SDK
  - **云豹智能**：Yunsilicon Metasilicon DPU
  - **星云智联**：Nebulon NebulOS
  - **大禹智芯**：Paratus DPU + DAYU 平台
  - **Asterfusion 星融元**：Helium EC2000 / ET2500 + 软件栈
  - **中国电信紫金 DPU**：天翼云 4.0 集成方案
- 与权利要求对应：F1（N ≥ 2 DPU 卡 + M VM）+ F2（卡内 LACP / VF-LAG）+ F3（跨卡 OVS bond port → 第一端口）+ F4（OVS miss-to-upcall）+ F5（精确流表同步下发到所有卡 e-switch）—— 但**关键拓扑歧义**：若 N=1 单卡 dual-PF（VF-LAG 默认部署），F1 不命中
- 商业价值 / 紧迫性：**高** — DPU 商业化 + 5G UPF / 公有云 HA 强需求
- 反例：单卡部署（N=1）；卡内 VF-LAG 但跨卡无 OVS bond → F3 / F5 不满足

### 场景 2：公有云宿主机 LACP 双 NIC + OVS HW offload

- 场景描述：公有云宿主机配 2 张独立 NIC（含或不含 DPU），通过 Linux bond / OVS bond port 聚合；HW offload 委托给底层 NIC driver；精确流表通过 vSwitch 多卡下发
- **具名产品列表**：
  - **AWS**：Nitro 系统 + ENA driver + 双 NIC HA bond（具体实现闭源）
  - **Microsoft Azure**：Hyper-V Virtual Switch + AccelNet + 双 NIC + Catapult FPGA / Azure Boost SmartNIC
  - **Google Cloud**：Andromeda + Snap + 多卡 LAG（自研 SDN）
  - **阿里云**：神龙（X-Dragon）+ CIPU DPU + 多卡 bond + AVS（Apsara vSwitch）
  - **腾讯云**：TGW 网关 + 银杉 DPU + bond
  - **中国电信天翼云**：天翼云 4.0 + 紫金 DPU + LACP bond
  - **中国移动云**：移动云 5G UPF NFV 平台
  - **中国联通沃云**：联通 NFV 平台
- 命中独立权：权 1 / 11（方法）
- 商业价值：**高**
- 反例：Active/standby 主备 bond（仅单卡 active）不命中 F2 / F5

### 场景 3：电信运营商 NFV / vEPC / vBNG / 5G UPF + LAG

- 场景描述：5G UPF / vEPC 用户面网元跑在 OpenStack + OVS-DPDK / VPP 的 NFV 平台；运营商集采明确要求多 NIC HA + HW offload
- **具名产品列表**：
  - **中国电信**：天翼云 NFV + 紫金 DPU；5G UPF
  - **中国移动**：BC-MANO + 移动云 5G UPF
  - **中国联通**：沃云 NFV
  - **AT&T**：Domain 2.0 / Network Cloud + ONAP + Open vSwitch
  - **Verizon**：Verizon Cloud Platform + ONAP
  - **Vodafone**：Pan-EU NFV + Red Hat OpenStack + OVS
  - **Deutsche Telekom**：DT TeraStream + OPNFV + OVS-DPDK
  - **NTT DoCoMo**：5G NFV
- 命中独立权：F1-F5（权 1 / 11）
- 商业价值：**高** — 5G 商用 + 集采有公开技术应答

### 场景 4：商业虚拟化平台 LAG + DPU offload（VMware NSX-T + vSphere 8 DSE）

- 场景描述：商业 hypervisor + 集成 DPU 整机方案
- **具名产品列表**：
  - **VMware NSX-T + vSphere 8 with DSE（Distributed Services Engine）** + BlueField/Pensando 双卡集成
  - **VMware Cloud Foundation**（VCF）+ DPU
  - **Red Hat OpenStack Platform 17 + OpenShift 4.x + BlueField DPU**（OVN-Kubernetes + bond + DPU offload）
  - **Microsoft Azure Stack HCI + Azure Stack Edge**
- 命中独立权：F1-F5（权 1 / 11 / 23 / 33 / 34 / 36）
- 商业价值：**中-高**

### 场景 5：开源 vSwitch（OVS）+ Linux 内核 bond + tc-flower offload（上游本身）

- 场景描述：OVS 主线代码本身 ship bond port 抽象 + tc-flower offload 接口；多 NIC bond + HW offload 是上游能力
- **具名产品列表**：
  - **Open vSwitch 主仓库**（github.com/openvswitch/ovs）
  - **Linux kernel mlx5_core driver**（Mellanox/NVIDIA 贡献的多卡 bond + LAG offload 实现）
  - **DPDK rte_eth_bond PMD**（用户态 LAG 框架）
  - **OVN（Open Virtual Network）**
- 命中独立权：F1-F5（权 1 / 23 / 36）
- 商业价值：**中**（影响所有下游使用方）

### 场景 6（次要）：容器网络 DPU 卸载

- 场景描述：K8s 容器网络（CNI）+ DPU 卸载方案；K8s 节点配多 DPU 卡 + Antrea / Kube-OVN + bond
- **具名产品列表**：VMware Antrea + DPU；灵雀云 Alauda Kube-OVN；NVIDIA DOCA OVS Offload + Kubernetes
- 命中：F1-F5（部分命中——K8s 节点 ≥ 2 DPU 卡是 niche 部署）
- 商业价值：**中-低**（K8s + DPU 集成尚未规模化）

---

排序：场景 1（DPU 多卡 HA）= 场景 2（公有云 HA bond）= 场景 5（OVS 上游本身）> 场景 3（运营商 NFV）= 场景 4（商业 NSX-T / OSP）> 场景 6（容器）

**注意整数限定 N ≥ 2 的拓扑歧义**：场景 1 中 NVIDIA VF-LAG 默认描述是**单卡 dual-PF**（N=1，不满足 F1）。Step 4 应用 R-2 拓扑变体细分规则——把单卡部署 / 跨卡部署作为不同候选条目分别 verdict。
