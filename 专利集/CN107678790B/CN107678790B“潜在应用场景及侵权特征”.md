## 一、潜在应用场景（严格来自背景技术 — 不外扩）

> 背景技术只把"金融服务、传感监测、网络监控"作为**数据流产生的应用领域**举例，并指出现有流计算系统的核心痛点：用户须在 IDE 里用**拖拽基本算子**的方式手工构建流图（DAG），基本算子粒度太细 → 流图复杂、臃肿、性能差。本专利的"典型使用形式"是**同一个流计算引擎 / 平台**：用户改用 **SQL** 定义流处理逻辑，由管理节点把 SQL 动态编译成流图（再优化成算子 DAG）下发执行。下列场景共用这一典型形式，差异只在数据来源领域。

### 场景 1：金融服务实时流计算
- **背景技术 verbatim 依据**：> "在诸如金融服务、传感监测和网络监控之类的应用领域中，数据流具有实时性、易失性、突发性、无序性和无限性等特征。流计算(Stream computing)系统作为一种能对实时的数据流进行计算处理的系统，得到了越来越广泛的应用。"
- **场景说明**：金融机构 / 金融科技公司对交易流、行情流做实时风控、反欺诈、实时对账、实时指标，数据流实时且突发。
- **使用本专利的典型形式**：部署一套流计算引擎 / 平台，用户用 **SQL（流式 SQL）** 定义流作业，引擎把 SQL 编译成逻辑流图，再优化成算子 DAG 下发到工作节点执行（背景技术明示的"用 SQL 替代拖拽基本算子构建流图"形式）。

### 场景 2：传感监测 / IoT 实时流处理
- **背景技术 verbatim 依据**：> "在诸如金融服务、传感监测和网络监控之类的应用领域中，数据流具有实时性、易失性、突发性、无序性和无限性等特征。"
- **场景说明**：工业 / 物联网传感器持续上报数据流，需实时聚合、告警、异常检测。
- **使用本专利的典型形式**：同上——流计算引擎 / 平台以 SQL 定义对传感数据流的计算逻辑，编译为算子 DAG 执行。

### 场景 3：网络监控（流量 / 日志实时分析）
- **背景技术 verbatim 依据**：> "在诸如金融服务、传感监测和网络监控之类的应用领域中……流计算(Stream computing)系统作为一种能对实时的数据流进行计算处理的系统，得到了越来越广泛的应用。"
- **场景说明**：电信 / 互联网 / 安全厂商对网络流量、日志做实时监控、统计、入侵 / 异常检测。
- **使用本专利的典型形式**：同上——以 SQL 定义流式监控逻辑，引擎编译成算子 DAG 实时执行。

> 三个场景的"潜在侵权产品"在抽象层上是**同一类东西**：**支持以 SQL 定义流作业、并将其编译成算子 DAG（含逻辑图→物理算子图的优化 / 算子合并）下发执行的流计算引擎 / 平台**。Step 3/4 的真正排查对象是这类引擎 / 平台的提供方，而非单纯的下游应用业务方。

## 二、独立权利要求 + 侵权特征 F1-Fk

本专利有 3 条独立权利要求：**权 1（方法）、权 8（装置）、权 15（系统）**——三者是同一技术方案的方法 / 装置 / 系统三种形式，F1-F5 同构。以权 1 为基准拆 F#。

### 权 1 verbatim
> "1.一种流计算方法，其特征在于，应用于包括管理节点和工作节点的流计算系统中，所述方法包括：
> 所述管理节点从客户端获取输入通道描述信息、结构化查询语言SQL语句和输出通道描述信息，所述SQL语句包括若干条SQL规则，每条SQL规则包括至少一条SQL子语句；
> 所述管理节点根据所述输入通道描述信息、所述若干条SQL规则和所述输出通道描述信息生成第一流图，所述第一流图包括若干个逻辑节点；
> 所述管理节点将所述第一流图中的各个逻辑节点进行划分，以得到若干个逻辑节点组；在预设算子库中选择每个所述逻辑节点组对应的公共算子，并根据选择出的所述公共算子生成第二流图；所述第二流图中的每个算子用于实现所述算子对应的逻辑节点组中的一个或多个逻辑节点的功能；所述管理节点根据所述第二流图控制所述工作节点执行所述流计算任务；
> 其中，所述输入通道描述信息用于定义输入通道，所述输入通道是用于将来自数据生产系统的数据流输入流图的逻辑通道；所述输出通道描述信息用于定义输出通道，所述输出通道是用于将流图的输出数据流输出至数据消费系统的逻辑通道。"

### F1-F5 清单

- **F1（从客户端获取 SQL + 输入/输出通道描述信息）｜[AND]**：> "所述管理节点从客户端获取输入通道描述信息、结构化查询语言SQL语句和输出通道描述信息，所述SQL语句包括若干条SQL规则，每条SQL规则包括至少一条SQL子语句"
  - 可能描述方式：① 用户提交一段流式 SQL / Streaming SQL / Stream SQL 作业；② 用 `CREATE TABLE … WITH (connector=…)` 之类 DDL 定义 source / sink（"输入/输出通道描述信息" ≈ source/sink connector 配置）+ 一段查询 SQL；③ 通过 SQL CLI / SQL gateway / REST 提交 SQL 作业；④ "submit a SQL query that reads from a source and writes to a sink"。
- **F2（由 SQL + 通道信息生成"第一流图"=逻辑节点图）｜[AND]**：> "所述管理节点根据所述输入通道描述信息、所述若干条SQL规则和所述输出通道描述信息生成第一流图，所述第一流图包括若干个逻辑节点"
  - 可能描述方式：① SQL 解析器把 SQL 转成逻辑执行计划 / logical plan / logical graph / AST / RelNode 树；② "parse SQL → logical plan / logical DAG"；③ Calcite 等优化器生成逻辑算子树；④ 称作 StreamGraph / 逻辑流图 / 逻辑管线。
- **F3（逻辑节点分组 + 从预设算子库选公共算子 → "第二流图"；一个算子实现一组逻辑节点的功能 = 算子合并）｜[AND]**：> "所述管理节点将所述第一流图中的各个逻辑节点进行划分，以得到若干个逻辑节点组；在预设算子库中选择每个所述逻辑节点组对应的公共算子，并根据选择出的所述公共算子生成第二流图；所述第二流图中的每个算子用于实现所述算子对应的逻辑节点组中的一个或多个逻辑节点的功能"
  - 可能描述方式：① 逻辑计划 → 物理计划 / physical plan / execution graph / JobGraph；② **算子链 / operator chaining / operator fusion / task chaining**——把多个逻辑算子合并进一个物理算子 / task（正对应"一个算子实现多个逻辑节点功能"）；③ 从算子库 / 函数库 / 物理算子集合中选出实现该逻辑节点组的物理算子；④ "fuse / chain multiple logical operators into one physical operator/task"。
- **F4（按"第二流图"控制工作节点执行流计算任务）｜[AND]**：> "所述管理节点根据所述第二流图控制所述工作节点执行所述流计算任务"
  - 可能描述方式：① JobManager / Master / Coordinator 把算子（task）调度到 TaskManager / Worker / Executor 执行；② "scheduler dispatches the execution graph's operators to worker nodes"；③ 主从（master-worker）调度执行。
- **F5（输入/输出通道定义：数据生产系统→流图→数据消费系统）｜[AND]**：> "所述输入通道描述信息用于定义输入通道，所述输入通道是用于将来自数据生产系统的数据流输入流图的逻辑通道；所述输出通道描述信息用于定义输出通道，所述输出通道是用于将流图的输出数据流输出至数据消费系统的逻辑通道"
  - 可能描述方式：① source connector 从上游数据生产系统（Kafka / 消息队列 / CDC 等）读取数据流，sink connector 把结果写到下游数据消费系统（DB / 消息队列 / 存储）；② "input/source stream from a producer, output/sink stream to a consumer"；③ 用逻辑表 / 逻辑流抽象屏蔽具体 IO。

**结构标签说明**：权 1 的所有步骤是方法步骤，全部 **[AND]**——侵权须落入 F1–F5 全部。F3 内部含"逻辑节点分组 + 算子库选算子 + 算子合并"三个子动作（全须满足）。F5 含"输入通道 AND 输出通道"两半。注意权 1 字面用"和/或"出现在从属权（权 6 修改信息），独权 1 主干无 [OR] 分支。

### 其他独立权（同构）
- **权 8（装置）**：把权 1 的方法拆成"获取单元 / 生成单元 / 执行单元"三个功能单元——F1↔获取单元，F2+F3↔生成单元，F4↔执行单元，F5 限定同。判定一个**产品/装置**是否落权 8，与判方法权 1 的 F# 等价。
- **权 15（系统）**：流计算系统 = 管理节点 + 至少一个工作节点，管理节点含权 8 的装置。判定一个**部署的系统**落权 15，仍归结到 F1–F5。

### 关键限定词与隐含约束
- **两级流图是核心区分限定**：必须存在"第一流图（逻辑节点构成）"和"第二流图（算子构成）"两层，且第二流图由对第一流图逻辑节点**分组 + 选算子**得到。只有"一张图"而无逻辑图→物理图两级映射的实现，F2/F3 可能落空——需 Step 6 核实候选是否真有 logical plan→physical/execution plan 两级。
- **算子合并 / 一个算子承载多个逻辑节点（F3 末句）是另一关键区分限定**：对应业界 operator chaining / operator fusion。候选若有 chaining/fusion 即强命中 F3 末句。
- **必须由 SQL 驱动**：F1/F2 要求"结构化查询语言 SQL 语句 / 若干 SQL 规则"。纯用 DSL / API / 拖拽 GUI 构图、不支持 SQL 定义流逻辑的引擎，F1/F2 不命中。
- **管理节点 / 工作节点分布式架构**：master-worker（JobManager/TaskManager、Driver/Executor、Coordinator/Worker 之类）。
- **预设算子库（preset operator library）**：物理算子 / 内置函数 / built-in operators 库——F3 要求从中"选择"算子。
