## 说明
- **初判 F# 偏高的原因**：01–06、08、10–14 都是基于 Apache Flink 或 Spark 的引擎（或同构架构），其 SQL→逻辑计划→物理执行图（含 operator chaining / whole-stage codegen 算子融合）+ master-worker 调度是公开的技术常识，故初判 F1–F5 全覆盖；最终仍以 Step 6 逐条 verbatim 取证为准（尤其 F3 的"逻辑节点分组+算子库选算子+算子合并"两级映射，需核实候选是否真有 logical→physical 两级）。
- **07 Azure Stream Analytics / 09 Materialize / 15 Snowflake** 初判未覆盖 F3：Azure ASA 的内部物理计划/算子融合是否成立公开度低；Materialize 走 differential dataflow，"逻辑节点分组→预设算子库选公共算子→第二流图"的两级结构是否对应需核实；Snowflake Dynamic Tables 是声明式增量、是否落"逐 SQL 规则生成逻辑节点 + 算子 DAG 调度"待证——三者初判保守，留待 Step 6。
- 多数为 Flink 衍生托管服务，底层共用 Apache Flink 这一**技术本体**——Step 5 会触发"产品 vs 技术二元归并"，可能把 Apache Flink 提为一个"技术"条目、各云托管服务作为其应用案例。Step 5 处理。
