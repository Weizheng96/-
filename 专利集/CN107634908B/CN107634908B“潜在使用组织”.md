## C1. WebRTC 上游 (Google) — 核心技术候选 ★★★

- **类型**：开源协议参考实现（R-OPENSOURCE 双层 P0 上游侧）
- **主体类型**：T4
- **关联的独立权**：权 1（方法）+ 权 8（设备——浏览器 / 终端 OS 内嵌 WebRTC 库构成"含 WebRTC 的设备"，OEM/OS 厂商可能命中设备权）
- **产品族二级条目**：
  - **C1-a**：`04-webrtc-upstream-google` — WebRTC 主分支 (libwebrtc, M91-M122 release lines) 中的 **RTP-FEC (RFC 5109)、ULPFEC、FlexFEC-03、NACK + GCC (Google Congestion Control)、Pacer (PacedSender)、TWCC (Transport-Wide Congestion Control)、REMB**——这些组件直接对应 F1（按 audio/video stream 业务类型差异化处理）+ F2（按 cwnd / RTT / loss 计算 FEC 速率）+ F3（按 SDP `priority` / 实时 deadline 约束）+ F4（PacedSender 决定 burst 节奏即"调度方法"）+ F5（实际经 UDP socket 发送）
  - **C1-b**：`04b-webrtc-forks-libwebrtc` — 各商业 fork 派生维护 (LiveKit、CloudWebRTC、Cisco WebRTC、Mozilla WebRTC、Apple WebKit-WebRTC、libwebrtc-rust)——可视为上游证据的延伸
- **命中场景**：场景 1（RTC SDK）— 几乎所有商业 RTC SDK 都基于 WebRTC fork
- **使用该专利的合理性**：WebRTC 主分支自 2014 年起在 `modules/rtp_rtcp/` / `modules/pacing/` / `modules/congestion_controller/` 中持续维护 audio/video stream 差异化处理 + NACK + ULPFEC + GCC 完整链路；自适应 FEC 速率（基于 RTT/loss）由 `FecController` / `ProtectionBitrateCalculator` 实现；最近 5 年新增了 FlexFEC-03 / RFC 8627 支持。**这是本专利保护范围最直接的字面实施**
- **可能性等级**：**极高（P0）**
- **检索关键词建议**：`webrtc FecController site:webrtc.googlesource.com`、`webrtc FlexFEC implementation`、`webrtc ProtectionBitrateCalculator`、`webrtc GCC pacing redundancy`、`webrtc ULPFEC adaptive rate`、`google webrtc NACK FEC`、`libwebrtc retransmit redundancy`

## C2. SRT 协议 + libsrt (Haivision 主导) — 核心技术候选 ★★★

- **类型**：开源协议参考实现（R-OPENSOURCE 双层 P0 上游侧）+ Haivision 商业版本
- **主体类型**：T4
- **关联的独立权**：权 1 + 权 8（Haivision Makito X 编码器、SRT-enabled 推流硬件出厂固化 = 设备权）
- **产品族二级条目**：
  - **C2-a**：`22-srt-haivision-upstream` — `libsrt` 开源协议库 + SRT 协议规范（IETF draft + SRT Alliance 文档）。SRT 协议内置三种丢包恢复机制：**ARQ (NACK)、FEC (XOR / Reed-Solomon)、自适应调整**——参数 `fec=1`、`packetfilter=fec,cols:N,rows:M,layout:staircase` 直接对应权 2/9 的"调度方法枚举"。其 `latency` 参数即权 3/10 的"传输总时间 ≤ 时延要求"。`congestion=live` / `mss=N` 与 F4 联动。**这是非 RTC 赛道的最直接命中代表**
  - **C2-b**：`22b-haivision-makito-encoders` — Haivision Makito X1 / X4 / X4 Rugged 系列编码器，出厂时已固化 SRT-FEC 推流方法（权 8 设备权直接命中候选）
- **命中场景**：场景 3（直播推流）、场景 6（政企广电）
- **使用该专利的合理性**：SRT 协议规范文本明确写明业务类型 (live / file / ts) → 不同 retransmit/FEC 策略；libsrt 源码中 `fec.cpp`、`congctl.cpp`、`packet.cpp` 实现自适应 FEC 速率和 latency-bound 调度；广播业 / 直播业大量采用
- **可能性等级**：**极高（P0）**
- **检索关键词建议**：`libsrt fec adaptive`、`SRT protocol latency packetfilter`、`Haivision SRT FEC scheme`、`SRT Alliance reference architecture`、`SRT congestion live ARQ`、`srt-rfc draft`、`libsrt CongCtl LIVE`

- **类型**：协议规范族（多方贡献）
- **主体类型**：T4
- **关联的独立权**：权 1
- **产品族二级条目**：
  - **C4-a**：`05-quic-fec-ietf` — IETF QUIC WG 下的 FEC 实验性 draft（如 `quic-fec-cc`、`draft-michel-quic-fec`、PQUIC 框架 FEC 插件）。注：QUIC 核心 RFC 9000 / 9001 / 9002 不强制 FEC，只支持 loss recovery + packet pacing；FEC 是可选扩展，未广泛标准化
  - **C4-b**：`05b-quic-implementations-fec` — quiche (Google)、lsquic (LiteSpeed)、msquic (Microsoft)、ngtcp2 中的 FEC 实现

- **使用该专利的合理性**：QUIC 已被 Cloudflare、Google、Microsoft、Apple 广泛部署；FEC 仅作可选扩展，命中度比 WebRTC / SRT / RIST 低一档
- **可能性等级**：**中**（FEC 扩展尚未广泛标准化）
- **检索关键词建议**：`QUIC FEC draft michel`、`quiche google FEC`、`msquic loss recovery FEC`、`PQUIC FEC plugin`

## 架构层级不符预排除清单（R-7）

按 R-7 准则，下列候选 F1 概念上不构成本专利保护范围（主体抽象层不在同一层）：

| 候选名 | 名义所属领域 | 实际架构层级 | 与本专利 F1 的概念差距 | 排除依据 |
| --- | --- | --- | --- | --- |
| **Qualcomm Snapdragon X-series 5G modem 芯片**（X55/X65/X70/X75/X85） | 通信 — 移动通信 | L1 PHY + L2 MAC（含 3GPP NR LDPC/Polar FEC） | 标准 NR 物理层 FEC 由 3GPP 强制，并非"按业务类型/网络状态/时延要求自适应调度"——业务类型不在 modem 芯片可见层级 | 架构层级不符（modem 只看 RB 资源映射，看不到应用层业务类型） |
| **MediaTek Dimensity / Helio modem 芯片** | 通信 — 移动通信 | L1 PHY + L2 MAC | 同上 | 架构层级不符 |
| **紫光展锐 唐古拉 modem 芯片** | 通信 — 移动通信 | L1 PHY + L2 MAC | 同上 | 架构层级不符 |
| **Samsung Exynos modem / Apple-acquired Intel modem** | 通信 — 移动通信 | L1 PHY + L2 MAC | 同上 | 架构层级不符 |
| **Marvell / NVIDIA ConnectX 智能网卡 PHY FEC offload** | 通信 — 数据中心 | 以太网 PHY (BASE-R / RS-FEC) | 802.3 标准强制 FEC，与业务感知无关 | 架构层级不符 |
| **Cisco / Juniper / Arista 路由器 OS** | 通信 — 数据中心网络 | L3 路由 + MPLS | 中间设备非"发送端"——主体不对 | 主体不对（接收端 + 中转） |
| **接收端纯 NACK 重传 jitter buffer** | RTC | 接收端 | 主体不对 | 主体不对 |
| **CDN 边缘单纯 cache + 重传**（不带发送端业务感知 FEC） | 流媒体 CDN | 中间转发 | 不含 F1 / F2 / F3 / F4 完整链路 | 缺 F2-F4 |
