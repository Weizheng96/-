## 3. R-STANDARD 处置（partial / 不明）

按 Step 3c 子流程的判定：

| 标准候选 | 关系 | 处置 |
| --- | --- | --- |
| IETF RFC 5109 / 8627 / 6363 (RTP-FEC / FEC Framework) | 范围相邻、不完全重合（标准只规定 FEC payload 编码格式，本专利保护"业务类型 + 网络状态 + 时延要求三因子自适应调度逻辑"超出标准范围） | 候选清单不变；按常规侵权流程 |
| 3GPP TS 26.346 (MBMS FEC) / TS 38.214 (5G NR FEC) | 范围相邻、不完全重合 | 同上 |
| SMPTE ST 2022-1/-7 (Pro-MPEG FEC for broadcast) | 范围相邻、不完全重合 | 同上 |
| CCSA / 国内 YD/T 标准 | 工具能力下未充分检索 | 标"标准合规状态需法务确认"，按常规侵权流程处理 |

**结论**：未发现明确的 SEP 声明导致整批候选转走 FRAND 通道。所有候选按常规侵权流程处理。维权前建议法务通过 ETSI IPR database / 3GPP IPR declarations / CCSA 公开文件进一步核查华为是否在 SA4 / RAN1 中声明过本族系为 SEP。

### §4.2 现有技术 caveat

§A.9 检索范围内**未发现专利申请日（2016-07-19）前覆盖度 ≥ 60% 的现有技术**——本专利的"业务感知 + 自适应 FEC + 时延约束"三因子完整方案在 2016 年前未在公开顶会大规模出现。

注：RFC 5109 (2007) ULPFEC / SMPTE 2022-1 (2007) Pro-MPEG FEC / RFC 6363 (2011) FEC Framework 等只对应本专利 F4 / F5 部分；无 F1 业务类型识别 + F2 自适应数量 + F3 时延要求联动的完整方案。本专利的发明点在于"三因子自适应"的具体方法，而不在 FEC 编码格式本身。

建议法务通过 IP 律师 IncoPat / 智慧芽账号进一步核查 2015-2016 期间的中国 / 美国 / 欧洲专利申请——可能有少量同领域专利申请在专利申请日前 ≤ 24 个月内公开，需作完整覆盖度对比。

### §4.3 法律免责声明

本文件是基于 **2026-05-11** 检索时间窗的公开资料整理出的**侵权线索 + 证据链**，**不是法律结论**。每条 verdict 均仅表达技术特征落入保护范围的最强公开评估，最终是否构成侵权应由专利律师 + 法院基于完整证据（含 vendor 内部代码 / 技术规格 / 部署证据）独立判定。

维权前请：

1. **通过 IP 律师做反向专利墙 + 同族 PCT 检索** — 在候选 vendor 自身专利中查找对应族系
2. **NDA 渠道客户合作披露 / 询证函 / 律师函** — 取得 vendor 内部具体实现证据
3. **评估各候选所在法域下的 FRAND / patent pledge / 现有技术抗辩可能性**
4. **评估反诉风险 + 维权 ROI** — 头部 vendor 可能拥有大量反诉专利
5. **优先评估高 ROI 候选**：本表第 1 档（START 云游戏）+ 第 2 档（WebRTC / TRTC / Agora / NeRTC / Teams / Zoom）共 7 个候选构成"高置信侵权候选"，建议作为首要谈判 / 诉讼对象
6. **特别注意**：22-SRT 虽落"已排除（反向证据）"，但下游 Haivision Makito X / Cobalt / Wowza 等商业 SRT 编码器可能在 SRT 静态 FEC 上加自适应层（即 vendor 在 packet filter API 之外自行实现自适应控制），建议法务通过反向工程对该商业产品做单独取证
