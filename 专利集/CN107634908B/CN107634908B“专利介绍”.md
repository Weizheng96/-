## 一句话概括

本专利提出一种在传输前**根据数据流特征识别业务类型**，并结合**网络状态**、**传输成功率**、**时延要求****自适应计算冗余数据包数量与发送时间分布**的数据传输方法及对应设备，用以在丢包场景下既满足实时性又最大限度减少冗余开销。

## 解决的技术问题

应对网络丢包的两大传统手段——TCP 超时重传 与 线性网络编码的固定冗余传输——分别存在两个痛点：

1. **被动超时重传**：传输间隔大、时延高，无法支撑实时业务；
2. **固定速率冗余**：在数据已成功传输时仍盲发冗余，浪费带宽并加剧拥塞；在网络恶化时冗余量不够，反而退化到超时重传。

现有方案因此**无法对不同业务做差异化 QoS**——网络资源紧张、时延要求高的场景没有得到良好覆盖。本专利的核心目的就是：**让冗余数据包数量与调度策略随网络状态、业务类型、时延要求三者动态调整**，提供差异化的丢包恢复服务。

## 核心技术方案

1. **业务类型识别**：发送端读取数据流特征变量（包括缓存中待传输数据包的长度/数目、网络中在传数据包的长度/数目、数据包到达间隔、数据突发性 这 4 类中的至少 1 项），从中推断业务类型。识别方式包括门限值判断、特征变量值映射，或采用**人工智能 / 机器学习算法**。
2. **冗余数量计算**：以**网络状态变量**（丢包率、RTT、带宽、丢包分布模型——均匀/周期/高斯）+ **预先获取的传输成功率** + 已识别的**业务类型**为输入，计算需要发送的冗余数据包数量 N。说明书给了一个数值例：丢包率 5%、RTT 100 ms 场景下计算得 N = 2.07，取整为 3。
3. **冗余传输总时间窗口**：根据**预先获取的时延要求**，把冗余数据包发送总时间设置为**小于或等于时延要求**的一段时间。
4. **冗余调度方法生成**：以网络状态变量、传输总时间、冗余数据包数量为输入，生成具体调度方法。可选调度策略包括：**随机度策略 / 最短时间调度 / 最长时间调度 / 均匀时间调度**。
5. **可选预处理**：根据业务类型对待传输数据包做**编码 / 组合 / 拆分**预处理。
6. **执行发送**：按预处理方法发送原始数据包，再按冗余调度方法发送冗余数据包。

## 关键权利要求解读

### 权 1（方法权）拆解

权 1 是 5 步串行流程：

1. 由数据流特征变量 → 业务类型
2. 由网络状态变量 + 传输成功率 + 业务类型 → 冗余数据包数量
3. 由时延要求 → 冗余数据包传输总时间
4. 由网络状态变量 + 传输总时间 + 冗余数据包数量 → 冗余数据包调度方法
5. 按上述调度方法发送冗余数据包

注意 **5 步均冠以"发送端"主体**——这是一项严格意义上的"发送端实现"专利，而非接收端或网络中间设备。

### 权 8（设备权）拆解

权 8 把权 1 的 5 步方法**完整复述**到 "处理器执行存储器中的程序" 的执行结构上——即专利权人通过设备形式申请同一发明点的保护。**唯一的差异是侵权主体**：权 1 针对方法部署者，权 8 针对成品设备（含固化软件）制造商。

## 保护范围边界（重要）

### 必要技术特征清单 F1-F5（缺一不可，独立权字面要求）

权 1 和权 8 共享以下 F1-F5。逐特征逐 path 分析每个候选时，**任一独立权全部命中即构成侵权候选**。

- **F1**：**根据数据流特征变量获取业务类型**——数据流特征变量包括"缓存中待传输数据包的长度和数目 / 在传输数据包的长度和数目 / 数据包到达间隔 / 数据突发性"四项中的**至少一项**
- **F2**：**根据网络状态变量、传输成功率和业务类型，获取冗余数据包数量**——三输入决定冗余量
- **F3**：**根据时延要求获取冗余数据包传输总时间**
- **F4**：**根据网络状态变量、传输总时间、冗余数据包数量，获取冗余数据包调度方法**——三输入决定调度方法
- **F5**：**根据冗余数据包调度方法发送冗余数据包**——按上一步生成的调度方法实际执行发送

### 每条 F# 的 satisfaction paths（强制 — 含独立权字面 + 从属权 alt + 等同）

**F1（根据数据流特征变量获取业务类型）的可能 satisfaction paths**：
- Path 1.1（独立权字面 — 权 1/权 8）：直接以**缓存中待传输数据包的长度和数目、在网络中传输的数据包的长度和数目、数据包到达间隔、数据突发性**中的至少一项作为输入，输出业务类型分类
- Path 1.2（从属权 alt — 权 4/权 11）：业务类型可以来自"**默认值**"或"**应用层指定**"（如 SDK 调用方在传入 socket 时指定 service-type=video/audio/control），即业务类型并非必须由特征变量推断而出，**只要发送端"获取到"业务类型即满足**
- Path 1.3（从属权 alt — 权 5/权 12）：业务类型获取算法可以是**门限值判断**、**特征值映射**、或**AI/机器学习算法**——三选一即可
- Path 1.4（等同）：根据 socket type、应用打的 DSCP/ToS 标签、QoS class（如 5G 的 5QI、QUIC 的 stream priority、WebRTC 的 priority）做业务类型识别——这些是工业界对"业务类型"的等同表述

**F2（根据网络状态变量、传输成功率和业务类型，获取冗余数据包数量）的可能 satisfaction paths**：
- Path 2.1（独立权字面）：以丢包率 / RTT / 带宽 / 丢包分布模型中的至少一项 + 传输成功率（一次发送即成功的概率）+ 业务类型为输入，计算 FEC / 冗余包数量 N
- Path 2.2（从属权配合 — 权 6/权 13）：可在计算冗余数量后再做编码 / 组合 / 拆分预处理
- Path 2.3（等同）：根据 RTT + 丢包率自适应调整 FEC 冗余率的 AL-FEC / NACK-FEC / 主动重传策略；根据 RTT/RTO 自适应调整 packet pacing burst 数量；以"应用 QoS 等级 + 实时丢包统计"为输入决定 FEC 编码 N/K 参数

**F3（根据时延要求获取冗余数据包传输总时间）的可能 satisfaction paths**：
- Path 3.1（独立权字面）：**显式以应用层时延预算 / Deadline 为输入**，输出冗余包发送窗口长度
- Path 3.2（从属权 alt — 权 3/权 10）：将冗余包传输总时间设置为 ≤ 时延要求（即"卡死在 deadline 之内"是合法路径）
- Path 3.3（等同）：QUIC / SRT 中的 PTO（probe timeout）阈值、TS-SRT 的 latency setting、WebRTC 的 jitter buffer + retransmit deadline、RIST 的 reorder buffer、5G uRLLC 的 PDB（packet delay budget）作为冗余包发送窗口上界

**F4（根据网络状态变量、传输总时间、冗余数据包数量，获取冗余数据包调度方法）的可能 satisfaction paths**：
- Path 4.1（独立权字面）：基于三输入（网络状态、传输总时间、冗余数量）输出调度方法
- Path 4.2（从属权枚举 — 权 2/权 9）：调度方法 = **随机度策略 / 最短时间调度 / 最长时间调度 / 均匀时间调度** 中的至少一种
- Path 4.3（等同）：在 transmit window 内的 packet pacing、随机抖动、burst 平铺、按 RTT 比例分布等任何"决定 N 个冗余包以何种时间间隔发送"的算子；尤其 packet pacing rate 控制（按当前 cwnd 与发送窗口长度计算单包间隔）

**F5（根据冗余数据包调度方法发送冗余数据包）的可能 satisfaction paths**：
- Path 5.1（独立权字面）：按 F4 输出的调度方法，将冗余数据包实际经由网络 socket 发出
- Path 5.2（从属权配合 — 权 7/权 14）：先按预处理方法（编码 / 组合 / 拆分）发送原始数据包，再发送冗余包——这是常见的 FEC 发送顺序

### 等同特征讨论（doctrine of equivalents）

业内对应表述等同关系（在"侵权三步法 — 方式 / 功能 / 结果"判定下高度可能落入保护范围）：

- "**自适应 FEC**" / "**adaptive FEC**" / "**动态冗余编码**"——多家 RTC / 直播 / 云游戏 / 视频会议厂商常用的对外营销词，几乎都是这套 RTT-loss-loop 控制 FEC 速率的实现
- "**业务感知**" / "**service-aware FEC**" / "**QoS-aware redundancy**"——把应用层语义（视频 I 帧、音频高优先级、控制信令）映射到不同 FEC 强度
- "**Deadline-aware**" / "**delay-budgeted FEC**"——把应用 deadline 作为 FEC 发送时间上界
- "**packet pacing + FEC interval scheduling**"——把 FEC 冗余包按 packet pacing burst 拉开

### 明显不在保护范围内的相邻方案（用作反向排除参考）

- **接收端单独的 NACK 重传**：纯接收端反馈机制，没有发送端的 F1-F5 全链路，落不进保护范围
- **网络中间设备（路由 / SDN 控制器）的丢包统计与限速**：主体不是发送端
- **完全固定的 FEC 冗余率**（k=N/M 写死）：背景技术明确批评的"现有线性网络编码固定冗余"——**不可能命中 F2 / F3 / F4**，因为缺少"由网络状态 / 传输成功率 / 业务类型 / 时延要求 自适应"这一关键步骤
- **TCP 自身的超时重传（RTO）**：背景技术明确否定的现有方案，不构成侵权
- **应用层简单复制包发送**（如 UDP 应用裸复制 2 次发包），但**不存在业务类型识别 + 网络状态评估 + 时延约束**三者联动：也不命中 F1 / F2 / F3
- **接收端 FEC 解码**：只是接收侧实施，主体不对
