# CN107634908B 潜在应用场景及侵权特征

## 基本信息
- 专利号：CN107634908B
- 标题：一种数据传输的方法和设备
- 当前权利人：Huawei Technologies Co Ltd（华为技术有限公司）
- 申请日 / 公开（授权）日：2016-07-19 / 2021-06-08

## 一、潜在应用场景（严格来自背景技术 — 不外扩）

> 背景技术段落集中描述了"TCP 超时重传机制无法满足实时业务时延要求"以及"现有线性网络编码冗余传输方法浪费网络资源、无法差异化 QoS"两类痛点。说明书在背景结束后给出了"实施例 1"作为应用场景锚定：物联网用户控制消息下发汽车 + 显式扩展到在线游戏 / 数据中心保活 / 机器人控制 / 虚拟现实 / 全息投影体感信息 / 电视电话会议 / 在线视频。下列场景全部 verbatim 引自专利原文。

### 场景 1：物联网 / 车联网稀疏控制消息下发
- **背景技术 verbatim 依据**：> "在此场景中，用户101通过无线接入点102A接入物联网云平台100，把用户101控制消息下发到汽车104。…物理网中的用户控制消息等具有极为稀疏的特点，即很长一段时间内仅有几个数据包需要传输。因为稀疏，这些消息无法触发TCP的快重传机制、一旦丢失，仅能依靠TCP超时重传机制完成传输。"
- **场景说明**：物联网 / 车联网云平台向终端（汽车、设备）下发控制消息，业务特点是数据稀疏（无法触发 TCP 快重传）、无线信道劣化、时延要求高。
- **使用本专利的典型形式**：物联网平台的传输层 / 数据链路层协议栈中内嵌"业务类型识别 + 冗余数据包数量自适应 + 冗余调度方法"的传输模块（背景技术明示"本发明实施例主要改进了该网元的硬件设备中运行的协议栈程序"）。

### 场景 2：在线游戏低时延传输
- **背景技术 verbatim 依据**：> "除了上述物联网场景，本发明实施例提出的方法也可以用于其它有低延时需要的场景，如在线游戏…"；> "例如在线游戏就对时延要求非常高，稍稍的时延都会大大降低用户体验"
- **场景说明**：在线游戏（特别是 MOBA / FPS / 云游戏）操作指令、状态同步对时延极敏感，丢包重传会破坏体验。
- **使用本专利的典型形式**：游戏客户端 / 服务端的网络加速 SDK 或网关，按操作指令业务类型自适应生成冗余包并按时延要求调度发送。

### 场景 3：数据中心间保活信息实时传输
- **背景技术 verbatim 依据**：> "本发明实施例提出的方法也可以用于其它有低延时需要的场景，如…数据中心间保活信息的实时传输"
- **场景说明**：跨数据中心心跳 / 保活 / 集群状态同步报文（稀疏 + 实时），丢失会触发集群误判主备切换。
- **使用本专利的典型形式**：内嵌于跨 DC 互联传输协议 / 长连接保活模块的自适应冗余传输逻辑。

### 场景 4：机器人控制
- **背景技术 verbatim 依据**：> "本发明实施例提出的方法也可以用于其它有低延时需要的场景，如…机器人控制"
- **场景说明**：工业机器人 / 服务机器人 / 远程遥操作机器人接收控制指令——指令稀疏、实时性强、丢一帧可能造成物理动作错误。
- **使用本专利的典型形式**：机器人控制器与上位机间的实时通信协议栈中内嵌业务类型识别 + 冗余调度模块。

### 场景 5：虚拟现实 / 全息投影体感信息实时传递
- **背景技术 verbatim 依据**：> "在未来，本发明实施例提出的方法还可以用于虚拟现实、全息投影中体感信息的实时传递等。"
- **场景说明**：VR / 全息投影中体感（位姿、力反馈、触觉）信息流时延要求毫秒级，丢包会引起眩晕或动作脱节。
- **使用本专利的典型形式**：VR / 全息传输平台的端到端协议层冗余调度模块。

### 场景 6：电视电话会议 / 在线视频
- **背景技术 verbatim 依据**：> "业务类型是指待传输数据的实际业务类型，包括但不限于，控制消息、在线游戏、在线视频或电视/电话会议等。"
- **场景说明**：实时音视频会议、直播——音视频包对时延敏感，部分包（如关键帧 / 控制信令）丢失影响极大。
- **使用本专利的典型形式**：音视频 SDK / RTC 协议栈中的丢包对抗模块（按业务子类型——关键帧 / 普通帧 / 控制信令——自适应生成冗余包并调度）。

## 二、独立权利要求 + 侵权特征 F1-Fk

### 权 1 verbatim

> 1.一种数据的传输方法，其特征在于，所述方法包括：
> 发送端根据待传输数据的数据流特征变量，获取所述待传输数据的业务类型，其中，所述数据流特征变量包括发送端缓存中待传输数据包的长度和数目、正在网络中传输的数据包的长度和数目、数据包的到达间隔、数据突发性中的至少一项；
> 发送端根据预先获取的网络状态变量、预先获取的传输成功率和所述业务类型，获取所述待传输数据包需要传输的冗余数据包的数量；
> 发送端根据预先获取的时延要求，获取冗余数据包的传输总时间；
> 发送端根据所述网络状态变量、所述传输总时间和所述冗余数据包的数量，获取冗余数据包调度方法；
> 发送端根据所述冗余数据包调度方法发送冗余数据包。

### F1-Fk 清单

- **F1（数据流特征变量 → 业务类型）**："发送端根据待传输数据的数据流特征变量，获取所述待传输数据的业务类型，其中，所述数据流特征变量包括发送端缓存中待传输数据包的长度和数目、正在网络中传输的数据包的长度和数目、数据包的到达间隔、数据突发性中的至少一项"
  - 隐含约束：数据流特征变量**至少一项**来自指定的 4 类（缓存包长/数、在传包长/数、到达间隔、突发性）；**不能**只由应用层指定（应用层指定属说明书可选实现，非独权字面）。

- **F2（网络状态变量 + 传输成功率 + 业务类型 → 冗余数据包数量）**："发送端根据预先获取的网络状态变量、预先获取的传输成功率和所述业务类型，获取所述待传输数据包需要传输的冗余数据包的数量"
  - 隐含约束：必须**同时**使用 3 个输入——网络状态变量、传输成功率、业务类型；只用其中 2 项不构成字面命中。

- **F3（时延要求 → 冗余数据包传输总时间）**："发送端根据预先获取的时延要求，获取冗余数据包的传输总时间"
  - 隐含约束：传输总时间由时延要求**显式推导**得到（说明书给出"小于或等于时延要求"的设置方法）；固定值或经验值不算 F3 命中。

- **F4（网络状态变量 + 传输总时间 + 冗余数量 → 冗余调度方法）**："发送端根据所述网络状态变量、所述传输总时间和所述冗余数据包的数量，获取冗余数据包调度方法"
  - 隐含约束：必须**同时**用到 3 个输入；调度方法可为随机/最短/最长/均匀 4 类之一（说明书穷举）；固定单一调度（如永远均匀）不构成 F4。

- **F5（按调度方法发送冗余数据包）**："发送端根据所述冗余数据包调度方法发送冗余数据包"
  - 隐含约束：发送行为本身必须按 F4 输出的调度方法实施；同时根据说明书"一旦原数据包确认则停止冗余发送"——这是 F5 的关键运行时表现，可作等同命中信号。

### 其他独立权（仅当存在多条独立权时填）

- **权 8（设备形式 — 处理器 + 存储器执行方法）**：权 8 字面是权 1 方法的设备载体（"所述设备包括：处理器和存储器；所述存储器用于存储程序；所述处理器用于执行所述存储器中的程序，用以实现下述方法：…"），方法步骤与权 1 完全同构。任何命中权 1 五特征的实现，其承载硬件设备同步落入权 8 保护范围——Step 6 时无需单独分析权 8，沿用权 1 的 F1-F5 即可。

### 关键限定词与隐含约束

- 数据流特征变量必须是 4 类**至少一项**："发送端缓存中待传输数据包的长度和数目"、"正在网络中传输的数据包的长度和数目"、"数据包的到达间隔"、"数据突发性"——超出这 4 类的特征不算 F1 字面命中。
- 冗余数据包数量必须由"网络状态变量 + 传输成功率 + 业务类型" 3 个输入共同决定（F2 字面）——仅用网络丢包率反推冗余量、不含业务类型分支，是反向证据信号。
- 冗余调度方法的 4 种穷举：随机度策略、最短时间调度方法、最长时间调度方法、均匀时间调度方法。
- 传输总时间限定：≤ 时延要求（说明书"小于或等于"措辞）。
- 网络状态变量列举：丢包率、RTT、网络带宽、丢包分布模型（均匀 / 周期 / 高斯）——可用其一或多种。
- 业务类型列举：控制消息、在线游戏、在线视频、电视/电话会议等（开放枚举）；业务类型可由"默认值 / 应用指定 / 数据流特征变量推断" 3 种途径之一获取。
- 整数限定：冗余数据包数量为整数（说明书 S303 示例显示理论值 2.07 → 取 3，即向上取整 / 非整数需补到整数）。
- 终止条件：若原数据包在冗余调度方法执行过程中被确认，发送端立即停止剩余冗余包发送（实施例 4 显式描述 "ACK 到达即停止 R12/R13"）。这是 F5 的运行时关键表现，等同命中场景下可作判定依据。
- 适用网元：通用计算机 + 终端 + 服务器，凡支持 TCP 协议者皆可（说明书"网元可以是…任何支持TCP协议的设备"）——但本专利改进点在协议栈传输层 / 数据链路层，纯应用层冗余（如应用层 FEC）原则上不落入权 1 保护范围。
