## 一、潜在应用场景（严格来自背景技术 / 实施例 — 不外扩）

### 场景 1：物联网稀疏控制消息的实时传输
- **背景技术 verbatim 依据**：> "用户101通过无线接入点102A接入物联网云平台100，把用户101控制消息下发到汽车104……物理网中的用户控制消息等具有极为稀疏的特点，即很长一段时间内仅有几个数据包需要传输。因为稀疏，这些消息无法触发TCP的快重传机制、一旦丢失，仅能依靠TCP超时重传机制完成传输……本发明实施例提供的方法根据网络状态和业务类型提供自适应的主动冗余传输，可以指数级降低TCP超时重传的概率、保证控制消息传输的实时性。"
- **场景说明**：IoT / 车联网平台向终端下发稀疏控制指令，弱网下用主动冗余（FEC）替代 TCP 超时重传，保证指令实时送达。
- **使用本专利的典型形式**：内嵌于设备 / 平台协议栈（传输层或数据链路层）的自适应冗余传输算法模块。

### 场景 2：在线游戏 / 实时音视频 / 电话电视会议
- **背景技术 verbatim 依据**：> "业务类型是指待传输数据的实际业务类型，包括但不限于，控制消息、在线游戏、在线视频或电视/电话会议等。"　以及　> "在线游戏就对时延要求非常高，稍稍的时延都会大大降低用户体验；相比于在线游戏，视频、音乐下载对时延的要求相对会低一些。"
- **场景说明**：实时音视频通信（RTC）/ 云游戏 / 在线会议在弱网下用业务类型感知的自适应 FEC 控制冗余量与发包节奏，兼顾低时延与抗丢包。
- **使用本专利的典型形式**：RTC / 音视频 SDK 或传输引擎内的自适应 FEC + 拥塞 / 冗余调度模块。

### 场景 3：数据中心间保活信息实时传输 / 机器人控制
- **背景技术 verbatim 依据**：> "本发明实施例提出的方法也可以用于其它有低延时需要的场景，如在线游戏、数据中心间保活信息的实时传输、机器人控制；在未来，本发明实施例提出的方法还可以用于虚拟现实、全息投影中体感信息的实时传递等。"
- **场景说明**：数据中心间 keep-alive / 机器人远程控制 / VR / 全息体感等低时延场景下的主动冗余传输。
- **使用本专利的典型形式**：传输层 / 协议栈内的自适应冗余调度（与场景 1/2 同一机制，应用域不同）。

## 二、独立权利要求 + 侵权特征 F1-F5

本专利有 **2 项独立权利要求**：权 1（方法）与权 8（设备）。两者技术内容一致，权 8 为"处理器+存储器执行权1方法"的装置形式。F1-F5 以权 1 字面拆分。

### 权 1 verbatim
> 1.一种数据的传输方法，其特征在于，所述方法包括：
> 发送端根据待传输数据的数据流特征变量，获取所述待传输数据的业务类型，其中，所述数据流特征变量包括发送端缓存中待传输数据包的长度和数目、正在网络中传输的数据包的长度和数目、数据包的到达间隔、数据突发性中的至少一项；
> 发送端根据预先获取的网络状态变量、预先获取的传输成功率和所述业务类型，获取所述待传输数据包需要传输的冗余数据包的数量；
> 发送端根据预先获取的时延要求，获取冗余数据包的传输总时间；
> 发送端根据所述网络状态变量、所述传输总时间和所述冗余数据包的数量，获取冗余数据包调度方法；
> 发送端根据所述冗余数据包调度方法发送冗余数据包。

### F1-F5 清单
- **F1（业务类型识别 — 数据流特征变量驱动）**：> "发送端根据待传输数据的数据流特征变量，获取所述待传输数据的业务类型，其中，所述数据流特征变量包括发送端缓存中待传输数据包的长度和数目、正在网络中传输的数据包的长度和数目、数据包的到达间隔、数据突发性中的至少一项"
- **F2（冗余数据包数量 = f(网络状态变量, 传输成功率, 业务类型)）**：> "发送端根据预先获取的网络状态变量、预先获取的传输成功率和所述业务类型，获取所述待传输数据包需要传输的冗余数据包的数量"
- **F3（冗余包传输总时间 = f(时延要求)）**：> "发送端根据预先获取的时延要求，获取冗余数据包的传输总时间"
- **F4（冗余包调度方法 = f(网络状态变量, 传输总时间, 冗余包数量)）**：> "发送端根据所述网络状态变量、所述传输总时间和所述冗余数据包的数量，获取冗余数据包调度方法"
- **F5（按调度方法发送冗余包）**：> "发送端根据所述冗余数据包调度方法发送冗余数据包"

**拆分规则说明**：权 1 主体为五个并列动作步骤（业务类型识别 → 冗余量计算 → 传输总时间计算 → 调度方法计算 → 发送），逐一拆为 F1-F5，未合并、未意译。

### 其他独立权（权 8 — 设备形式）
- 权 8（设备：处理器+存储器）：> "一种数据的传输设备……处理器用于执行所述存储器中的程序，用以实现下述方法"，其 method body 与权 1 的 F1-F5 逐句对应（仅"发送端"主语换为"处理器实现的方法"）。判定候选时，软硬件一体产品（含处理器+存储器执行该传输算法的设备 / 芯片 / 终端）落入权 8；纯软件 SDK / 协议栈实现落入权 1。

### 关键限定词与隐含约束
- **F1 "至少一项"（开放枚举）**：数据流特征变量列举了 4 类（缓存包长度数目 / 在网包长度数目 / 到达间隔 / 突发性），命中**任一项**即可，且"包括"为开放枚举，清单外的等效流特征也可能命中。
- **F2 三输入耦合**：冗余包数量必须**同时**基于"网络状态变量 + 传输成功率 + 业务类型"三者；若候选只按丢包率（网络状态）算冗余、未引入业务类型维度 → F2 可能仅部分命中（需细看是否做业务/流类型区分）。
- **F1↔F2 的"业务类型"链条**：F1 产出的业务类型是 F2 的输入；二者共同体现"业务类型感知的自适应冗余"——这是本专利区别于固定冗余 FEC 的核心。候选若只做"网络自适应 FEC"而**无业务/内容类型区分**，是 F1/F2 的关键缺失点。
- **F3→F4 的"时延要求 → 传输总时间 → 调度"链条**：调度方法须考虑由时延要求导出的传输总时间（即冗余包要在一个 ≤ 时延要求的时间窗内发完），体现"时延感知的发包节奏调度"，而非简单的固定间隔发包。
- **调度方法的具体形式（权 2 / 权 9）**：随机调度 / 最短时间调度 / 最长时间调度 / 均匀时间调度——四选一或多。候选若用"在一个时间窗内按某节奏分散发送冗余包"即可能命中 F4。
- **"主动冗余"而非"被动重传"**：本专利针对的是 FEC / 前向纠错式主动冗余，区别于 ARQ / TCP 超时重传。候选若纯靠重传（ARQ）而无前向冗余 → 领域层面即偏离。
