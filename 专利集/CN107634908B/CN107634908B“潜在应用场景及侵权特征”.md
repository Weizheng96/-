## 一、潜在应用场景（严格来自背景技术 — 不外扩）

### 场景 1：实时音视频 / 会议的低时延抗丢包传输
- **背景技术 verbatim 依据**：> "应对网络丢包的基本策略是TCP…超时重传机制。…存在不能解决实时业务对时延要求高的技术问题。现有技术中较为先进的避免重传的方法是，采用线性网络编码技术对数据进行主动冗余传输，即在传输数据包的同时根据某个较小的、固定的时间间隔进行待传输数据包的冗余数据包的传输"；发明内容举例业务类型 > "包括但不限于，控制消息、在线游戏、在线视频或电视/电话会议等"
- **场景说明**：实时音视频 / 电话会议 / 直播场景下，发送端用主动冗余（FEC / 网络编码）抗丢包，又要把冗余量压到最小以免拥塞——正是本专利"自适应冗余调度"针对的核心痛点。
- **使用本专利的典型形式**：内嵌在实时传输 SDK / 媒体引擎的发送侧拥塞-FEC 控制模块（按网络状态 + 业务类型自适应决定冗余包数量与发送调度）。

### 场景 2：在线游戏 / 实时控制消息传输
- **背景技术 verbatim 依据**：> 业务类型 "包括但不限于，控制消息、在线游戏…等"；> "现有技术仅适用于用户的网络资源较为丰富、时延要求不高的业务场景；无法满足网络资源较少、时延要求高的业务场景，现有技术没有提供差异化的QoS服务"
- **场景说明**：在线游戏、云游戏、实时控制信令等对时延极敏感的业务，按业务类型做差异化 QoS 的主动冗余传输。
- **使用本专利的典型形式**：游戏 / 云游戏实时网络传输层，或通用实时传输协议栈中按业务类型分级的自适应 FEC 冗余控制。

### 场景 3：差异化 QoS 的通用实时数据传输（含弱网 / 移动网）
- **背景技术 verbatim 依据**：> "现有技术没有提供差异化的QoS(Quality of Service，服务质量)服务"；> "无法满足网络资源较少、时延要求高的业务场景"
- **场景说明**：在弱网 / 移动 / 高丢包网络上，需要按时延要求与网络状态自适应调整冗余量、提供差异化 QoS 的通用传输方案（不限定具体上层业务）。
- **使用本专利的典型形式**：传输层 / 网络栈中的自适应前向纠错（FEC）+ 冗余调度算法，或开源实时传输库中的该类机制。

## 二、独立权利要求 + 侵权特征 F1-Fk

### 权 1 verbatim
> 1.一种数据的传输方法，其特征在于，所述方法包括：
> 发送端根据待传输数据的数据流特征变量，获取所述待传输数据的业务类型，其中，所述数据流特征变量包括发送端缓存中待传输数据包的长度和数目、正在网络中传输的数据包的长度和数目、数据包的到达间隔、数据突发性中的至少一项；
> 发送端根据预先获取的网络状态变量、预先获取的传输成功率和所述业务类型，获取所述待传输数据包需要传输的冗余数据包的数量；
> 发送端根据预先获取的时延要求，获取冗余数据包的传输总时间；
> 发送端根据所述网络状态变量、所述传输总时间和所述冗余数据包的数量，获取冗余数据包调度方法；
> 发送端根据所述冗余数据包调度方法发送冗余数据包。

### F1-F5 清单
- **F1（业务类型识别）**：> "发送端根据待传输数据的数据流特征变量，获取所述待传输数据的业务类型，其中，所述数据流特征变量包括发送端缓存中待传输数据包的长度和数目、正在网络中传输的数据包的长度和数目、数据包的到达间隔、数据突发性中的至少一项"
- **F2（冗余包数量自适应计算）**：> "发送端根据预先获取的网络状态变量、预先获取的传输成功率和所述业务类型，获取所述待传输数据包需要传输的冗余数据包的数量"
- **F3（冗余传输总时间）**：> "发送端根据预先获取的时延要求，获取冗余数据包的传输总时间"
- **F4（冗余调度方法生成）**：> "发送端根据所述网络状态变量、所述传输总时间和所述冗余数据包的数量，获取冗余数据包调度方法"
- **F5（按调度方法发送冗余包）**：> "发送端根据所述冗余数据包调度方法发送冗余数据包"

**拆分规则核对**：权 1 在"其特征在于，所述方法包括："后有 5 个并列动作步骤（业务类型识别 → 冗余数量 → 传输总时间 → 调度方法 → 发送），逐一对应 F1-F5，无合并、无意译。

### 其他独立权（存在多条独立权）
- **权 8（设备/装置类，处理器+存储器形式）**：> "一种数据的传输设备…处理器…执行…程序，用以实现下述方法：" 其后所实现方法与权 1 的 F1-F5 步骤实质一致（业务类型识别 → 冗余数量 → 传输总时间 → 调度方法 → 发送）。即权 8 = 权 1 方法的"设备载体"权，命中判定可复用 F1-F5。

### 关键限定词与隐含约束
- **三输入冗余量约束**：冗余包数量须**同时**依赖「网络状态变量 + 传输成功率 + 业务类型」三者（F2）——仅靠丢包率单因子算冗余、或固定冗余比例的方案不完全落 F2。
- **业务类型由数据流特征变量自动推断**：F1 要求业务类型来自「缓存包长度/数目、在网包长度/数目、到达间隔、数据突发性」中至少一项的特征变量（自动识别），而非上层应用直接声明（应用声明属从权 4 的"应用制定的业务类型"分支）。
- **时延要求 → 传输总时间 → 调度三级链**：F3/F4 要求先由时延要求得出冗余传输总时间，再据此与网络状态、冗余数量共同生成调度方法（从权 3 进一步限定总时间 ≤ 时延要求）。
- **主动冗余（FEC / 网络编码）而非被动重传**：背景技术明确区别于 TCP 超时重传 / 纯 ARQ；命中对象应是带主动 FEC / 冗余包发送的实时传输方案。
- **调度方法可选集合（从权 2/9）**：随机度策略、最短时间调度、最长时间调度、均匀时间调度——属可选实现，不构成权 1 必要技术特征。
- **自适应核心**：区别于"固定时间间隔发送固定冗余"的现有技术，本专利核心是冗余**数量**与**调度时间**双自适应。
