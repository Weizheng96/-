## 一、潜在应用场景（严格来自背景技术 + 实施例 1）

> 说明：原专利"背景技术"段仅讨论"TCP 超时重传 vs 线性网络编码主动冗余"的技术演进，未直接列举具体应用场景。**应用场景定锚来自说明书"实施例 1"中明示的应用场景段** —— 该段就是背景技术段对场景的延展（说明书第 1/3 段）。引文均 verbatim。

### 场景 1：物联网（IoT）用户控制消息下发
- **verbatim 依据**：> "图2给出了本发明实施例的一种应用场景。在此场景中，用户101通过无线接入点102A接入物联网云平台100，把用户101控制消息下发到汽车104。…物理网中的用户控制消息等具有极为稀疏的特点，即很长一段时间内仅有几个数据包需要传输。因为稀疏，这些消息无法触发TCP的快重传机制、一旦丢失，仅能依靠TCP超时重传机制完成传输。"
- **场景说明**：稀疏控制消息（车控 / 远程操作 / 设备指令）通过无线接入点上行到云平台，需要在不可靠无线链路上做**主动冗余传输**以替代被动 TCP 超时重传；典型时延要求 << 1s。
- **使用本专利的典型形式**：嵌入云端 + 终端协议栈（传输层 / 数据链路层）的冗余调度逻辑；可以是 IoT 平台 SDK、车联网 V2X 协议、OTA 下发通道。

### 场景 2：在线游戏（低延时控制 / 状态同步流）
- **verbatim 依据**：> "本发明实施例提出的方法也可以用于其它有低延时需要的场景，如在线游戏、数据中心间保活信息的实时传输、机器人控制"
- **场景说明**：在线游戏的玩家输入 / 角色状态 / 战斗事件 UDP 数据流对时延高度敏感（毫秒级），需要根据丢包率自适应地决定是否多发冗余副本。
- **使用本专利的典型形式**：游戏服务器 / 客户端的网络传输库；游戏专用加速 SDK；云游戏控制流通道。

### 场景 3：数据中心之间（DC interconnect）保活与控制消息实时传输
- **verbatim 依据**：（同上"…数据中心间保活信息的实时传输…"）
- **场景说明**：跨地域 DC 之间的 keepalive / heartbeat / 分布式一致性控制消息（典型小包 + 低延时要求），通过广域 / 专线传输；丢包易致服务震荡。
- **使用本专利的典型形式**：DC 互联协议栈 / 私有 RPC 框架 / SDN 控制面冗余传输层。

### 场景 4：机器人 / 远程操控
- **verbatim 依据**：（同上"…机器人控制"）
- **场景说明**：工业机器人 / 服务机器人 / 远程手术 / 远程驾驶等控制信令对实时性敏感；无线或非可靠链路上需要冗余主动传输。
- **使用本专利的典型形式**：机器人 SDK 通信模块；远程操控低延时协议栈。

### 场景 5：虚拟现实 / 全息投影中体感信息的实时传递
- **verbatim 依据**：> "在未来，本发明实施例提出的方法还可以用于虚拟现实、全息投影中体感信息的实时传递等。"
- **场景说明**：VR / 全息系统的体感反馈（位置、姿态、力反馈）流，对时延和丢包都敏感。
- **使用本专利的典型形式**：VR 串流协议栈 / 云 VR 平台传输层。

### 场景 6：在线视频 / 电视电话会议
- **verbatim 依据**：> "业务类型是指待传输数据的实际业务类型，包括但不限于，控制消息、在线游戏、在线视频或电视/电话会议等"
- **场景说明**：实时视频流（直播、视频会议、低延时点播）经由不可靠网络传输时，按丢包率与业务类型动态决定冗余包数量与调度方式。
- **使用本专利的典型形式**：RTC 引擎 / 视频会议 SDK 的内置 FEC / 自适应冗余模块。

## 二、独立权利要求 + 侵权特征 F1-Fk

### 权 1 verbatim
> 1.一种数据的传输方法，其特征在于，所述方法包括：
> 发送端根据待传输数据的数据流特征变量，获取所述待传输数据的业务类型，其中，所述数据流特征变量包括发送端缓存中待传输数据包的长度和数目、正在网络中传输的数据包的长度和数目、数据包的到达间隔、数据突发性中的至少一项；
> 发送端根据预先获取的网络状态变量、预先获取的传输成功率和所述业务类型，获取所述待传输数据包需要传输的冗余数据包的数量；
> 发送端根据预先获取的时延要求，获取冗余数据包的传输总时间；
> 发送端根据所述网络状态变量、所述传输总时间和所述冗余数据包的数量，获取冗余数据包调度方法；
> 发送端根据所述冗余数据包调度方法发送冗余数据包。

### F1-F5 清单

- **F1（业务类型识别 — 基于数据流特征变量）**：> "发送端根据待传输数据的数据流特征变量，获取所述待传输数据的业务类型，其中，所述数据流特征变量包括发送端缓存中待传输数据包的长度和数目、正在网络中传输的数据包的长度和数目、数据包的到达间隔、数据突发性中的至少一项"
- **F2（冗余包数量 — 三元组联合计算：网络状态 + 成功率 + 业务类型）**：> "发送端根据预先获取的网络状态变量、预先获取的传输成功率和所述业务类型，获取所述待传输数据包需要传输的冗余数据包的数量"
- **F3（冗余包传输总时间 — 来自时延要求）**：> "发送端根据预先获取的时延要求，获取冗余数据包的传输总时间"
- **F4（调度方法 — 三元组联合计算：网络状态 + 传输总时间 + 冗余包数量）**：> "发送端根据所述网络状态变量、所述传输总时间和所述冗余数据包的数量，获取冗余数据包调度方法"
- **F5（按调度方法发送冗余包）**：> "发送端根据所述冗余数据包调度方法发送冗余数据包"

### 其他独立权
- **权 8**（设备形式 — 处理器 + 存储器）：方法步骤与权 1 完全相同，仅形式从"方法"变为"设备"。F# 集合等价。Step 6 时若证据是产品 / 终端设备形态，可直接套权 8；若证据是云端服务行为，仍套权 1（"发送端"可以是云端推流节点 / SDK 进程）。

### 关键限定词与隐含约束

- **F1 限定**："数据流特征变量"必须从下列 4 项中**至少包含 1 项**：
  - (a) 发送端缓存中待传输数据包的**长度和数目**
  - (b) 正在网络中传输的数据包的**长度和数目**
  - (c) 数据包的**到达间隔**
  - (d) **数据突发性**
  - 隐含：必须**由发送端自动从真实流量统计得到**，不是上层应用静态配置或用户手填的"业务类型"标签。若候选仅由应用层 hint / DSCP tag / 配置文件指明业务类型（=权 1 中"应用制定的业务类型"的从属权 4 形态），**不命中权 1**。
- **F2 三元组联合**："网络状态变量 + 传输成功率 + 业务类型"三者均参与冗余包数量计算。**缺业务类型则不命中权 1**（缺业务类型的方案是从属权 1 之父发明内容描述的更宽抽象，但权 1 本身把"业务类型"硬钉入了 F2）。
- **F3 限定**：冗余包传输**总时间**由时延要求决定（典型应小于等于时延要求）；不是单包 timeout，是 n 个冗余包累计的发送窗口长度。
- **F4 三元组联合**："网络状态 + 总时间 + 冗余包数量"三者均参与调度方法选择。固定调度（e.g. 永远等间隔）若不动态参考网络状态则**反向证据**。
- **F4 调度方法的可选集合（权 2 / 权 9 限定）**：随机度策略、最短时间、最长时间、均匀时间——**至少 1 种**。Step 6 时这条用于辅助识别"是否真在动态选调度"。
- **整数 ≥ 1 限定**：冗余包数量是整数，**至少 1 个**（n=0 不构成"发送冗余包"，对应直接传输无冗余的退化情形 → 反向证据信号）。
- **"发送端"主体**：可以是终端 SDK、服务器进程、网关、协议栈中的任一逻辑发送节点；只要"发送端"完整执行 F1-F5，即落入权 1（即使分布式拆分到多模块）。
- **F1-F5 顺序无关**：权 1 描述了一组方法步骤的并列限定，专利说明书显式说明 S200 / S201 / S202 顺序可调；只要 5 个动作都执行即命中。
- **"主动冗余"vs"重传 / ARQ"区别**：本专利核心是**主动 FEC / packet-level 冗余**（发送前主动生成 n 个冗余副本）；纯 ARQ / TCP 超时重传机制**不构成 F5 命中**——它们是 reactive 重传不是 proactive 冗余。
- **"基于业务类型决策"是核心区别**：相比"现有技术线性网络编码"（不区分业务类型，固定时间间隔），本专利的关键创新在于**业务类型自适应**——若候选只做了网络状态自适应但不做业务类型区分（即 F1/F2 中"业务类型"输入缺失），即使做了 F3/F4 也不命中权 1。
