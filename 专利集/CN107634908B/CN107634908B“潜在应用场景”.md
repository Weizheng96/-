## 推演方法

从专利核心特征出发：
- **F1** "数据流特征 → 业务类型识别" → 适用于发送端可观测应用语义、且不同业务（视频 / 音频 / 控制 / 文件传输）有差异化 QoS 需求的场景
- **F2-F4** "网络状态 + 传输成功率 + 时延要求 → 冗余数量 + 总时间窗口 + 调度方法" → 适用于网络丢包率波动、且应用层有明确 deadline / 时延预算的场景
- **F5 + 权 7/14** "调度发送 + 预处理（编码 / 组合 / 拆分）" → 即典型 FEC + 复用编码场景

结合 2026 年产业现状（5G+ 商业化深入、云游戏 / VR 串流向消费市场普及、视频会议远程办公常态化、TikTok / 抖音直播全球月活破 20 亿、3GPP 5G-A 标准引入 XR / cloud gaming 专项 QoS class），推演 6 个高命中场景，按可能命中权利要求的可能性由高到低排列。

## 反例总结（同样关键 — 防误判）

下列场景看似相关但**不构成 F1-F5 的全链路实施**，应明确排除：

- **接收端纯 NACK 重传 / jitter buffer 平滑**：主体不对，不命中"发送端"限定
- **网络中间设备（SDN / SD-WAN 控制器）的丢包统计 + 限速**：主体不对
- **完全固定的 FEC 冗余率 / RS 编码 (k, n) 写死**：背景技术明确否定，不命中 F2 / F4
- **TCP RTO 超时重传 / SACK**：背景技术明确否定的现有方案
- **应用层简单复制包发送（无业务类型识别 + 无网络状态评估 + 无时延约束）**：缺 F1 / F2 / F3 中任意一项即不命中
- **接收端 FEC 解码**：主体不对
