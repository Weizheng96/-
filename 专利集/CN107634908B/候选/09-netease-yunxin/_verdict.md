# 09-netease-yunxin verdict

## 候选基本信息（专利公开日 2021-06-08）
- 类型：产品
- 名称：网易云信 NERTC（NetEase Real-Time Communication）
- 组织：网易（网易云信事业部）
- 公开度：中
- 时间窗：NERTC 自 2018 年起对外提供；技术博客发布时间集中在 2020-12 / 2021-04 / 2023-04；本次评估侧重专利公开日 2021-06-08 后仍持续提供的对外服务行为 → 时间窗整体合规
- 一句话定位：网易对外 RTC SDK，自家游戏 + 在线教育 + 第三方接入复用

## F# 命中表（F1-F5）

| F# | 限定 | 网易云信公开技术细节 | 命中性 |
|---|---|---|---|
| **F1** | 业务类型由发送端**根据数据流特征变量（缓存包长度数目 / 网络中包长度数目 / 到达间隔 / 突发性 ≥1 项）自动识别** | 业务/场景由上层 App 通过 `setChannelProfile(channelProfile)` 显式指定（COMMUNICATION / LIVE_BROADCASTING / VideoCall / HighQualityVideoCall / Chatroom 等）；属于专利所述"应用制定的业务类型"（从属权 4 形态）；**未检索到任何"基于数据流特征变量自动识别业务类型"的描述** | **反向证据**（上层 hint，非数据流自动识别） |
| **F2** | 冗余包数量 = f(网络状态变量, 传输成功率, **业务类型**) 三元组联合计算 | nxrte 综述明示："FEC 冗余度和原始包分组长度计算准则是：根据**丢包和 rtt** 预置分组长度，在该分组长度下，根据**贝叶斯定律**，在当前**丢包率**下，确定至少需要多少冗余包"——**业务类型未作为 F2 三元组之一**；GCC 算法基于反馈估计带宽再分配到 FEC/RED/RTX/PADDING 码率，输入仍是网络状态而非业务类型 | **不命中**（缺业务类型这一输入；权 1 F2 把业务类型硬钉为联合计算因子） |
| **F3** | 冗余包**总传输时间**由**时延要求**决定 | 综述"原始包与冗余包的最大序号差，决定了最大恢复延迟"；时延是 FEC 设计的**约束目标**（恢复延时 = Block 个数 × 帧长）而非"由时延要求显式换算得到一个传输窗口长度"；NERTC 文档未提供按时延要求换算 FEC 传输总时间的 API/算法描述 | **弱命中 / 不能确认**（有时延约束意识，但缺"由时延要求 → 总时间"的显式映射） |
| **F4** | 调度方法（随机度 / 最短时间 / 最长时间 / 均匀时间 ≥1 种），由网络状态 + 总时间 + 冗余包数量三元组联合选择 | RED 打包策略仅两类：连续相邻序号（低延迟）/ 跳跃序号（更大平均延迟）；属于打包顺序选择而非"调度方法集合"中的随机/最短/最长/均匀类型；GCC 平滑发送 (Pacer) 是固定速率平滑，未提到动态在多种调度算法间选择 | **不命中**（无符合权 2/9 限定的调度集合） |
| **F5** | 按调度方法**发送冗余包**（主动 FEC，非 ARQ/重传） | NERTC 在通信场景下"更多用 FEC，重传作为辅助"——确认会主动发送 FEC 冗余包；属于 proactive 冗余，符合 F5 主动性要求 | **命中**（孤立看是 F5，但无 F4 调度方法支撑） |

**权 1 整体判定**：F1 反向证据 + F2 缺业务类型输入 + F4 缺调度集合 + F3 仅弱命中 → **三个关键环节落空**。即便 F5 孤立命中，亦无法支撑权 1 整体侵权。

## 最终判定 **第 5 档：可排除**

**判定理由**：
1. **F1 有反向证据**：NERTC 业务/场景由上层 App 通过 `setChannelProfile` 显式指定，对应专利从属权 4（"应用制定的业务类型"），不落入权 1 F1 限定的"由发送端基于数据流特征变量自动获取业务类型"。这是 SKILL R-F1-hint-vs-auto 的典型反向证据。
2. **F2 缺业务类型输入**：网易公开技术资料一致显示冗余包数量由"丢包率 + RTT"经贝叶斯定律计算，业务类型未参与该计算。权 1 F2 把"业务类型"硬钉为冗余包数量的联合计算因子，因此缺业务类型即不命中权 1 F2（即便整体专利说明书有更宽抽象的从属权）。
3. **F4 无符合限定的调度集合**：权 2/9 限定调度方法属于随机度/最短/最长/均匀时间四类至少之一；NERTC 的 RED 打包仅有连续/跳跃二选一，Pacer 是固定速率平滑——均不属于权利要求所限定的调度集合。
4. **场景差异化不等于权 1 命中**：即便 NERTC 在通信/直播场景下 FEC/重传配比不同（"通信场景更多用 FEC"），这是"上层指定场景 → SDK 切换预置参数表"的实现路径，对应从属权 4 的"应用制定的业务类型"，仍不命中权 1 的 F1。

**降第 5 档"可排除"硬条件核对**：
- (a) 有真反向证据 — **是**：F1 上层 hint 是直接反向证据（与权 1 F1 自动识别要求互斥）
- (b) 不仅是 0 命中 — **是**：F2/F4 是基于网易公开技术细节的实质性不匹配，而非简单 0 命中
- (c) 时间窗不合规 — 否
- (d) 领域无关 — 否（领域完全对口）
- 需 ≥1 条 (a)/(b)/(c)/(d)，本案 (a)+(b) 均满足 → 落第 5 档充分

## 升级路径（3-4 档时）— 本案落第 5 档，列条件性升级触发器供后续复核

若出现以下新证据，应将本候选从第 5 档**升级**：
- **升到第 4 档（疑似侵权 - 弱命中）**：检索到 NERTC 源码或专利反向工程证据，显示其 SDK 内部存在"基于数据包到达间隔 / 包长统计 / 突发性 → 业务类型分类器"模块（即真有 F1 的数据流自动识别），且该分类结果被用于冗余包数量计算（F2 命中）
- **升到第 3 档（疑似侵权 - 强命中）**：除上述 F1+F2 命中外，再有公开材料明示冗余包传输总时间由时延 SLA 显式换算（F3 命中）+ 多种调度算法动态选择（F4 命中）
- **升到第 2 档或以上**：需在网易已授权专利 / 公开论文 / 商业产品文档中找到与权 1 五个步骤几乎一一对应的技术描述（目前未见）

## 总结一句话
网易云信 NERTC 通过上层 `setChannelProfile` 由 App 显式指定业务场景，FEC 冗余包数量由"丢包率+RTT"贝叶斯计算（不含业务类型）且调度方法不属于权利要求所列集合，**与权 1 F1/F2/F4 关键限定均存在结构性差异，落第 5 档：可排除**。
