## 最终判定
**第 4 档：资料不足（弱）**

判定依据（1-3句）：公开技术博客确证声网在弱网下用自适应 FEC + 多路冗余 + ARQ 抗丢包，且冗余/FEC 策略"按网络链路评估状况 + QoS 级别"决定（F2 部分命中、F5 发送冗余包成立，均按等同），全部材料均晚于 2021-06-08、领域与架构层级一致。但权 1 的三处特有限定——F1"由数据流特征变量自动识别业务类型"、F3"时延要求→冗余传输总时间"中间量、F4"传输总时间+冗余数量→冗余包调度方法"三输入链——在公开资料中均未披露，逐字命中比例 <60%，且无任何反向证据，故落"资料不足（弱）"而非"已排除"（0 命中≠已排除，且无真反向/时间不合规/层级不同）。

## 升级路径（仅第3-4档填）
- 优先攻声网自有专利：在 Google Patents / Justia 以受让人 "Agora Lab, Inc." / "上海兆言" 检索 2021-06 后 FEC/冗余/抗丢包专利权利要求全文，最可能 verbatim 披露 F1 自动识别 / F3 总时间 / F4 调度三输入（本轮 6 条 search 未命中其名下同主题专利，需换 CPC=H04L1/1867/1887、关键词 "redundancy quantity / scheduling / service type" 再试）。
- 取声网 SDK 弱网模块源码 / API 文档（如 FEC 冗余比例配置、QoS profile 与 streamType 映射）核验是否按数据流特征自动选 profile（F1）。
- 抓声网 RTC 工程深度文（Dev for Dev 系列冗余/FEC 专题）核验冗余量是否显式含"传输成功率/到达率"第三输入及"时延→总时间→调度"链（F3/F4）。

## 总结一句话
声网 RTC SDK 确用自适应 FEC+多路冗余按"网络状态+QoS 级别"抗丢包（F2/F5 等同），但 F1 自动业务识别、F3 时延→总时间、F4 三输入调度链公开资料未披露、命中<60% 且无反向证据，落第 4 档（资料不足-弱）。
