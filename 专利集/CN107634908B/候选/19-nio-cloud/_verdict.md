# 19-nio-cloud verdict

## 候选基本信息（专利公开日 2021-06-08）

- 候选 NN: 19
- 类型: 产品
- 名称: NIO Cloud / OTA 通道
- 组织: 蔚来 NIO
- 初判命中 F#: F2, F3, F4, F5
- 公开度: 低（已确认极低）
- 一句话定位: 蔚来车云通道 / OTA 下发链路 — 业务上确属"车控指令 / 状态同步 / 升级包下发"场景，与专利场景 1（IoT 用户控制消息下发到汽车）业务定位高度匹配。

## F# 命中表（F1-F5）

| F# | 专利特征 | 蔚来公开材料是否可证 | 结论 |
| --- | --- | --- | --- |
| F1 | 发送端按数据流特征变量（包长/数目、到达间隔、突发性等≥1 项）识别业务类型 | 无任何公开材料披露 NIO 是否在车端 / 云端按流量特征自动分类业务 | 不可证（无正向也无反向证据） |
| F2 | 按"网络状态 + 传输成功率 + 业务类型"三元组联合算冗余包数量 | 无 NIO 私有传输层算法公开 | 不可证 |
| F3 | 按时延要求算冗余包传输总时间 | 无 | 不可证 |
| F4 | 按"网络状态 + 总时间 + 冗余包数量"算调度方法（随机度 / 最短 / 最长 / 均匀≥1 种） | 无 | 不可证 |
| F5 | 按调度方法发送冗余包 | 无 NIO 公开承认其车云通道使用主动 packet-level 冗余 / FEC；TOX 公开材料仅声明"0 丢包阈值提升 109%"，但这是车内总线指标，与车云上行无关 | 不可证 |

补充说明：
- TOX 协议栈是蔚来自研的**车内跨域**通信协议（SkyOS 中间件层 / SOME/IP 替代品），定位是 ECU 与 ECU 之间、域控制器与域控制器之间的传输，**不是**车-云通道，套不上专利的"发送端 → 无线接入点 → 云平台"模型；即使其内部用了 FEC，也不在本专利权 1 的"发送端"主体范围内（专利场景 1 明确画的是 user→AP→IoT cloud→car 的链路）。
- NIO OTA 公开材料仅停留在用户功能层（"FOTA 让控制更智能"），未披露任何传输层是否使用业务类型自适应冗余的细节；常见 OTA 实践（HTTPS 下载 + 断点续传 + 双区回滚）属于**整包重传 / ARQ 范畴**，与本专利"主动 packet-level 冗余"是不同技术路线，但 NIO 是否仅采用前者也无法从公开材料证实。

## 最终判定 **第 4 档：公开资料不足（弱候选）**

> 主 agent 复核纠正：sub-agent 原判第 5 档；按 SKILL 协议第 5 档专属"已排除"（需 a/b/c 硬条件之一），本候选无任何反向证据，仅 0 命中，应落第 4 档而非第 5 档。

判定依据：
- 4 次 WebSearch + 2 次 WebFetch 均未获得任何关于"NIO 车云上行 / OTA 下行"在传输层是否实施 F1-F5 的描述（既无正向证据也无反向证据）。
- 仅有的接近材料 TOX 协议栈定位是**车内跨域总线**，不是专利权 1 描述的"发送端→无线接入点→云平台→车"链路，不能反向直接排除（因为车-云通道是另一套未公开的栈）。
- 0 命中明确**不等于"已排除"**——本协议规定无反向证据时必须落"公开资料不足"档（第 3 或 4 档），而非"已排除"档（第 5 档）。F# 命中率 0/5 < 60%，落弱候选第 4 档。
- 蔚来作为造车新势力对私有传输栈一贯封闭，缺乏公开论文 / 技术白皮书 / 开源 SDK，导致 F# 比对在公开域内无法收敛。

## 升级路径

（本档为"公开资料不足"档，给出复核线索而非升级路径）

若后续要把本候选向 3-4 档升级，需要的新增证据形态：

- NIO App / TSP 后端 API 抓包（合法且伦理允许时）显示流量呈现 FEC 编码特征（系统包标识 + 冗余包标识）；
- 蔚来在专利数据库内的相关同主题反向专利或继受授权公告。

## 总结一句话

NIO Cloud / OTA 通道在公开域 4×WebSearch + 2×WebFetch 后均未获得任何 F1-F5 正向或反向证据，蔚来对私有车云传输栈极度封闭，落第 4 档：公开资料不足（弱候选）；后续仅可通过非公开渠道（论文 / 招聘 JD / 合规抓包）进一步取证。
