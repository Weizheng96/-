## 反向证据 vs 限定作用域语 区分

按 SKILL 严格剪枝条款：
- **反向证据**：DigiDow 2021 逆向工程论文明示"FEC / 冗余 / 包恢复"**不在其分析范围**，且论文专门研究 OcuSync 协议；OcuSync 公开材料对"自适应"的描述聚焦在**物理层信道选择**（20/10/1.4 MHz 带宽 + 2.4/5.8 GHz 频段 + 跳频），不涉及 packet-level 冗余 — 这构成"架构层级不同"的反向证据
- **限定作用域语**：DJI 官方未公开"OcuSync 不使用 FEC"这类明确否定声明 → 因此不能上升为强反向证据
- **0 命中 ≠ 已排除**：本候选不是"0 命中后早剪枝"，而是"有 1 个间接相关的 multi-service 多业务通道架构事实，但该事实属于物理层 / MAC 层固定通道分配，非权 1 所要求的数据流特征变量驱动的业务类型识别 + packet-level FEC 调度"

## 最终判定 **第 5 档：架构层级不同 / 信息不足**

判定理由：
1. OcuSync 公开的所有"自适应"均在物理层 / MAC 层（OFDM 带宽 / 频段 / 跳频），与权 1 要求的**传输层 / 应用层 packet-level FEC 调度**抽象层级不同
2. DJI 闭源，权利要求 F1-F5 所需的实现细节（业务类型推断算法 / 冗余包数量计算公式 / 调度时刻规则）**在公开材料中均缺失**
3. multi-service 多业务通道架构（control / video / telemetry 分通道）≠ F1 要求的"由数据流特征变量动态识别业务类型" — 前者是预先静态分配，后者是发送端动态推断

## 升级路径（如未来证据补足，则可重新评估）

若日后出现下列任一公开证据，可将本候选从第 5 档升至第 3-4 档：
1. **DJI 官方白皮书 / 专利 / 标准提案** 披露 OcuSync 在传输层 / 应用层做 packet-level FEC，且冗余决策输入包含"业务类型 + 网络状态 + 时延要求"
2. **独立逆向工程报告**（在 DigiDow 2021 基础上的后续工作）抓取到 OcuSync 控制 / 视频流的 packet-level 冗余字段，并证明冗余数量随业务类型 + 丢包率动态变化
3. **DJI 公开源码 / SDK** 中出现按 F1-F5 模式调度冗余包的代码路径

## 总结一句话

OcuSync 公开材料均描述其物理层 / MAC 层自适应机制（OFDM 带宽切换 + 双频段跳频 + 多通道），未见传输层 / 应用层基于业务类型 + 时延 + 网络状态的 packet-level FEC 调度证据，与权 1 抽象层级不同，**落第 5 档：架构层级不同 / 信息不足**。
