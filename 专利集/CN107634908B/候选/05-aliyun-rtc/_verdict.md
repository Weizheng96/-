## 最终判定

**第 4 档：弱候选**

判定依据：
1. **F1 存在反向证据**——`setAudioProfile` 文档明示场景类型由开发者**入会前手动配置、入会后冻结**，符合权 1 中"应用制定的业务类型"的从属/反向形态（特征文档明确判定为"不命中权 1"）；公开材料中未见 SDK 基于流量统计变量自动识别业务类型的机制。
2. **F2 因 F1 缺失而三元组不全**——动态冗余度只用"丢包率 + 帧类型"，权 1 限定的"业务类型"维度缺失。
3. **F3 / F4 公开材料未能证实**——内部实现是否存在该逻辑无法从公开文档判断。
4. F5 单点命中、F2 部分命中，整体不构成权 1 的等同侵权链路；但 ARTC 规模巨大、内部实现细节未公开，**不能满足第 5 档硬条件**（缺少"全无相关功能"的直接反向证据 / 仅在公开 API 面观察到反向 hint）。

## 升级路径（3-4 档时）

若要升至**第 3 档（强候选）**或更高，须补：
- **SDK 协议抓包**：抓取 ARTC SDK 的 FEC 控制流，确认 SDK 是否根据"流量统计特征（包长度 / 数目 / 到达间隔 / 突发性）"自动识别业务类型，并将"业务类型"作为冗余包数量的输入；
- **阿里云在 H04L1/00 / FEC 方向的自有专利反查**：若存在相关自有专利，可能描述其内部实现；
- **第三方 reverse-engineering**：是否有 GitHub repo 反编译 ARTC SDK 并讨论其 FEC 控制逻辑；
- **F3 / F4 直接证据**：抓取"冗余包传输总时间"参数推导逻辑、调度方式（连发 / 均匀 / 随机）的代码或文档。

## 总结一句话

阿里云 ARTC 存在 FEC 冗余传输 + 基于丢包率的动态调整（命中 F5、部分命中 F2），但业务类型由开发者通过 `setAudioProfile` 入会前手动配置且入会后冻结——属权 1 F1 限定的反向形态，F2 三元组中"业务类型"维度缺失，F3 / F4 公开材料未证实，**落第 4 档（弱候选）**，仍待 SDK 抓包或反编译证据进一步确认或排除。
