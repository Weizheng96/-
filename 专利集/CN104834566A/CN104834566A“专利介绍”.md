## 一句话概括

宿主机内运行的虚拟交换机 vSwitch 通常拥有 K 个转发处理线程，本专利通过**实时监测**这些线程的状态属性（NUMA 亲和性 / 剩余转发资源 / 剩余资源占比 / 关键流量额度），在状态满足预设触发条件时**动态重分配**线程↔交换端口的服务关系，从而摒弃 CPU↔Port 静态绑定，提高资源利用率和对网络流量变化的自适应能力。

## 解决的技术问题

NFV / 云数据中心场景下，传统做法将 vSwitch 的转发线程（绑定到具体 CPU）与宿主机的交换端口做**静态绑定**：每个端口固定由某条线程服务，业务上线后再不改动。此方案在两种情形下表现差：

1. **流量不均衡**：被分配到某线程的若干端口的流量突增，该线程 CPU 满载丢包，而其他线程 CPU 闲置——总转发能力被瓶颈线程压低；
2. **NUMA 亲和性破坏**：当 vSwitch 跨 NUMA 节点运行时，端口对应的物理网卡 / 网卡队列与处理它的转发线程 CPU 不在同一 NUMA 节点，跨 socket 内存访问延迟显著增加；
3. **关键业务无法保证**：高优先级（如语音、控制面）流量没有专门的资源预留机制，与普通数据混合在同一线程内，重载时 SLA 失守。

## 核心技术方案

1. **状态属性可观测化**：每条转发线程持续上报四类状态指标——
   - **NUMA 亲和性**（线程 CPU 与端口对应物理网卡之间是否同一亲和性节点）
   - **剩余转发资源额度**（绝对量）
   - **剩余转发资源额度占比**（相对量）
   - **关键流量实际占用 vs 额定占用 的转发资源额度**（关键流量优先级 > 阈值）

2. **触发条件**：以下任一成立时启动调整——
   - 关键流量实际占用 vs 额定占用差异超出预设范围
   - CPU↔端口亲和性低于预设范围
   - K 个线程之间负载均衡度低于预设范围

3. **调整动作**：将至少 1 个转发处理线程所服务的交换端口重分配，使下列任一目标提升——
   - 关键流量差异减小（保 SLA）
   - CPU↔端口亲和性提高（NUMA 局部性）
   - K 个线程负载均衡度提高（消瓶颈）

4. **三种典型重分配算法**（从属权 4-8 / 12-16 给出）：
   - **均衡度型**（权 4/12）：基于剩余资源额度计算 K 个线程的均衡度，未达阈值则迁移端口
   - **资源额度阈值型**（权 5/13、6/14）：当线程 x1 剩余资源 ≤ 阈值且线程 x2 剩余资源 > 阈值时，将端口从 x1 迁到 x2（含双向交换 k1 / k2 端口的复式迁移以避免单向迁移导致 x2 反过来超载）
   - **NUMA 亲和性型**（权 7/15）：当线程 x3 的 CPU 与其端口 j4 不同 NUMA 节点时，将 j4 迁给与之同 NUMA 的线程 x4
   - **关键流量保证型**（权 8/16）：当线程 x1 服务的端口 j5 实际关键流量 < 额定时，把 x1 同时服务的另一端口 j6 迁出去，腾资源给 j5 的关键流量

## 关键权利要求解读

### 权 1（方法权）逐字限定

> 一种调整转发处理线程的交换端口的方法，其特征在于，包括：
> ①检测宿主机中的虚拟交换机 vSwitch 的 K 个转发处理线程的状态属性；
> ②在检测到所述 vSwitch 的 K 个转发处理线程的状态属性满足预设调整触发条件的情况下，
> ③调整所述 K 个转发处理线程中的至少 1 个转发处理线程所服务的交换端口，
> ④所述 K 为正整数。

字面要求：宿主机 + vSwitch + K 条转发线程 + 状态属性检测 + 触发条件 + 端口重分配 + K ≥ 1。

### 权 9（装置权）特有限定

权 9 在权 1 基础上增加"装置"形态——必须有显式的"检测单元"（独立模块抽象）+"调整单元"。装置权的命中需要装置内部结构而非仅观察行为；侵权主体一般是 vSwitch 软件 / 装置制造商。

### 权 17（计算节点权）特有限定

权 17 比权 1 多出 2 项硬件结构限定：
- F17a：计算节点 = 硬件层 + 宿主机 + 至少一个虚拟机
- F17b：硬件层包含 K 个处理器；vSwitch 的 K 个转发处理线程与 K 个处理器**一一对应**

整机权的命中需要 OEM 出货的产品中**捆绑了**实施权 1 / 权 9 的 vSwitch 软件——卖空机框（不预装 vSwitch）的整机厂不构成整机权侵权。

### 从属权扩展（适用于所有独立权对应的从属链）

- 从属权 2 / 10：状态属性 ⊃ {亲和性 / 剩余资源 / 剩余资源占比 / 关键流量占用}（**至少一种**——这是关键 alternative）
- 从属权 3 / 11：触发条件 ⊃ {关键流量差异超阈值 / 亲和性低于范围 / 负载均衡度低于范围}（**至少一个**）
- 从属权 4 / 12：均衡度调整算法
- 从属权 5 / 13、6 / 14：资源额度阈值型迁移（含 k1 / k2 双向）
- 从属权 7 / 15：NUMA 亲和性调整
- 从属权 8 / 16：关键流量保证调整

### 等同特征讨论

- "K 个转发处理线程"vs"K 条 PMD 线程 / poll worker / 转发协程"—— 等同（都是"软件级别可调度的转发实体"）
- "状态属性"vs"telemetry 计数器 / 性能 metric / Prometheus exporter 输出"—— 等同（均为可观测的运行时指标）
- "调整线程所服务的交换端口"vs"重新映射 RSS Indirection Table / 重新分配 NIC 硬件队列 / 在线程间重 pin 软件 RX queue"—— 等同（均改变了 thread↔port 的服务关系）
- "vSwitch"vs"SmartNIC / DPU 内卸载的 fastpath 转发引擎"—— 部分等同（需个案判断；当 fastpath 仍由宿主机控制面调度时落入；当完全脱离宿主机控制面时存疑）

### 明显不在保护范围内的相邻方案（反向证据形态）

下述形态后续 Step 5/6 直接判"已排除"：

1. **完全静态绑定的实现**：vSwitch 启动时一次性把 thread↔port 映射写死，运行期不再调整 → 不命中 F4 / F5
2. **物理交换机的端口 / 队列调度**：本专利明确限定"宿主机中的虚拟交换机"——硬件交换机内部的端口调度不命中 F1
3. **VM / 容器内部应用层负载均衡**：VM 内的应用 LB / 服务网格 / Sidecar 重分配不在 vSwitch 转发处理层 → 不命中 F1 / F2
4. **物理网卡硬件级 RSS 默认配置**：NIC 硬件 hash 默认散列**不构成 F4 触发条件 + F5 调整动作** ；仅当软件主动检测线程状态后**主动重写** RSS 表时才命中
5. **kernel 内置 IRQ balancing / cgroup 通用 CPU 调度**：通用 CPU 调度器调整线程 CPU pin 不命中 F5（F5 要求"调整线程所服务的交换端口"，而非"调整线程绑定的 CPU"）
