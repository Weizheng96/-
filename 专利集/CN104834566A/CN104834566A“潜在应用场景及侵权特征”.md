# CN104834566A 潜在应用场景及侵权特征

## 基本信息
- 专利号：CN104834566A
- 标题：调整转发处理线程的交换端口的方法和相关装置
- 当前权利人：Huawei Technologies Co Ltd（华为技术有限公司）
- 申请日 / 公开（授权）日：2015-03-31 / 2015-08-12

## 一、潜在应用场景（严格来自背景技术 — 不外扩）

### 场景 1：NFV 运营商网络中的虚拟交换机多核转发
- **背景技术 verbatim 依据**：> "网络功能虚拟化(NFV，Network Function Virtualization)技术可使得移动运营商在如何建立自己网络和实施新服务等方面发生翻天覆地的变化……原来运行的网络功能，例如防火墙、流量计费、路由服务等网元设备在NFV场景下会迁移到VM中运行……这对虚拟化场景中的虚拟交换机(vSwitch，Virtual Switch)的交换能力，丢包率和时延等性能规格提出了更高的要求。"
- **场景说明**：电信运营商 / NFV 平台把防火墙、计费、路由等网元虚拟化进 VM，VM 间及上行链路的大流量由宿主机上的 vSwitch 承载；vSwitch 用多核（多转发处理线程，每线程绑一个 CPU）来提升交换性能。
- **使用本专利的典型形式**：vSwitch 数据面（转发线程 / PMD）内置的"线程↔端口动态再分配 / 负载均衡"机制——以内嵌算法形式存在于虚拟交换机软件或 NFVI 转发平面中。

### 场景 2：数据中心 / 云平台的虚拟机间高吞吐流量交换
- **背景技术 verbatim 依据**：> "由于网络虚拟化的过程中，在NFV场景下会将网络吞吐量较大的服务转移运行在VM中，使得跨物理主机的VM和同物理主机的VM均需要较大的网络流量交换能力，这些网络流量交换能力主要由vSwitch提供。然而vSwitch负责网络流量交换，如果使用单个中央处理器(CPU)则可能会成为瓶颈，因此，业内考虑引入多核并行处理能力以提高vSwitch的网络流量交换性能。"
- **场景说明**：公有云 / 私有云 / 数据中心宿主机上，同主机与跨主机 VM（或容器）间的东西向大流量由 vSwitch 多核转发；端口（VM port、物理网卡 uplink port）需分配给不同 CPU/转发线程处理。
- **使用本专利的典型形式**：宿主机虚拟交换机 / 智能网卡 DPU 卸载的 vSwitch 数据面中，按转发线程实时负载 / 亲和性自适应迁移 rxq（端口队列）到不同 CPU 的算法。

### 场景 3：业务链（ServiceChain）边缘的多 VAS 串接转发
- **背景技术 verbatim 依据**：> "在NFV领域，有一种业务链(ServiceChain)机制，即在移动、固定宽带和数据中心网络边缘，特定用户的特定业务流可需要按顺序通过多个虚拟应用服务(VAS，Virtual Appliance Service)实体进行逐个处理。"
- **场景说明**：在网络边缘按用户套餐把业务流串接通过多个 VAS（内容过滤、缓存、防火墙等），这些 VAS 实体间的流量同样经 vSwitch 多核转发，存在按关键流量优先级调整端口↔线程分配的需求。
- **使用本专利的典型形式**：边缘 NFV / SD-WAN / CPE 平台中 vSwitch 转发面的关键流量保障 + 端口重分配机制。

> 说明：背景技术核心只锚定一类技术——**vSwitch 多核转发中转发线程与交换端口的（动态）分配**；上述三场景是该技术在 NFV / 数据中心 / 业务链三类落地形态，未外扩到背景技术未提及的领域。

## 二、独立权利要求 + 侵权特征 F1-Fk

本专利含 3 条独立权利要求：权 1（方法）、权 9（装置）、权 17（计算节点）。三者共享同一发明核心（检测转发线程状态 → 满足触发条件 → 调整线程所服务的交换端口），F1-F4 以权 1 为基准拆分；权 9 / 权 17 为同一核心的装置 / 系统形态。

### 权 1 verbatim
> 1.一种调整转发处理线程的交换端口的方法，其特征在于，包括：
> 检测宿主机中的虚拟交换机vSwitch的K个转发处理线程的状态属性；
> 在检测到所述vSwitch的K个转发处理线程的状态属性满足预设调整触发条件的情况下，调整所述K个转发处理线程中的至少1个转发处理线程所服务的交换端口，所述K为正整数。

### F1-F4 清单
- **F1（检测转发线程状态属性）**：> "检测宿主机中的虚拟交换机vSwitch的K个转发处理线程的状态属性"——宿主机上的 vSwitch 有 K 个转发处理线程（每线程对应一个 CPU），系统检测这 K 个线程的状态属性。
- **F2（判断是否满足预设调整触发条件）**：> "在检测到所述vSwitch的K个转发处理线程的状态属性满足预设调整触发条件的情况下"——把检测到的状态属性与一个预设的"调整触发条件"比较，作为是否触发重分配的判据。
- **F3（调整至少 1 个线程所服务的交换端口）**：> "调整所述K个转发处理线程中的至少1个转发处理线程所服务的交换端口"——触发后，把至少 1 个转发线程当前服务的交换端口重新分配（迁移端口↔线程映射，非业务运行前固定绑定）。
- **F4（K 为正整数，多线程多端口）**：> "所述K为正整数"——K ≥ 1 的整数限定；结合背景"多核并行 / 每端口交给某个 CPU 处理"，实质语境是 K 个线程对应 K 个 CPU、服务多个端口的多核形态。

### 其他独立权（仅当存在多条独立权时填）
- **权 9（装置）**：> "一种调整转发处理线程的交换端口的装置……包括：检测单元，用于检测宿主机中的虚拟交换机vSwitch的K个转发处理线程的状态属性；调整单元，用于在检测到……满足预设调整触发条件的情况下，调整……所服务的交换端口"——与权 1 同核心，落到"检测单元 + 调整单元"装置形态。
- **权 17（计算节点）**：> "一种计算节点……包括：硬件层、运行在上述硬件层之上的宿主机、以及运行在上述宿主机之上的至少一个虚拟机，其中，所述硬件层包括K个处理器，所述宿主机中的虚拟交换机vSwitch的K个转发处理线程与所述K个处理器之间一一对应；其中，所述宿主机用于，检测……K个转发处理线程的状态属性；在检测到……满足预设调整触发条件的情况下，调整……所服务的交换端口"——系统级形态，额外限定 **K 个转发处理线程与 K 个处理器一一对应**（线程↔CPU 一一映射）。

### 关键限定词与隐含约束
- **K 为正整数，且（权 17）K 个转发线程与 K 个 CPU 一一对应**——典型实施为多核 vSwitch，线程绑核（PMD per-core）。
- **"调整……所服务的交换端口"= 运行时动态再分配**，与背景所批判的"业务运行前固定绑定 CPU↔Port"形成对立——动态性是侵权判定关键。
- **状态属性的可选集合（权 2/10）**：处理器与端口的亲和性、剩余转发资源额度、剩余转发资源额度占比、关键流量实际占用的转发资源额度（"包括如下……中的至少一种"——开放/择一，非封闭枚举）。
- **调整触发条件的可选集合（权 3/11）**：关键流量额度差异超范围、亲和性低于预设、K 线程间负载均衡度低于预设（"至少一个"——择一即可命中）。
- **调整目标效果（权 11）**：使关键流量额度差异减小 / 亲和性提高 / 负载均衡度提高之一被满足。
- 命中 F1-F3 即落入独权保护范围；状态属性 / 触发条件的具体子类型属从权，命中其一即强化侵权信号。
