## 一句话概括

针对 I/O 虚拟化中"软件方案性能差、硬件 SR-IOV passthrough 兼容性差"的双向矛盾，提出**前后端 BE/FE 抽象 + SR-IOV VF + VF software instance 桥接**的混合架构：VM 内 FE 看到标准 virtio-like 接口（兼容性），数据路径直通 VF 硬件（性能），Host 内 VF software instance 通过 BE exporting API 把 FE 预分配的 DMA 缓存地址写入 VF 设备的 storage unit；VF 设备直接 DMA 写入这些 cache，写入完成后通知 VF software instance → BE → FE 收数据 — 这是 **vDPA（vhost Data Path Acceleration）架构原型**。

## 解决的技术问题
1. SR-IOV passthrough：性能好但 guest OS 与硬件直接耦合，热迁移困难、跨厂商兼容性差
2. 软件 virtio + vhost-net：兼容性好但性能损失大（CPU 负载高 + 延迟）
3. 二者矛盾：本专利通过 BE/FE + VF software 桥接抽象层，**同时**获得性能（DMA 直通）和兼容性（VM 看到 virtio 接口）

## 核心技术方案

1. **5 层架构**：硬件层 → Host → BE → VF software instance → VF device <-> VM 内 FE
2. **绑定关系**：BE ↔ idle VF software instance（创建阶段）；VF software instance ↔ VF device（一一对应）
3. **DMA workflow**：
   a. FE 预分配 DMA cache
   b. VF software instance 通过 BE exporting API 获取 cache 地址
   c. 写入 VF device 第一存储单元
   d. VF device 收到数据时从 storage unit 选取地址 → DMA 写入 cache
   e. VF device 通知 VF software instance → 触发 FE 接收数据

## 等同特征讨论

- "VF device" vs "PCI passthrough device with SR-IOV" — 等同
- "VF software instance" vs "vDPA bus driver / VFIO PCI driver" — 等同
- "BE" vs "vhost backend / vhost-vdpa kernel module" — 等同
- "FE" vs "virtio-net frontend in guest" — 等同
- "exporting API" vs "vDPA control path API" — 等同

## 明显不在保护范围内的相邻方案

1. **Pure SR-IOV passthrough**（VM 直接看到 VF 硬件接口，无 BE/FE 中间层）— 不命中 F4
2. **Pure software vhost-net + virtio**（无 VF device 直通）— 不命中 F2
3. **Para-virtual 驱动如 PV-NET 历史方案**（无 VF/SR-IOV）— 不命中 F2
4. **Container 网络（CNI / netns）**（非传统 hypervisor / VM 模型）— 不命中 F1
