## 一、潜在应用场景（严格来自背景技术 — 不外扩）

### 场景 1：云计算平台的虚拟机 I/O 虚拟化（高性能 + 兼容性兼顾）
- **背景技术 verbatim 依据**：
  > "The virtualization technology, as one of the important supporting technologies of the currently popular cloud computing (Cloud Computing) platform, can greatly improve the resource utilization efficiency of a physical device."
  > "In an input/output (I/O, Input/Output) virtualization field, the virtualization I/O solution having high performance and low delay is still a key technical challenge in the virtualization field."
  > "the conventional software solution is advantageous in compatibility, but has great performance loss; the hardware solution can bring about desired performance, but has problems such as feature compatibility and guest operating system (Guest OS) compatibility."
- **场景说明**：云/数据中心里运行大量虚拟机（VM）的计算节点，需要在保证虚拟机 I/O（网络、存储等）高性能、低时延的同时，保留软件虚拟化方案的设备兼容性 / Guest OS 兼容性（如热迁移能力）。本专利用"前端实例（FE，在 VM 内）+ 后端实例（BE，在 Host 内）+ BE 绑定一个空闲 VF 软件实例 + VF 设备直接 DMA 到/从 VM 预分配缓存"的混合架构同时拿到两者的优点。
- **使用本专利的典型形式**：宿主机虚拟化栈（Hypervisor / Host 内核模块 + 用户态后端）中实现的一套 I/O 虚拟化数据通路框架——即"控制面走半虚拟化前后端设备模型、数据面直通硬件 VF（SR-IOV）"的混合直通方案。背景技术仅明示"软件方案"与"硬件方案（SR-IOV 类）"两条既有路线及其取舍，本专利在两者间架桥，**不外扩到与 I/O 虚拟化无关的领域**。

> 背景技术只描述了"云计算/数据中心虚拟化 I/O"这一个核心场景（软件方案 vs 硬件方案的取舍），故仅列 1 个场景——宁少勿编。其内部可细分为网络 I/O 虚拟化与存储/块设备 I/O 虚拟化两条已商用产品线（均属同一抽象层，Step 3 据此枚举）。

## 二、独立权利要求 + 侵权特征 F1-Fk

本专利有 **2 条独立权利要求**：权 1（方法）与权 13（计算节点/装置）。权 13 是权 1 的"装置形态"镜像，特征集合与权 1 一一对应（仅把"由 X 执行某步骤"改写为"X 被配置为执行某步骤"）。因此 F1–F9 同时刻画权 1 与权 13；命中判定对两条独权通用。

### 权 1 verbatim
> 1. A virtualization processing method, applied to a computing node, wherein the computing node comprises: a hardware layer, a Host running on the hardware layer, and at least one virtual machine (VM) running on the Host, wherein the hardware layer comprises an input/output (I/O) device, several corresponding virtual function (VF) devices are virtualized from the I/O device, the Host has several VF software instances, the several VF software instances and the several VF devices are in one-to-one correspondence; the Host further has a back-end instance (BE) of an I/O virtual device having the same type with the I/O device, the VM has a front-end instance (FE) of the I/O virtual device, wherein the BE in the Host is bound with an idle VF software instance; the method comprising: pre-allocating, by the FE, a cache for direct memory access (DMA); acquiring, by the VF software instance bound with the BE, an address corresponding to the cache for DMA through an exporting application programming interface of the BE, writing the acquired address corresponding to the cache for DMA into a first storage unit of a VF device corresponding to the VF software instance; selecting, by the VF device, the address corresponding to the cache for DMA from the first storage unit when there is data to be received, and initiating a DMA write request by using the selected address as a target address; notifying, by the VF device, the VF software instance which is corresponding to the VF device and is in the Host after the DMA write request is executed, so that the VF software instance triggers the FE to receive the data written into the cache corresponding to the address.

### 权 13 verbatim（装置形态独权，仅摘前提与差异）
> 13. A computing node, comprising: a hardware layer, a Host running on the hardware layer, and at least one virtual machine (VM) running on the Host, wherein the hardware layer comprises an input/output (I/O) device several corresponding virtual function (VF) devices are virtualized from the I/O device, the Host has several VF software instances ... the Host further has a back-end instance (BE) of an I/O virtual device having the same type with the I/O device the VM has a front-end instance (FE) of the I/O virtual device, wherein the BE in the Host is bound with an idle VF software instance, wherein, the FE is configured to pre-allocate a cache for direct memory access (DMA); the VF software instance bound with the BE is configured to acquire an address ... through an exporting application programming interface of the BE ... the VF device is configured to select the address ... initiate a DMA write request ... notify the VF software instance ... so that the VF software instance triggers the FE to receive data ...

### F1-F9 清单

**架构前提类（F1–F5，刻画"混合直通"架构 — 本专利核心结构特征）**

- **F1（VF 设备从 I/O 设备虚拟化而来）**：verbatim「the hardware layer comprises an input/output (I/O) device, several corresponding virtual function (VF) devices are virtualized from the I/O device」——即 SR-IOV 式从物理 I/O 设备虚拟出多个 VF。
- **F2（VF 软件实例与 VF 设备一一对应）**：verbatim「the Host has several VF software instances, the several VF software instances and the several VF devices are in one-to-one correspondence」——Host 侧每个 VF 设备有一个对应的 VF 软件实例（VF 驱动/管理实体）。
- **F3（Host 有同类型 I/O 虚拟设备的后端实例 BE）**：verbatim「the Host further has a back-end instance (BE) of an I/O virtual device having the same type with the I/O device」——半虚拟化后端，且其设备类型与底层 I/O 设备同类。
- **F4（VM 有该 I/O 虚拟设备的前端实例 FE）**：verbatim「the VM has a front-end instance (FE) of the I/O virtual device」——Guest 内的半虚拟化前端驱动。
- **F5（BE 绑定一个空闲 VF 软件实例）★关键限定★**：verbatim「the BE in the Host is bound with an idle VF software instance」——后端实例绑定到一个**空闲（idle）**的 VF 软件实例。这是把"控制面（FE/BE 半虚拟化）"与"数据面（VF 直通）"桥接起来的核心动作，也是与纯软件 / 纯 SR-IOV 直通方案的最关键区别点。

**接收数据通路方法步骤类（F6–F9）**

- **F6（FE 预分配 DMA 缓存）**：verbatim「pre-allocating, by the FE, a cache for direct memory access (DMA)」——由 Guest 内前端预先分配用于 DMA 的缓存。
- **F7（绑定 BE 的 VF 软件实例经 BE 导出 API 取地址并写入 VF 设备 first storage unit）**：verbatim「acquiring, by the VF software instance bound with the BE, an address corresponding to the cache for DMA through an exporting application programming interface of the BE, writing the acquired address corresponding to the cache for DMA into a first storage unit of a VF device corresponding to the VF software instance」——经 BE **导出的应用编程接口（exporting API）**取得 DMA 缓存地址，写入对应 VF 设备的第一存储单元（接收队列）。
- **F8（VF 设备收到数据时取地址发起 DMA 写请求）**：verbatim「selecting, by the VF device, the address corresponding to the cache for DMA from the first storage unit when there is data to be received, and initiating a DMA write request by using the selected address as a target address」——VF 设备硬件直接以该缓存地址为目标地址发起 DMA 写（数据面直通）。
- **F9（VF 设备通知 VF 软件实例 → 触发 FE 收数据）**：verbatim「notifying, by the VF device, the VF software instance which is corresponding to the VF device and is in the Host after the DMA write request is executed, so that the VF software instance triggers the FE to receive the data written into the cache corresponding to the address」——DMA 写完成后，VF 设备通知 Host 内对应 VF 软件实例，由其触发 FE 接收数据。

> 说明：权 1/权 13 还隐含一条对称的"发送通路"（见从属权 6/9/16/18：FE 写待发数据 GPA → BE 取 → 写 VF 设备 second storage unit（发送队列）→ VF 发起 DMA 读）。但独权 1/13 字面仅完整描述**接收**通路，发送通路出现在从属权，故 F6–F9 锚定接收通路；判定命中以独权为准。

### 关键限定词与隐含约束
- **"idle"（空闲）VF 软件实例**：F5 要求 BE 绑定的是一个空闲 VF 软件实例，而非任意 VF 软件实例。
- **"one-to-one correspondence"（一一对应）**：VF 软件实例数 = VF 设备数，严格一一对应（F2）。
- **"having the same type with the I/O device"（同类型）**：BE/FE 所属的 I/O 虚拟设备类型须与底层物理 I/O 设备同类（F3），保证半虚拟化前后端与直通硬件是"同一种设备"。
- **"exporting application programming interface of the BE"（BE 的导出 API）**：VF 软件实例通过 BE 对外导出的 API 取 DMA 地址（F7），而非直接读 Guest 内存——这是 FE/BE/VF 三者解耦的接口约束。
- **first / second storage unit = 接收队列 / 发送队列**（从属权 12 明示：first storage unit is a receiving queue；second storage unit is a sending queue）。
- **DMA 目标地址可为 GPA 或 HPA**：从属权 2–11 给出两种地址翻译路径——(a) IOMMU 在硬件侧把 GPA→HPA（权 2/3/6/8），或 (b) Host 软件侧先把 GPA→HPA 再下发（权 4/5/9/11）。独权 1/13 不限定，命中独权不要求一定有 IOMMU。
- **半虚拟化前后端 + 硬件 VF 直通的"混合"性质**：整体架构 = 控制/兼容性走半虚拟化设备模型（FE/BE），数据面走 VF 硬件 DMA 直通——判定候选是否命中时，须同时看到这两层都存在且通过"BE 绑定空闲 VF 软件实例"桥接，而非仅有 SR-IOV 直通或仅有纯 virtio 软件后端。

**Step 2 自检**：① F# 是否覆盖权 1 所有并列动作？——架构前提 5 项（F1–F5）+ 接收通路 4 步（F6–F9）已覆盖权 1 字面全部限定。② 整数限定 / 多层结构 / 时间窗有没有漏？——"idle""一一对应""同类型""导出 API""接收/发送队列"均已列入关键限定；本专利无整数下界（N≥k）类限定。③ 第二条独权（权 13 装置形态）已标注，与权 1 共用 F1–F9。
