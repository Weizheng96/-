## 一句话概括

针对行车数据安全（防篡改、防删除、可独立验证）问题，提出"中心节点 + 多个辅助节点"双层架构：中心节点用 Merkle 树 + TEE + 随机私钥 + 时间戳生成签名数据包头，链式哈希连接，向多个辅助节点（车厂云、车主、监管）同步；任一辅助节点可独立验证完整性 — 类似车联网场景下的"轻量化区块链"或"可审计 immutable log"。

## 解决的技术问题
1. 行车数据被恶意篡改 / 删除：现有车联网系统多为单点存储，无法事后审计
2. 单点存储信任问题：仅由车厂存储数据 → 监管 / 车主无法独立验证
3. 高带宽问题：完整数据上传到所有节点带宽开销大 → 仅同步 hash 头 + 部分数据

## 核心技术方案
1. **多节点架构**：中心节点（车端）+ 多辅助节点（车厂云、车主手机、监管平台）
2. **Merkle 树根哈希**：将一段时间内行车数据生成 Merkle root hash
3. **TEE 内生成**：根哈希 + 时间戳 + 前一包头哈希 → 数据包头；TEE 中执行（防 host 篡改）
4. **随机私钥签名**：包头随机生成私钥签名
5. **链式结构**：前后包头哈希相连，可发现数据删除
6. **灵活同步**：发送包头 + 部分数据（其余 hash） → 节点可按需验证

## 等同特征讨论

- "Merkle 树"vs"区块链 chain of blocks"——等同
- "随机私钥"vs"DAA / EPID 群签名 / 一次性密钥"——等同
- "TEE 中生成"vs"HSM / 安全 enclave / SGX / TrustZone"——等同
- "辅助节点"vs"区块链验证节点 / 监管节点 / 车主端"——等同

## 明显不在保护范围内的相邻方案

- 单一中心存储（车厂自存）：不命中 F1（无多节点）
- 普通签名 + 单数据 hash（无 Merkle 树）：不命中 F3
- 静态私钥签名（无随机私钥）：不命中 F5
- 不同 OBD2 / V2X 通用协议中的认证（无车端中心节点 + Merkle 链结构）：不命中 F1/F3
