## 一、潜在应用场景（严格来自背景技术 — 不外扩）

### 场景 1：车辆行车数据防篡改存证（用于事故责任认定 / 监管追溯）

- **背景技术 verbatim 依据**：
  > [0002] "随着技术的发展，现在越来越多的车辆开始对于自身的行车数据进行存储，用于在出现行车事故时作为可提供证据的数据进行使用。这样就需要保证车辆的行车数据的正确，以及未篡改。"
  > [0003] "目前通常是通过车辆的行车数据记录系统以及车厂云端数据库来实现对行车数据的存储。而车辆自身的行车数据记录系统的存储空间有限，为保证后续行车数据的存储，会对行车数据的保存时间进行限定。而存储在车厂云端数据库的行车数据，理论上车厂可以任意修改行车数据，从而掩盖车辆的技术缺陷，或在事故发生后推卸责任。"
  > [0004] "因此现在急需要一种可以防止行车数据篡改的方法。"

- **场景说明**：车辆（智能网联汽车 / 自动驾驶车 / 商用车 / 网约车 等）在行驶过程中持续采集传感器数据（速度、转向、刹车、加速度、雷达 / 摄像头融合输出、ADAS 决策日志 等），需要在事故 / 纠纷 / 监管检查 / 保险理赔 等场景下提供**未被篡改且时间真实**的原始数据作为证据。背景技术明示当前痛点是 **车厂云端可单方修改**（数据归属与可信第三方分离问题）。

- **使用本专利的典型形式**：
  - 车端 SDK / TEE 内组件：在车载主控 SoC 的可信执行环境（TEE / TrustZone / SGX 等）内对传感器数据做默克尔树根哈希 + 时间戳 + 私钥签名，生成数据包头
  - 车厂云数据库或独立云存证服务：作为辅助节点同步数据包头并对外公布
  - 第三方监管 / 保险公司 / 车主自留备份：作为辅助节点接收数据包头，事后做交叉验证
  - 注：背景技术明确点名"车厂云端数据库、车主自用的存储设备、车主对应的验证设备或者监管平台对应的数据库"四类辅助节点（见说明书 [0007]）

> **注**：背景技术（[0002]-[0004]）仅描述了"行车数据用于事故证据"这一个场景。说明书更详细段落（如 [0007]）虽然列举了 4 类辅助节点角色，但这些是**同一场景的实现细节**，不构成独立场景。**本 Step 严格只展开背景技术明示场景，不向"区块链通用存证"、"工业数据存证"、"医疗数据存证"等相邻领域外扩**。

## 二、独立权利要求 + 侵权特征 F1-Fk

### 权 1 verbatim
> 1.一种防止行车数据篡改的方法，应用于行车数据验证系统，所述系统包括中心节点和多个辅助节点，其特征在于，包括：
> 所述中心节点获取车辆在预设时间段内行车数据；
> 所述中心节点根据行车数据计算得到根哈希值；
> 所述中心节点根据所述根哈希值和时间戳生成数据包头，并利用私钥对所述数据包头进行签名，所述时间戳为系统时间，所述私钥为随机生成；
> 所述中心节点向所述辅助节点发送所述数据包头。

### F1-Fk 清单（从权 1 字面拆解）

- **F1（系统架构 — 中心 + 多辅助节点）**：> "应用于行车数据验证系统，所述系统包括中心节点和多个辅助节点"
  - **关键限定**：辅助节点必须是**多个**（N ≥ 2）。仅 1 个辅助节点的实现 → 不字面命中。
- **F2（车辆行车数据采集）**：> "所述中心节点获取车辆在预设时间段内行车数据"
  - **关键限定**：必须是**车辆**的**行车数据**（车速、转向、刹车、加速度、传感器原始流等）；且按**预设时间段**分段采集（窗口式批处理，而非单事件单包）。
- **F3（根哈希计算）**：> "所述中心节点根据行车数据计算得到根哈希值"
  - **隐含限定**（结合从属权 3、说明书 [0010]）：典型实现为"多传感器数组 → 各自哈希 → **默克尔树**两两结合 → 上层哈希 → 根哈希"。说明书称该结构为"默克尔树计算逻辑"。
- **F4（包头由根哈希 + 时间戳构造）**：> "所述中心节点根据所述根哈希值和时间戳生成数据包头"
  - **关键限定**：数据包头**必须同时包含**根哈希值 **和** 时间戳两个要素，缺一不命中。
- **F5（私钥签名 + 时间戳系统时间 + 私钥随机生成）**：> "并利用私钥对所述数据包头进行签名，所述时间戳为系统时间，所述私钥为随机生成"
  - **关键限定 (5a)**：必须用**私钥**对**数据包头**做**签名**（非对称密码 / 数字签名，不是 HMAC / MAC 这类对称机制）
  - **关键限定 (5b)**：时间戳必须是**系统时间**（设备本地时间或 TEE 内可信时间），不是网络时间 / 外部时间源协议化时间
  - **关键限定 (5c)**：私钥必须**随机生成**（每次或按会话生成，不是出厂烧写的固定密钥）
- **F6（中心向辅助节点同步包头）**：> "所述中心节点向所述辅助节点发送所述数据包头"
  - **关键限定**：必须存在**主动向辅助节点同步包头**的动作（不只是车端内部留存）

### 其他独立权（基于权 1 的同族变体；用于扩大命中范围）

- **权 7**（中心节点装置 verbatim 摘录）：> "一种中心节点... 获取模块，用于获取车辆在预设时间段内行车数据；处理模块，用于根据行车数据计算得到根哈希值；根据所述根哈希值和时间戳生成数据包头，并利用私钥对所述数据包头进行签名，所述时间戳为系统时间，所述私钥随机生成；发送模块，用于向辅助节点发送所述数据包头。"
  - 权 7 是权 1 的**装置形式**。判定 F1-F6 命中的实物形态是中心节点设备 / 芯片 / SoC / 模组 时，套用权 7。

- **权 13**（辅助节点方法 verbatim 摘录）：> "一种防止行车数据篡改的方法... 所述辅助节点同步所述中心节点发送的数据包头... 所述辅助节点利用所述数据包头验证所述行车数据的是否篡改。"
  - 权 13 是**辅助节点视角**的方法。判定主体是云存证服务 / 监管平台 / 第三方验证方时，套用权 13（核心动作：同步包头 + 利用包头验证篡改）。

- **权 16**（辅助节点装置）、**权 19**（系统）、**权 22**（计算机程序）、**权 23**（计算机可读存储介质）、**权 24**（智能汽车）—— 均为权 1 / 权 13 的不同保护客体形式，命中要素同 F1-F6（或权 13 的辅助节点版本）。

### 关键限定词与隐含约束（Step 6 sub-agent 必须严守）

| 约束 | 内容 | 反向证据信号举例 |
|---|---|---|
| **N ≥ 2 辅助节点** | 系统必须含 ≥ 2 个辅助节点（"多个"硬约束） | vendor 文档只描述"车端 + 单云端"双节点架构 → 反向证据信号 |
| **数据为"行车数据"** | 必须是**车辆**采集的**行驶**相关数据；非车辆数据（如手机日志、IoT 工业数据、医疗数据）不在保护范围 | 候选实现是通用区块链存证 SDK，未与车辆传感器深度耦合 → 反向证据 / 架构层级不同 |
| **窗口式批处理** | 按**预设时间段**采集（而非单事件单包） | 候选实现是单条事件即时签名（无窗口聚合）→ 不命中 F2 |
| **默克尔树（隐含）** | 根哈希需具备 Merkle 结构（说明书明示） | 候选用扁平哈希链 / 单根 SHA-256（无树结构）→ 不命中 F3 |
| **时间戳 = 系统时间** | 设备本地 / TEE 内可信时间，非 NTP / RFC 3161 外部时间戳协议 | 候选用 RFC 3161 TSA 第三方时间戳 → 不命中 F5（5b）|
| **私钥随机生成** | 运行时 / 会话级随机生成，非出厂固定密钥 | 候选用 TPM 固定 EK / 厂商烧录的车辆 PKI 长期证书 → 反向证据 / 不命中 F5（5c）|
| **私钥签名 ≠ HMAC** | F5 限定为非对称数字签名 | 候选只做 HMAC-SHA256 / 对称 MAC → 不命中 F5 |
| **主动向辅助节点同步包头** | 必须有"中心节点 → 辅助节点"的包头同步链路 | 候选只在车端本地存档、不向外同步 → 不命中 F6 |
| **公开（授权）日时间窗** | 候选产品 / 论文 / 实现的**公开日期**必须 ≥ 2023.01.17 才构成侵权证据；早于此为现有技术（不计） | 候选首次公开 < 2023.01.17 → 时间窗不合规，落第 5 档 |
