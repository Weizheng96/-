## 时间窗判定

专利公开日 2021-06-08。GME 官方 CSDN 博客（103857430）发表于 2020-01，SegmentFault 同期同步；GME 产品本身 2018 年已商用。**核心 FEC 4+4 + 房间类型映射架构在专利公开前已公开**，构成 prior-art 性质的时间不合规反向证据。

## 最终判定

**第 5 档：已排除**

**硬条件（c）反向证据 + 硬条件（b）时间不合规 同时触发**：

- **(c) 反向证据 — F1 不命中**：腾讯云官方 API 文档和 GME 技术博客明确：业务类型 / 场景类型由 SDK 调用方通过 `ITMG_ROOM_TYPE_*` 枚举**显式传入**，对应专利**从属权 4**"应用制定的业务类型"形态；而权 1 的 F1 限定为"发送端根据数据流特征变量（缓存长度/在网长度/到达间隔/数据突发性）自动获取业务类型"，明确**排除**应用层 hint 形态。F1 硬条件不满足，F2 三元组的"业务类型"输入因此也不真满足（业务类型仅作 ceiling profile，非运行时计算变量）。

- **(b) 时间不合规**：GME 4+4 上限 + 房间类型 → FEC/ARQ/UDT 策略映射的核心架构在 2020-01 已被官方公开技术文章描述，**早于专利公开日 2021-06-08**；属 prior-art 性质材料，无法构成专利公开后的实施侵权证据。

判定依据：权 1 F1 是核心限定（业务类型获取方式必须是"发送端自动从数据流特征统计"），GME 公开技术架构是"上层应用静态选房间类型→驱动预设策略"，二者技术路径根本不同；叠加时间窗早于公开日，无任何升级路径。

## 升级路径

不适用（第 5 档已排除）。

理论上若未来腾讯云公开新版 GME 引入"发送端按数据流特征自动判断业务类型"机制（如不再依赖房间类型枚举，转为按 RTP 流统计自动归类），且该机制公开于 2021-06-08 之后，可重新评估。当前公开材料范围内无此迹象。

## 总结一句话

GME 按"房间类型"（流畅/标准/高清）由 SDK 调用方显式选择 FEC+PLC / ARQ+PLC / UDT 策略，与权 1 F1 要求的"发送端根据数据流特征变量自动获取业务类型"路径不同，叠加核心架构 2020 年已公开早于专利 2021-06-08 公开日，**落第 5 档（已排除）**。
