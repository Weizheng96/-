## 最终判定 **第 5 档：已排除（反向证据）**

**判定依据（多重反向证据，0 命中 ≠ 已排除——以下为真实反向证据三角化）：**

1. **平台技术范式与权 1 不属同一传输层流派**：根云是典型 IIoT 接入平台（MQTT pub/sub + 1100+ 工业协议 + 设备数据采集 + 大数据分析 + 预测性维护），其传输层属"reactive ARQ / 应用层 QoS"范式；权 1 要求"proactive 主动冗余 + 业务类型自适应"范式。两者在传输层机理上属不同流派——这是反向证据本身，对应专利"潜在应用场景及侵权特征.md"中明示的 `"主动冗余" vs "重传/ARQ" 区别`，根云行为完整落在 ARQ 一侧。

2. **MQTT/Modbus/OPC UA 协议本身均无 packet-level 冗余生成能力**：MQTT QoS 1/2 的 PUBACK / PUBREC / PUBREL 是被动重传；Modbus RTU/TCP 是同步请求响应；OPC UA 走 TCP 可靠传输。**无一种支持"主动生成 n 个冗余副本并按时延要求调度发送"**。"支持 1100+ 工业协议"本身意味着平台是协议适配层 / 接入层，这一类协议的全集**结构上排除了** F2-F5 所述能力。这是反向证据，不是 0 命中。

3. **业务类型判定方式与权 1 限定相左**：IIoT 平台靠 topic / schema / 设备型号在应用层区分业务（=权 1 之父发明描述中"应用制定的业务类型"形态，对应专利的**从属权 4** 而非权 1）；权 1 硬钉了"由发送端从数据流特征变量自动得到"。这是 F1 的硬反向证据，等同于权 1 vs 从属权 4 的关键区别。

4. **场景簇与专利目标场景错位**：根云专攻"工程机械、混凝土、环保、铸造、农机、发电机组、纺织机械"等工业设备远程运维场景，核心数据流是"批量周期性 telemetry 上行"；专利权 1 + 实施例定锚的场景是"稀疏控制消息 + 严格时延要求 + 不可靠链路"（场景 1-6：车控下发、在线游戏、DC 间保活、机器人控制、VR 体感、视频会议）。两个场景簇都含"IoT / 工业"字眼但**核心数据流形态相反**——telemetry 上行高吞吐 vs 稀疏控制消息低时延。

5. **4 互补 query + 1 官方页 + 0 命中的反向三角化**：4 次 WebSearch（中文 + 英文 + 联合方品牌名 + FEC 显式 OR 词）+ 1 次官方产品页 WebFetch，全部来源在描述根云时**从未使用过**"FEC / 前向纠错 / 冗余包 / 主动冗余 / packet-level redundancy / 业务类型自适应冗余 / 时延驱动调度"任何一个语义；公开技术语料中"根云"与"FEC"互斥分布于不相交的 URL 簇。这是"覆盖性反向证据"，不是"漏检的 0 命中"。

**Step 5 初判 F2/F3/F4/F5 命中结论 推翻**：Step 5 将"工业互联网 + 远程运维"误对应到了权 1 的"IoT 控制消息"场景，但根云的"IoT"侧重批量设备 telemetry 上行（数据采集 + 分析），权 1 的"IoT"侧重稀疏控制消息下发（如车控指令），且根云的传输层范式与权 1 要求的 proactive FEC 完全不同。

## 总结一句话

根云平台属于 MQTT / Modbus / OPC UA 路线的 IIoT 接入与远程运维平台，公开技术材料未见任何主动 FEC / 业务类型自适应冗余 / 时延驱动调度的描述，与权 1 F1-F5 在传输层机理上属不同范式，**落第 5 档（已排除：多重反向证据）**。
