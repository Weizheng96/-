# 37-baijiayun-brtc verdict

## 候选基本信息（专利公开日 2021-06-08）

- 候选 NN: 37
- 类型: 产品（RTC PaaS / SDK）
- 名称: 百家云 BRTC（Baijiayun Real-time Communication）
- 组织: 百家云（Baijiayun Group Ltd，原 NASDAQ:RTC，已退市）
- 初判命中 F#: F1, F2, F3, F4, F5
- 公开度: 中（官方文档 + 多篇技术博客 + 英文产品页 + 招股 / 股票资料）
- 主要场景: 教育（一对一/小班/大班/双师课堂）、互动直播、视频会议、连麦
- 时间窗: 2024-Q3 仍在售（晚于专利公开日，合规）

## 检索粗筛（Phase 1 — 3 次 WebSearch）

- q1 `百家云 BRTC 教育 RTC FEC 抗丢包 冗余` → 命中信号：官方文档明示 "智能 FEC + 30% 丢包恢复"
- q2 `Baijiayun BRTC adaptive redundancy FEC education streaming` → 命中信号：英文官网明示 "redundant packet encoding algorithm" + 70%/80% 抗丢包率
- q3 `百家云 BRTC 业务类型 自适应 网络状态 调度 教育 直播` → 命中信号：网络自适应宣传语；**但业务类型→冗余量映射、调度算法选择均未公开披露**

→ 粗筛**通过**（有 F5 信号），进入 Phase 2 深抓。

## Phase 2 深抓（5 次 WebFetch）

详见同目录 _sources.md。要点：百家云 BRTC 公开材料**只到产品宣传层**——反复声明"智能 FEC / 码率动态调整 / 网络自适应 / 抗 30% 丢包 / 抗 800ms 抖动"，但**未披露**：
- 是否根据业务类型（音频/视频/控制/会议）区分冗余策略（F1 核心限定）
- 冗余包数量是否由"网络状态 + 成功率 + 业务类型"三元组联合计算（F2）
- 冗余包传输总时间是否由时延要求驱动（F3）
- 调度方法是否在随机/最短/最长/均匀时间间选择（F4）

## F# 命中表（F1-F5）

| F# | 限定要点 | 公开证据 | 判定 |
| --- | --- | --- | --- |
| F1（业务类型识别 — 由发送端从数据流特征变量 a/b/c/d 至少一项推得） | 区分音频/视频/会议/直播/游戏的业务类型，且**由发送端自动从流量统计推得** | 官方文档仅列举"业务场景"（一对一 / 小班 / 大班 / 双师 / 会议 / 连麦），属于**应用层 SDK 接入维度**而非"发送端从数据流特征变量自动识别业务类型"；亦未披露 (a)缓存包长度/数目 / (b)在途包 / (c)到达间隔 / (d)突发性 任一项作为输入 | **未确认命中**（弱信号） |
| F2（冗余包数量 = f(网络状态, 成功率, 业务类型)） | 三元组联合计算 | 仅说"智能 FEC"，未公开 n 的计算输入是否含业务类型；现行 WebRTC / RTC 主流仅按丢包率+码率算 FEC 冗余度，业务类型通常不在输入侧 | **未确认命中**（弱信号） |
| F3（冗余包传输总时间 = f(时延要求）） | 时延要求驱动的发送窗口 | 官网仅声明"端到端低延迟 68ms~250ms"产品指标，未披露 FEC 发送总时间窗口的算法逻辑 | **未确认命中** |
| F4（调度方法 = f(网络状态, 总时间, 冗余包数量)) | 在随机/最短/最长/均匀时间间选择 | 未披露任何调度算法细节 | **未确认命中** |
| F5（按调度方法发送冗余包） | 实际发送冗余包 | 多源明示 "redundant packet encoding"、"智能 FEC"、抗 30%/70%/80% 丢包 | **命中**（高置信） |

合计：**F5 命中 1 / 5；F1-F4 因缺乏公开技术细节均"未确认命中"，且 F1 出现弱反向信号**（"业务场景"维度更像 SDK 接入分类而非权 1 中"发送端从数据流特征变量自动识别业务类型"）。

## 最终判定 **第 3 档：技术信号一致但权 1 关键限定无公开证据**

依据：
1. F5 公开命中（明示主动冗余 / FEC）—— 与权 1"发送冗余数据包"动作一致；
2. F1-F4 公开材料**全部停留在产品宣传层**，无任何技术博客 / 论文 / 招股书 / 专利申请披露"业务类型识别+冗余量三元组+时延驱动总时间+四种调度选择"中的任意一项；
3. **未发现反向证据**——百家云并未声明"不做业务类型自适应"或"仅按丢包率计算 FEC 冗余度"；
4. 行业先验：RTC PaaS（声网/腾讯 TRTC/网易云信/即构）公开材料中关于 FEC 的描述通常都停留在同等抽象层级，要进一步判定 F1-F4 命中与否，需要技术博客 / 源码 / 专利申请 / 工程师演讲等更深层材料；本候选未发现这类材料。

故判定第 3 档 — **技术宣传信号一致，但权 1 关键限定（业务类型自适应 + 三元组联合计算 + 时延驱动 + 调度方法多选）无公开证据**，**0 命中 ≠ 已排除**。

## 升级路径

若需将本候选升至第 2 档（高度疑似）甚至第 1 档（公开实证），应补充以下证据之一：
1. 百家云技术博客 / 工程师演讲 / 知乎 Live 中明示"对不同业务（音频/视频/会议/直播）采用不同 FEC 冗余度策略"或"FEC 调度按随机化/均匀时间发送"；
2. 百家云专利申请（CNIPA）中权要表述与 CN107634908B 权 1 限定结构相同；
3. 百家云开源 SDK（GitHub）源码层证据：FEC 冗余度计算函数同时引用 traffic_type / network_state / success_rate；
4. 招股书（SEC F-1 / 20-F）或并购尽调文档中明示业务类型自适应冗余调度模块；
5. 百家云员工在 LiveVideoStackCon / 全球音视频开发者大会 / RTC.dev 等行业大会的演讲材料披露 FEC 调度算法细节。

若需将本候选降至第 4-5 档（基本排除 / 已排除），应补充以下反向证据之一：
1. 公开技术资料明示百家云 BRTC 的 FEC **仅按丢包率**线性映射冗余度，**不分业务类型**；
2. 公开调度文档明示百家云 BRTC FEC 冗余包**等间隔**或**随报文实时发出**，不在多种调度方法间选择；
3. 招股书 / 工程文档明示 FEC 仅用于音频 PLC、视频侧仅靠 NACK/ARQ（即不在"主动冗余"路径）。

## 总结一句话

百家云 BRTC 公开宣称"智能 FEC + 网络自适应 + 抗 30%~80% 丢包"，F5 公开命中且时间窗合规，但 F1-F4 关键限定（业务类型自适应 + 三元组联合计算 + 时延驱动 + 调度选择）在所有公开资料中均无技术细节，**落第 3 档（技术信号一致但权 1 关键限定无公开证据），0 命中 ≠ 已排除**。
