## 最终判定

**第 3 档：公开资料不足（强候选）**

判定依据：
1. **正向信号充足**：腾讯会议公开披露做了"动态 FEC + 网络状态自适应 + 接收端反馈 + 关键性分级 + 主动冗余（非纯 ARQ）"，与权 1 五个特征中的 F2 部分、F4 部分、F5 全部均在概念上一致。
2. **F1 的关键限定无法证实也无法证伪**：权 1 F1 严格要求"业务类型由发送端从下列数据流特征变量统计得到：包长度数目 / 在网包长度数目 / 到达间隔 / 数据突发性"中的至少一项。腾讯公开材料只描述 codec/帧级别的"内容关键性"静态分级，未公开是否做了 F1 要求的运行时流量统计推断。**这不是反向证据，仅是公开资料披露深度不够**——产品级闭源代码中是否实际执行 F1，无法仅凭公开信息判定。
3. **时间合规**：腾讯会议自 2019 年 12 月上线，主要技术演进文档（2018、2020-12-31、2023）均横跨专利申请日 2016-07-19 与授权日 2021-06-08 前后；2021-06-08 之后的产品形态（持续运营至今）落入侵权时间窗。

## 升级路径（第 3 档）：
1. **抓取腾讯会议官方技术白皮书**（特别是 source.meeting.qq.com 上后续更新版本、《腾讯会议技术架构》系列、SIGCOMM / NSDI / ACM MM 论文）确认 F1 的数据流特征统计是否实现。
2. **检索腾讯发明专利申请**（CN 专利搜索：申请人=腾讯科技 + IPC=H04L1/00 + 关键词"业务类型 冗余 数据流特征"），若腾讯自家专利覆盖了 F1 同类机制，可证 F1 字面命中；若腾讯专利刻意回避 F1（如改用 codec-level 静态分类），则可推断为规避设计。
3. **抓取腾讯会议 / TRTC SDK 实测**（弱网模拟、Wireshark 抓包），统计冗余包发送是否与会话内的包到达间隔/突发性相关，作为 F1 命中的直接证据。
4. **检索华为 vs 腾讯 现有诉讼 / 许可信息**——若已有许可协议或公开诉讼，可直接采信。

## 总结一句话

腾讯会议在弱网 FEC、关键性分级、动态冗余率等方面公开披露与权 1 多个特征概念吻合，但 F1 要求的"业务类型从数据流特征变量统计得到"的具体实现未见公开披露，落第 3 档（公开资料不足-强候选）。
