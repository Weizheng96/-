## F# 命中表（F1-F5）

| F# | 特征要点 | 候选侧证据 | 是否命中 |
|---|---|---|---|
| F1 | 发送端从数据流特征变量（缓存长度/数目、网中包长度/数目、到达间隔、突发性 ≥1 项）**自动识别**业务类型 | 公开资料显示 CTWing 接入侧的"业务类型"区分由设备 / 应用上层属性 + 物模型字段静态配置（非透传 / 透传 / 自定义），**非由发送端从真实流量统计自动获得** | 不命中 |
| F2 | 冗余包数量 = f（网络状态 + 传输成功率 + 业务类型） | 公开协议栈（T-Link/MQTT/MQTT-SN/LWM2M/HTTP/TCP）使用 MQTT QoS 0/1/2 与 TCP 重传保证可靠性，**不发送主动冗余包**，更未基于"业务类型"动态决定冗余包数量 | 不命中 |
| F3 | 冗余包传输总时间 = f（时延要求） | 未见基于"时延要求"动态决定多包发送窗口的描述；可靠性兜底仍是 ack-超时-重传 | 不命中 |
| F4 | 调度方法 = f（网络状态 + 总时间 + 冗余包数量），可选随机度/最短/最长/均匀 ≥1 种 | 公开文档无任何冗余包调度策略描述（随机度 / 等间隔 / 最短最长时间均未提及） | 不命中 |
| F5 | 按调度方法发送冗余数据包 | CTWing 公开协议栈完全是 reactive 重传（MQTT QoS ack / TCP 重传），**不进行 proactive packet-level 冗余发送** | 不命中 |

**反向证据汇总**：
1. CTWing 公开接入协议明确为 T-Link（MQTT 改进）/ MQTT / MQTT-SN / LWM2M / HTTP(S) / TCP，全部采用 ARQ 类机制（broker ack + 客户端重传 / TCP 超时重传）。**ARQ ≠ FEC**，是 reactive 不是 proactive。
2. EMQX 客户案例确认中国电信 NB-IoT 平台底层采用 EMQX MQTT broker，可靠性靠 MQTT QoS 等级，**broker 层不引入 FEC / 冗余副本**。
3. 4 次定向搜索（含 `"CTWing" "FEC"`、`"天翼物联" "前向纠错"`、`"网络编码"`、`"主动冗余"`、`"业务类型自适应"`）在公开检索域均**无候选侧任何耦合证据**——即 CTWing 与本专利核心 F# 在公开语料中不共现。
4. 注意区分：MQTT QoS 是消息送达级别（at-most-once / at-least-once / exactly-once），由 ack + 重传实现，与本专利 F5 要求的"主动发送 n≥1 个冗余副本"无任何关联。

## 最终判定 **第 5 档：已排除**

**判定依据**（同时满足两条排除硬条件之一即可，本候选满足两条）：
1. **反向证据存在**：CTWing 公开协议栈（T-Link/MQTT/MQTT-SN/LWM2M/HTTP/TCP）整体属于"ack + 重传"族 ARQ 机制，与本专利核心 F5（proactive 冗余包发送）的技术路线**互斥**——不是"缺证据"，是"证据指向相反方向"。
2. **领域 / 主体技术路线不耦合**：本专利核心是传输层 / 数据链路层的主动 FEC + 业务类型自适应冗余调度（典型用法：RTC / 低延时控制流），而 CTWing 平台职责是 IoT 设备接入、连接管理、物模型、设备影子 / OTA / 鉴权——其可靠性诉求在 broker-ack 与 TCP 重传层面已被满足，**业务上无引入主动 FEC 的动因**。

（重申：0 命中 ≠ 已排除；本档同时存在公开协议栈反向证据 + 业务动因背离，方可入第 5 档。）

## 升级路径（仅作为后续如证据出现时的复评提示，不改变本轮判定）
- 若日后出现 CTWing **自身发布**或**与第三方合作**的 IoT 加速 / 弱网增强 SDK 文档，其中明确包含：(a) 发送端基于真实流量统计自动识别业务类型；(b) 主动发送 n≥1 冗余副本；(c) 冗余数量同时依赖网络状态 + 成功率 + 业务类型——可考虑回升至 第 3 档（疑似但证据不足）复评。

## 总结一句话
CTWing 天翼物联公开协议栈仅采用 MQTT/T-Link/LWM2M/HTTP/TCP + QoS/ARQ 重传，与本专利"主动 FEC + 业务类型自适应冗余调度"技术路线互斥，4 次定向检索无候选侧任何耦合证据，**落第 5 档（已排除）**。
